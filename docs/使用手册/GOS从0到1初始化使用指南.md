# GOS 从 0 到 1 初始化使用指南

> 这份文档面向第一次接触 GOS 的实施同学、运维同学和平台管理员。
> 目标不是介绍概念，而是帮助你把系统真正跑起来，并完成第一条可用的发布链路。

## 1. 文档目标

完成这份指南后，你应该能够：

- 启动 GOS 前后端
- 登录系统并完成基础初始化
- 创建应用、绑定 CI/CD 能力
- 创建发布模板
- 创建一张发布单
- 完成审批、执行、查看结果
- 理解后续如何接入 Agent、通知 Hook、GitOps

## 2. 先理解 GOS 是什么

GOS 不是 Jenkins、ArgoCD、GitOps 的替代品，而是这些能力之上的发布治理层。

可以把它理解成：

- Jenkins 负责构建、执行流水线
- ArgoCD / GitOps 负责部署到集群
- Agent 负责在特殊网络环境中主动执行任务
- GOS 负责把这些能力组织成一条可治理、可审批、可追踪、可回滚的发布流程

## 3. 推荐初始化顺序

请严格按这个顺序做，最省时间：

1. 启动后端
2. 启动前端
3. 登录管理员账号
4. 初始化系统设置
5. 配置用户与权限
6. 创建应用
7. 配置 Jenkins / ArgoCD / GitOps / Agent / 通知源（按需）
8. 绑定应用与执行器
9. 创建发布模板
10. 创建并执行第一张发布单

## 4. 环境准备

### 4.1 基础依赖

- Go `1.25+`
- Node.js `20+`
- MySQL `8+` 或 SQLite
- Jenkins（如果你要走 CI/CD 流水线）
- ArgoCD + GitOps 仓库（如果你要走 GitOps 发布）
- 可选：Agent、钉钉、企业微信

### 4.2 推荐准备两套环境

#### 本地体验环境

适合：

- 前端联调
- 功能开发
- 页面验证
- 轻量冒烟

推荐：

- `SQLite`
- 不开启 Jenkins / ArgoCD / GitOps

#### 真实联调环境

适合：

- 接 CI/CD
- 接 GitOps
- 接审批、Hook、Agent、通知链路

推荐：

- `MySQL`
- 按实际接入 Jenkins / ArgoCD / GitOps / Agent

## 5. 后端启动

### 5.1 最小可用启动（本地 SQLite）

在仓库根目录执行：

```bash
go run ./cmd/server
```

默认监听：

- `http://127.0.0.1:8081`

健康检查：

```bash
curl http://127.0.0.1:8081/healthz
```

Swagger：

- `http://127.0.0.1:8081/swagger/index.html`

### 5.2 使用 MySQL 启动（推荐真实环境）

```bash
go run ./cmd/server -config configs/config.production.json
```

### 5.3 关键说明

- 服务端只读取配置文件，不再使用环境变量覆盖配置值
- 上线前请直接编辑 `configs/config.production.json`
- `security.encryption_key` 会用于加密：
  - Agent Token
  - GitOps / ArgoCD 凭据
  - 通知源 Secret

## 6. 前端启动

```bash
cd frontend
npm install
VITE_API_BASE_URL=http://127.0.0.1:8081 npm run dev
```

默认地址：

- `http://127.0.0.1:5174`

## 7. 第一次登录

默认管理员用户名：

- `admin`

管理员密码：

- 由配置文件中的 `auth.admin_password` 决定

登录后先确认下面几个页面能打开：

- `/applications`
- `/releases`
- `/release-templates`
- `/system/users`
- `/system/permissions`
- `/system/settings`

## 8. 系统初始化清单

### 8.1 发布环境设置

进入：

- `系统管理 -> 发布环境设置`

确认至少包含：

- `dev`
- `test`
- `prod`

并确认并发策略、锁范围符合你们当前口径。

### 8.2 创建用户并授予权限

进入：

- `系统管理 -> 用户管理`
- `系统管理 -> 权限授权`

建议先准备三类账号：

1. 平台管理员
2. 发布发起人
3. 审批人

当前系统口径建议：

- 应用级权限负责：展示、发起
- 发布执行负责：谁发起谁执行
- 审批负责：模板里明确配置审批人

### 8.3 创建应用

进入：

- `我的应用` 或 `应用管理`

创建应用时至少填写：

- 应用名称
- `app_key`
- 仓库地址
- 负责人
- 语言/制品类型

如果应用接 GitOps，请额外关注：

- 仓库地址填 Git 仓库根地址
- 不要填 `/tree/分支/目录` 这种页面 URL

## 9. 执行器接入

这一部分不是必须一步到位，可以按你们实际场景逐步接。

### 9.1 只走 Jenkins 管线

适合：

- 先把发版流程跑通
- 不上 GitOps
- 只需要 CI/CD 流水线能力

需要做：

1. 在 `组件管理 -> Jenkins` 确认可连通
2. 同步管线
3. 在应用下绑定：
   - CI 管线
   - CD 管线（如果有）

### 9.2 走 ArgoCD / GitOps

适合：

- 需要 Helm / values 驱动发布
- 需要声明式部署

需要做：

1. 创建 GitOps 实例
2. 创建 ArgoCD 实例
3. 做环境绑定：
   - `env -> ArgoCD 实例`
4. 在 ArgoCD 实例上绑定 GitOps 实例
5. 确认 GitOps 仓库结构存在

当前推荐约定：

- Helm 路径统一为 `apps/helm`
- GitOps 分支可按应用与环境隔离
- 发布时平台会解析：
  - 应用
  - 环境
  - ArgoCD 实例
  - GitOps 实例
  - 目标仓库与分支

### 9.3 走 Agent（特殊网络环境）

适合：

- 平台不能直连生产环境
- 需要生产侧主动拉取任务

需要做：

1. 在平台创建 Agent
2. 导出安装配置
3. 在目标主机启动 Agent
4. 在平台里维护：
   - 脚本管理
   - 任务管理

### 9.4 通知模块

进入：

- `系统管理 -> 通知模块`

可以维护：

- 通知源（钉钉 / 企业微信）
- Markdown 模板
- 通知 Hook

后续发布模板的 Hook 可以直接选择这些通知 Hook。

## 10. 应用绑定与参数准备

### 10.1 管线绑定

进入：

- `应用详情 -> 管线绑定`

完成：

- CI 绑定
- CD 绑定（如果有）
- 参数映射

### 10.2 标准字库

进入：

- `标准字库`

维护平台标准参数，例如：

- `app_key`
- `project_name`
- `env`
- `git_ref`
- `image_version`

后面这些标准 Key 会被用于：

- 发布模板参数
- Hook 变量
- 通知模板变量
- Agent 任务变量

## 11. 创建发布模板

进入：

- `发布模板`

### 11.1 一张最小模板至少包含

- 模板名称
- 所属应用
- 操作类型（发布 / 回滚）
- CI 参数定义
- CD 参数定义（如果需要）

### 11.2 如果要上审批

在模板里配置：

- 是否启用审批
- 审批方式
- 审批人

当前系统支持：

- 审批工作台
- 详情页审批卡片
- 审批通过后才能执行发布
- 发起人=唯一审批人时可自动通过

### 11.3 如果要上 Hook

在模板里可以配置发布后 Hook，例如：

- Agent 任务 Hook
- 通知 Hook

推荐先用一个简单通知 Hook 验证链路。

## 12. 从 0 到 1 跑通第一条真实链路

这里给你一条最稳的最小闭环。

### 12.1 路线建议

优先推荐你们第一次这样跑：

1. 创建一个应用
2. 绑定 Jenkins CI
3. 创建一张发布模板
4. 不上审批或只上单审批人审批
5. 不先接复杂 Hook
6. 先创建一张最简单的发布单跑通

### 12.2 创建发布单

进入：

- `发布单 -> 新建发布单`

按模板填写参数后创建。

### 12.3 审批（如果模板启用了审批）

进入：

- 发布单详情页
- 或 `审批工作台`

完成：

- 提交审批
- 审批通过

### 12.4 执行发布

当前系统规则建议理解成：

- 应用级权限：负责可见、发起
- 执行发布：谁发起的单，谁执行

所以请用发起人账号执行。

### 12.5 查看执行结果

在发布单详情里重点看：

- 基础信息
- 执行单元
  - `CI 执行单元`
  - `CD 执行单元`
  - `Hook 执行单元`
- 取值进度
- 步骤流
- 日志

## 13. 推荐的第一条正式可用链路

如果你想让系统尽快在团队里“真的能用起来”，推荐按这条顺序落地。

### 阶段 1：最小可用

- 后端 + 前端跑起来
- 管理员可登录
- 用户、权限可配置
- 应用可创建
- Jenkins 可同步
- 模板可创建
- 发布单可创建并执行

### 阶段 2：治理补齐

- 模板审批
- 审批工作台
- 发布单执行权限口径统一
- 并发与锁规则稳定
- 发布前预检

### 阶段 3：扩展能力

- GitOps / ArgoCD
- Hook
- 通知模块
- Agent 主动执行

## 14. 真实实施建议

### 14.1 建议先选一类应用试点

不要一上来全量推广，先选：

- 1 个 Java 服务
- 1 条 CI 流水线
- 1 套简单发布模板
- 1 个审批人

把流程跑顺后再复制。

### 14.2 建议先定 3 条组织口径

1. 谁能创建发布单
2. 谁能审批
3. 谁能执行发布

这三条不先定，系统再完整也会乱。

### 14.3 建议先定 1 条 GitOps 规范

至少统一：

- 仓库结构
- Helm 路径
- 分支策略
- ArgoCD Application 命名

## 15. 常见问题

### 15.1 登录失败

先检查：

- 后端是否启动
- `auth.admin_password` 是否正确
- 前端是否连到了正确后端地址

### 15.2 页面没数据

先确认当前后端连的是：

- SQLite 还是 MySQL
- 是空本地库，还是历史业务库

### 15.3 Jenkins / ArgoCD 接不通

先把对应开关关闭，确保平台主链路先能跑，再逐步打开集成能力。

### 15.4 Hook 没触发

检查：

- 模板是否真的保存了 Hook
- 触发条件是否命中
- 发布详情的 Hook 执行单元是否有记录

## 16. 一页式初始化清单

### 必做

- 启动后端
- 启动前端
- 登录管理员
- 创建用户
- 配权限
- 创建应用
- 创建模板
- 建一张发布单
- 执行一次发布

### 按需做

- Jenkins 接入
- ArgoCD / GitOps 接入
- 审批流
- Hook
- 通知
- Agent

## 17. 推荐的首次验收标准

当下面这些都通过，就说明系统已经从 0 到 1 真正跑起来了：

1. 管理员能登录
2. 发布发起人能看到自己的应用
3. 能创建一张发布单
4. 审批人能在工作台完成审批
5. 发起人能执行自己的发布单
6. 发布单详情能看到执行单元和步骤
7. 至少一条 Hook 能成功触发
8. 至少一个通知能发出去（如果启用了通知）

## 18. 建议下一步

如果你已经完成第一条链路，下一步最值得做的是：

1. 固化一套标准发布模板
2. 固化一套标准审批策略
3. 固化一套 GitOps 目录与分支规范
4. 让通知 Hook 成为默认模板能力
5. 逐步接入 Agent 解决生产网络隔离问题

---

如果你是第一次部署 GOS，建议先不要追求“所有能力一次上线”，而是先把：

- 登录
- 应用
- 模板
- 发布单
- 审批
- 执行

这一条最短链路跑通。这样团队最快能感知价值，后面的 Hook、通知、Agent、GitOps 都会更容易落地。
