## 2. 本版目标
在 `v0.0.10` 已具备 `GitOps 替换规则`、`GitOps 定位字段`、以及 YAML 字段选择体验优化的基础上，前端正式把 Helm 场景收口为：
- 用户不再围绕渲染后的 Kubernetes YAML 做配置
- 用户改为围绕 Helm `values` 文件与 `values_path` 做配置
- 平台只负责修改 values
- ArgoCD + Helm 负责渲染与部署

当前已经落地的前端方向：
- Helm 模式不再渲染整块 YAML/values 预览弹窗
- Helm 模式改为轻量下拉选择 `Values 路径`
- Helm 模式固定写入平台专用文件 `platform.values-{env}.yaml`
- 模板页打开时实时拉取最新 values 候选，尽量避免仓库变更后选错字段

## 3. 范围
包含：
- 发布模板页新增 `GitOps 类型` 选择
- `CD=ArgoCD` 且 `gitops_type=helm` 时，规则编辑切换为 `Values 模式`
- `values_path` 下拉选择
- `values_file_template` 在 Helm 模式下固定为平台文件
- `value_template` 手工编辑能力

不包含：
- 平台直接渲染 Helm chart 结果
- 可视化编辑整个 values 文件
- Chart 仓库接入/管理
- Helm CLI 调用页面

## 4. 背景
当前前端已经支持：
- `CD=ArgoCD` 的模板提示
- `GitOps 替换规则`
- 基于样板 YAML 的字段选择

但如果 GitOps 资产本身是 Helm：
- 再让用户选 `Deployment.spec...` 这类 Kubernetes YAML 字段，会越来越别扭
- 用户真正理解和维护的，是 `values.yaml` 里的键路径

因此 Helm 场景下前端应切换到：
- 选择 values 文件
- 选择 values 路径
- 配置值模版

## 5. 发布模板页
文件目标：
- `/Users/lingyunxieqing/Desktop/gos/frontend/src/views/release/ReleaseTemplateView.vue`

### 5.1 GitOps 类型
当模板 `CD` 留空走 `ArgoCD` 时，页面新增：
- `GitOps 类型`
  - `Kustomize`
  - `Helm`

默认：
- 若历史模板已有 YAML 规则，则显示为 `Kustomize`
- 新模板可默认选 `Helm` 或按产品决定默认值

### 5.2 Helm 模式说明
当选择 `Helm` 时，页面提示文案改为：
- 当前 `CD` 使用 `ArgoCD + Helm`
- 平台会在发布执行阶段修改 Helm values 文件并触发同步
- 平台不会直接修改 Helm 渲染后的 Kubernetes YAML

### 5.3 当前收口
本版前端对 Helm 规则先做产品收口：
- Helm 模式不再展示 `定位字段`
- Helm 模式不再允许选择任意业务 values 文件
- 目标文件统一固定为：
  - `platform.values-{env}.yaml`

这样做的目的：
- 避免平台误改业务 values 文件
- 让平台只接管镜像版本等受控发布变量
- 降低模板编辑页的复杂度和内存占用

## 6. GitOps 规则区
### 6.1 Kustomize 与 Helm 区分
当 `gitops_type = kustomize`：
- 沿用现有 `YAML 字段` 规则 UI

当 `gitops_type = helm`：
- 规则 UI 切换为 `Values 规则`
- 不再显示 `YAML 字段`
- 改为显示：
  - `标准字段`
  - `目标文件（只读）`
  - `Values 路径`
  - `取值模版`

### 6.2 目标文件
当前 Helm 模式下，目标文件不再允许用户修改，页面直接只读展示：
- `platform.values-{env}.yaml`

说明：
- 前端不再暴露任意 `values_file_template` 编辑能力
- 业务 values 文件仍由研发/运维维护
- 平台当前只接管平台专用 values 文件

### 6.3 Values 路径
用户通过下拉选择一个 `values_path`，例如：
- `image.tag`
- `nameOverride`
- `config.nacosNamespace`

当前交互：
- 页面实时请求后端扫描到的候选
- 仅展示 `platform.values-{env}.yaml` 中的路径
- 保存后展示为可读标签，不再让用户面对整块 YAML 文本

### 6.4 取值模版
保留可编辑能力：
- 默认值：`{标准Key}`
- 可自行调整为：
  - `{image_version}`
  - `{project_name}`
  - `{env}-{project_name}`

## 7. Values 候选同步
### 7.1 交互要求
前端不再渲染大块 YAML/values 选择弹窗，而是直接使用轻量下拉：
- 打开模板编辑时，主动拉取最新 values 候选
- 切换应用、切换 `GitOps 类型` 时，重新拉取
- 用户直接在 `Values 路径` 下拉中选目标路径

### 7.2 同步行为
- values 候选以当前 GitOps 仓库真实内容为准
- 仓库文件变化后，再次打开模板或刷新候选即可拿到最新路径
- 候选为空时，页面需要明确提示：
  - 当前应用尚未准备好 `platform.values-{env}.yaml`
