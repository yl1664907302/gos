## 2. 本版目标
将当前错误的“按应用找最近成功单并复制参数”的回滚逻辑，重构为两种明确且可解释的恢复方式：
- `标准回滚`
- `重放回滚`

前端需要把这两种能力在页面中明确区分，避免继续把所有历史恢复动作都混叫为“回滚”。

## 3. 当前问题
当前平台的“回滚”存在以下问题：
- 回滚来源不明确：按应用自动找最近成功单，而不是用户明确选中的成功单。
- 恢复维度不明确：没有严格绑定到某一张成功发布单的真实部署结果。
- 对 `ArgoCD + Helm` 场景，本质上是在复制参数，不是在恢复历史部署快照。
- 对 `CD=管线` 场景，本质上没有区分“真正回滚”和“历史参数重放”。
- 列表与详情页虽然展示了“回滚”标签，但用户并不知道这次恢复到底是恢复历史状态，还是只是又发了一次。

## 4. 产品定义
### 4.1 标准回滚
适用范围：
- `CD = ArgoCD`
- `gitops_type = helm`

定义：
- 用户从一张明确的成功发布单发起“回滚到此版本”。
- 平台读取该成功单对应的历史部署快照。
- 新建一张 `rollback` 类型的发布单。
- 该发布单只执行 `CD`，不执行 `CI`。
- 平台重新写入该历史快照对应的 Helm values 结果，再触发 ArgoCD Sync 与健康检查。

前端定位：
- 这是平台中唯一应该叫“回滚”的恢复动作。

### 4.2 重放回滚
适用范围：
- `CD = 管线`
- `仅 CI`

定义：
- 用户从一张明确的成功发布单发起“回滚到此版本”。
- 平台优先读取该成功单中与恢复作用域一致的历史参数快照：
  - 若存在 `CD` 参数快照，则优先使用 `CD`
  - 若模板仅有 `CI`，或 `CD` 存在但无参数快照，则回退使用 `CI`
- 新建一张 `replay` 类型的发布单。
- 该发布单仅执行对应作用域，不再强制要求同时存在 `CI + CD`。
- 当作用域为 `CI` 时，本质上是“基于历史成功单参数重新执行 CI”。

前端定位：
- 这不是标准回滚。
- 页面操作入口统一显示为：`回滚到此版本`
- 但新建单的类型标签要明确显示为：`重放回滚`

## 6. 弹窗交互
### 6.1 标准回滚确认弹窗
展示内容：
- 来源发布单号
- 应用名称
- 环境
- 来源镜像版本
- ArgoCD 实例
- GitOps 目标文件
- 说明：`本次仅执行 CD，平台将恢复该成功单的历史部署快照。`

按钮：
- `确认回滚`
- `取消`

### 6.2 重放回滚确认弹窗
展示内容：
- 来源发布单号
- 应用名称
- 环境
- 本次命中的恢复作用域：`CI` 或 `CD`
- 对应作用域的关键参数摘要
- 说明：
  - `CI`：`本次将基于该成功单的历史 CI 参数重新执行 CI。`
  - `CD`：`本次将基于该成功单的历史 CD 参数重新执行部署管线。`

按钮：
- `确认重放`
- `取消`

## 7. 新建出的恢复单展示
### 7.1 公共要求
新建出的恢复类发布单在详情与列表中都必须明确来源与类型。

建议新增字段展示：
- `操作类型`
  - `普通发布`
  - `标准回滚`
  - `重放回滚`
- `来源发布单号`

### 7.2 列表页标签
列表页建议新增恢复类型标签：
- `标准回滚`
- `重放回滚`

不再仅使用单一的 `回滚` 标签覆盖两种不同语义。

### 7.3 详情页标题区
详情页顶部标题区展示：
- `标准回滚自 RO-xxxx`
- `重放回滚自 RO-xxxx`

## 8. 页面文案规范
前端文案必须严格区分：
- `回滚`
- `重放回滚`

禁止再出现以下混用：
- 将 `replay` 类型错误显示成“普通发布”
- 将 `replay` 类型标签继续写成“参数重放”

推荐文案：
- `回滚到此版本`
- `标准回滚`
- `重放回滚`

## 9. 模板页说明
文件目标：
- `/Users/lingyunxieqing/Desktop/gos/frontend/src/views/release/ReleaseTemplateView.vue`

模板页在 `CD` 区域新增只读说明：
- 若 `CD = ArgoCD`：`该模板支持标准回滚，回滚时仅执行 CD 并恢复历史部署快照。`
- 若 `CD = 管线`：`该模板支持重放回滚，优先复用历史 CD 参数；若无 CD 参数快照，则回退复用历史 CI 参数。`
- 若 `仅 CI`：`该模板支持重放回滚，恢复时将复用历史 CI 参数重新执行。`

此处不增加额外选择器，恢复策略由 `CD 类型` 自动推导。

## 10. 详情页执行结构
### 10.1 标准回滚单
详情页中应表现为：
- `CI 执行单元` 不显示，或显示为 `未参与`
- `CD 执行单元` 为主执行单元
- 步骤重点展示：
  - `写回历史 values`
  - `Git 提交`
  - `Git 推送`
  - `ArgoCD 同步`
  - `健康检查`

### 10.2 重放回滚单
详情页中应表现为：
- 若本次重放作用域为 `CD`：
  - `CI 执行单元` 不显示，或显示为 `未参与`
  - `CD 执行单元` 为主执行单元
  - 步骤重点展示：
    - `参数装载`
    - `CD 管线触发`
    - `CD 管线运行`
    - `CD 管线完成`
- 若本次重放作用域为 `CI`：
  - `CI 执行单元` 为主执行单元
  - `CD 执行单元` 不显示，或显示为 `未参与`
  - 步骤重点展示：
    - `参数装载`
    - `CI 管线触发`
    - `CI 管线运行`
    - `CI 管线完成`

## 11. 权限建议
恢复类动作仍沿用应用级发布权限：
- `release.create`
- `release.execute`

前端不新增独立权限码，但需要在交互上区分两种动作名称与说明。

## 13. 结论
`v0.0.13` 前端的关键不是再做一个“回滚按钮”，而是把恢复动作的语义彻底讲清楚：
- `ArgoCD` 走 `标准回滚`
- `CD 管线 / 仅 CI` 走 `重放回滚`

用户在页面上应始终知道：
- 我恢复的是哪一张成功单
- 这次恢复到底是不是标准回滚
- 本次是重放 `CI` 还是重放 `CD`
