- 版本：`v0.0.6`
- 对应后端基线日期：`2026-03-14`
- 上一版本：`/Users/lingyunxieqing/Desktop/gos/docs/前端/前端需求v0.0.5.md`

## 2. 版本目标
新增“用户管理模块”并完成权限驱动前端展示，达成：
- 新增登录页面与登录态管理（登录/退出/会话恢复）
- 应用创建时可下拉选择负责人用户
- 区分管理员与普通用户前端视图与操作权限
- 普通用户仅展示/可操作已授权能力
- 细化到发布单创建页参数级展示与编辑控制

## 3. 范围与边界
包含：
- 登录页（未登录访问拦截）
- 用户管理页面（管理员）
- 权限授权页面（管理员）
- 应用创建/编辑负责人下拉
- 菜单与按钮级权限控制
- 发布参数字段级展示/禁用控制

不包含：
- SSO 登录页 UI
- 组织架构树、岗位体系
- 审批流前端流程

## 4. 菜单与路由增量

### 4.1 菜单结构
- 新增一级菜单：`系统管理`
- 二级菜单：
- `用户管理`
- `权限授权`

### 4.2 新增路由
- `/login`：登录页
- `/system/users`：用户列表页
- `/system/users/:id`：用户详情页（可选）
- `/system/permissions`：权限授权页

### 4.3 菜单显示规则
- 登录页不展示左侧菜单与业务布局壳
- 管理员：显示全部菜单
- 普通用户：仅显示已授权模块菜单；未授权模块不展示（非仅禁用）

## 5. 用户与权限前端模型

### 5.1 用户对象
- `id`
- `username`
- `display_name`
- `email`
- `phone`
- `role`（`admin | normal`）
- `status`（`active | inactive`）

### 5.2 权限对象
- `permission_code`
- `module`
- `action`
- `scope_type`
- `scope_value`

### 5.3 参数级权限对象
- `param_key`
- `application_id`（可空）
- `can_view`
- `can_edit`

### 5.4 当前用户上下文
- `GET /me` 返回：
- 当前用户基础信息
- 角色
- 授权权限列表
- 参数级权限快照（可按需懒加载）

## 6. 页面需求

### 6.1 登录页（新增）
- 表单字段：
- `username`（必填）
- `password`（必填）
- 基本交互：
- 点击登录后调用 `POST /auth/login`
- 登录成功后拉取 `GET /me`，初始化权限并跳转首页
- 登录失败展示后端错误（不暴露敏感细节）
- 补充交互：
- 支持回车提交
- 登录中按钮 loading，避免重复提交
- 风格要求（必须）：
- 需与现有管理后台风格一致（色系、圆角、阴影、按钮规范）
- 遵循样式规范目录：`/Users/lingyunxieqing/Desktop/gos/docs/样式规范/`
- 避免与当前系统视觉割裂（不可单独换一套 UI 语言）

### 6.2 用户管理页（管理员）
- 查询条件：用户名、姓名、角色、状态
- 表格字段：用户名、姓名、角色、状态、邮箱、更新时间、操作
- 操作：新增、编辑、停用/启用、删除（按后端能力）
- 表单字段：`username`、`display_name`、`role`、`status`、`email`、`phone`

### 6.3 权限授权页（管理员）
- 用户选择区：先选用户
- 权限勾选区：
- 模块权限（应用/管线/发布/组件/系统）
- 动作权限（view/create/update/delete/execute/cancel/grant）
- 参数级授权区（重点）：
- `param_key` 下拉（来源标准字库）
- `application_id` 可选
- `can_view`、`can_edit` 开关
- 支持保存、撤销授权

### 6.4 应用创建/编辑页变更
- 负责人字段从输入改为下拉选择
- 数据源：`GET /users/options`
- 提交字段：`owner_user_id`
- 展示字段：`owner`（显示负责人名称）

### 6.5 发布单创建页（重点）
- 参数列表使用后端返回参数（已含权限过滤结果）
- 对每个参数按权限渲染：
- 无 `can_view`：前端不展示
- `can_view=1, can_edit=0`：只读控件（禁用态）
- `can_edit=1`：可编辑
- 必填且只读参数：
- 若后端已给默认值，直接展示默认值
- 若无默认值，展示风险提示并阻止提交（提示联系管理员授权）
- 提交时仅提交可编辑参数与允许提交字段

## 7. 交互与 UX 约束
- 未登录访问受控路由：
- 统一跳转 `/login`
- 未授权访问路由：
- 直接跳转 `403` 页面或首页并提示“未授权”
- 未授权按钮：
- 不展示优先；必要时展示禁用态并带提示
- 权限变更后：
- 前端需刷新当前用户权限缓存（重新拉取 `/me`）
- 全局指令建议：
- `v-permission`（菜单/按钮/区块级）
- `v-param-permission`（发布参数字段级）

## 8. API 对接清单

### 8.0 认证模块
- `POST /auth/login`
- `POST /auth/logout`
- `GET /me`

### 8.1 用户模块
- `GET /users`
- `GET /users/{id}`
- `POST /users`
- `PUT /users/{id}`
- `DELETE /users/{id}`
- `GET /users/options`

### 8.2 权限模块
- `GET /permissions`
- `GET /users/{id}/permissions`
- `POST /users/{id}/permissions`
- `DELETE /users/{id}/permissions`
- `GET /users/{id}/param-permissions`
- `POST /users/{id}/param-permissions`
- `PUT /users/{id}/param-permissions/{permission_id}`
- `DELETE /users/{id}/param-permissions/{permission_id}`

### 8.3 上下文与业务联动
- `POST /applications`（新增 `owner_user_id`）
- `PUT /applications/{id}`（支持 `owner_user_id`）
- `GET /applications/{id}/pipeline-param-defs`（返回 `can_view/can_edit`）

## 9. 状态管理与缓存
- 新增 `auth store`（Pinia）：
- 保存当前用户、角色、权限点
- 保存登录 token（建议内存 + 本地存储兜底）
- 提供 `hasPermission(code)`、`canViewParam(paramKey, appId)`、`canEditParam(...)`
- 首屏加载策略：
- 登录后先拉 `/me`
- 成功后再渲染动态菜单
- 默认不依赖 Redis，前端权限展示以 `/me` 返回结果为准

## 10. 验收标准（v0.0.6）
- 管理员可在前端完成用户 CRUD 与授权
- 创建应用时负责人可下拉选择用户并保存成功
- 普通用户仅能看到并操作已授权模块
- 发布单创建页参数展示/编辑行为符合授权规则
- 未授权访问受控页面与操作有统一提示与拦截

## 11. 里程碑建议
- M1：用户管理页面与 API 对接
- M2：权限授权页面与权限缓存
- M3：菜单/按钮级权限接入
- M4：发布参数页面字段级权限接入

## 12. Redis 可选增强项对前端影响（非必选）
- 前端无需新增专属 Redis 接口，仍通过 `POST /auth/login`、`POST /auth/logout`、`GET /me` 工作。
- 若后端启用 Redis 会话与权限缓存，前端行为不变，只会体现在登录与鉴权响应更快。
- 权限变更后的前端策略保持不变：关键操作后刷新 `/me`，确保菜单与按钮状态及时更新。
