## 2. 背景
当前前端已经具备：
- 应用卡片、项目筛选、环境权限
- 发布模板管理
- 发布单创建、详情、列表
- 标准回滚 / 标准重放
- 确认生效、当前版本 / 上次版本气泡

但平台仍缺少一层可视化、可复用、可审计的 K8s 发布策略配置。
目前 K8s 发布行为更多依赖：
- GitOps 仓库中的 YAML / Helm values
- 人工约定
- 模板中的零散字段
- 以及缺失统一的 K8s 目标实例 / 运行时实例展示

这会导致：
- 用户无法直接知道当前环境采用什么发布策略
- 发布单无法显式展示本次发布使用的 K8s 行为
- 回滚时也无法明确恢复的是哪套发布策略

因此 v2.0 需要把 K8s 发布策略提升为前端可配置、可查看、可追踪的一等对象。

## 3. 目标
前端需要支持：
1. 在应用维度按环境配置发布策略
2. 明确区分 `交付引擎` 与 `发布策略引擎`
3. 引入 `K8s 目标实例 / 运行时实例` 的可视化配置与展示
4. v2.0 提供原生 K8s 策略配置：
   - `RollingUpdate`
   - `Recreate`
   - 探针
   - 资源
   - PDB
5. 在发布单创建、预检、详情中展示策略摘要
6. 在标准回滚预审中展示“将恢复的策略摘要”
7. 为后续 `Argo Rollouts / Canary / BlueGreen` 预留界面结构

## 4. 核心原则

### 4.1 交付引擎与发布策略引擎要分开展示
前端配置和展示必须区分：
- `交付引擎`
- `发布策略引擎`

用户应该能理解：
- `ArgoCD` 负责把配置同步进集群
- `K8s Native / Argo Rollouts` 决定集群里怎么发布

### 4.2 v2.0 先做结构化表单，不做 YAML 编辑器
本期不做通用 YAML 编辑器。

前端应提供：
- 分组表单
- 结构化输入
- 合法值约束

而不是让用户直接编辑大段 YAML。

### 4.3 策略配置入口放在应用环境侧
策略不应只出现在模板页。

主入口建议放在：
- 应用详情 / 应用卡片的环境配置区

因为策略最终绑定的是：
- `app + env`

### 4.4 平台里最好开始引入一层 K8s 目标实例 / 运行时实例模型
前端在 v2.0 就应该把以下信息显式化：
- 当前环境落在哪个 K8s 集群目标
- 当前环境对应哪个 namespace / workload / service
- 当前环境是否支持原生 K8s 策略
- 当前环境是否支持 Argo Rollouts

这层模型在前端的作用不是立即支持“直连集群”，而是把环境运行时信息从隐式约定改成显式配置。

### 4.5 发布单只展示“解析结果”
发布单创建页和详情页不应再次变成完整策略配置器。

发布单侧重点应是：
- 本次命中的策略
- 是否允许覆盖字段
- 实际策略快照

## 6. 产品模型

### 6.1 策略模板
策略模板页用于维护：
- 模板名称
- 策略引擎
- 策略类型
- 通用配置

v2.0 模板类型：
- `k8s_native / rolling_update`
- `k8s_native / recreate`

### 6.2 应用环境策略绑定
每个应用的每个环境都可配置：
- 交付引擎
- 策略引擎
- 策略类型
- 引用的策略模板
- 是否允许发布时覆盖
- 允许覆盖的字段白名单

### 6.3 K8s 目标实例 / 运行时实例
前端需要引入两层概念：

1. `K8s 目标实例`
- 集群目标信息
- 支持能力信息
- 与 ArgoCD / Agent / 未来直连方式的关系

2. `应用环境运行时绑定`
- 某个 `app + env` 实际落到哪个 K8s 目标实例
- 使用哪个 namespace / workload / service / argo app

### 6.4 发布单中的策略快照
发布单只显示：
- 当前命中的运行时目标
- 当前命中的交付引擎
- 当前命中的策略引擎
- 当前命中的策略类型
- 实际生效的探针 / 资源 / PDB 摘要

## 8. 表单设计

### 8.1 交付引擎
前端枚举：
- `argocd`
- `jenkins`
- `agent`

展示文案建议：
- `ArgoCD`
- `Jenkins`
- `Agent`

### 8.2 策略引擎
前端枚举：
- `none`
- `k8s_native`
- `argo_rollouts`

v2.0 可选范围：
- `none`
- `k8s_native`

### 8.3 策略类型
当 `strategy_engine = k8s_native` 时：
- `rolling_update`
- `recreate`

当后续 `strategy_engine = argo_rollouts` 时再扩展：
- `canary`
- `blue_green`

### 8.4 探针区
每种探针建议统一表单结构：
- 是否启用
- 协议类型
  - `http`
  - `tcp`
  - `command`
- 路径 / 端口 / 命令
- initialDelaySeconds
- periodSeconds
- timeoutSeconds
- successThreshold
- failureThreshold

### 8.5 资源区
字段建议：
- `cpu request`
- `memory request`
- `cpu limit`
- `memory limit`

### 8.6 PDB 区
字段建议：
- 启用开关
- 模式：
  - `minAvailable`
  - `maxUnavailable`
- 数值

说明：
- 同时只能填一种模式

## 12. 标准回滚与标准重放

### 12.1 标准回滚预审
在应用卡片点击 `标准回滚` 后，预审弹窗中需展示：
- 当前版本
- 目标版本
- 当前运行时目标
- 当前策略
- 目标恢复策略

明确告知用户：
- 这次恢复的不只是版本，也包括策略快照
- 也基于当前环境的运行时目标做恢复

### 12.2 标准重放预审
若当前环境仅支持 `标准重放`，也应展示：
- 将复用的策略快照摘要

避免用户误解为只是重新跑流水线。

## 14. 后续 Argo Rollouts 的 UI 预留
虽然 v2.0 不做真正接入，但页面结构需预留：
- `策略引擎 = Argo Rollouts`
- `策略类型 = Canary / BlueGreen`

建议方式：
- 当前置灰显示“即将支持”
- 或在策略类型中隐藏，保留后端兼容枚举

后续扩展时，前端主要新增：
- Canary steps 配置器
- BlueGreen service 配置区
- 分析模板选择器
- 手动晋级 / 自动晋级配置

届时会直接复用：
- `K8s 目标实例`
- `应用环境运行时绑定`

来判断某环境是否可用 Rollouts。

## 15. 交互要求
- 配置页字段分组清晰，不堆成一屏长表单
- 资源、探针、PDB 需有默认值提示
- 危险字段变更需给出轻量风险说明
- 发布单中策略摘要要“扫一眼能懂”，不要塞成 YAML
