# 后端需求文档 v0.0.10（基于 v0.0.9 增量）

## 1. 文档信息
- 文档版本：`v0.0.10`
- 基线日期：`2026-03-18`
- 上一版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端/后端需求v0.0.9.md`
- 本版主题：`ArgoCD/GitOps 变量替换前置约束 + project_name 单值场景`

## 2. 本版目标
在 `v0.0.8` 已落地的 `ArgoCD/GitOps` 管理能力、以及 `v0.0.9` 的回滚与取值进度增强基础上，先把 `project_name` 在 `CD=ArgoCD` 场景下收成单值语义，为下一版 GitOps YAML 替换规则打地基。

本版重点不是立刻开放“任意 YAML 替换”，而是先解决一个更基础的问题：
- `project_name` 如果仍按多值（多个微服务）使用，平台就无法稳定判断应该改哪一组资源
- 因此在 `CD=ArgoCD` 场景下，先只允许单值 `project_name`

## 3. 范围
包含：
- `CD=ArgoCD` 发布建单时的 `project_name` 单值校验
- 前端与后端错误口径对齐
- GitOps 变量替换能力的后续设计前置说明

不包含：
- GitOps 替换规则落表
- 任意 YAML 路径编辑
- `project_name` 多值批量写回 YAML
- 多服务 ArgoCD 同步编排

## 4. 背景
当前 `CD=ArgoCD` 的后端执行逻辑是：
1. 根据应用与 `env` 定位 GitOps overlay
2. 解析 `image_version`
3. 修改 `kustomization.yaml -> images[0].newTag`
4. `git commit + git push`
5. 触发 ArgoCD Sync

这一版还没有进入：
- `ConfigMap`
- `Service`
- `Deployment.metadata.name`
- `namespace`
- 其他 YAML 字段

但后续如果要做这些替换，`project_name` 必须先是单值。

## 5. 核心规则
### 5.1 适用范围
当发布模板满足以下条件时，启用本版规则：
- 模板启用了 `CD`
- `CD` 执行器为 `ArgoCD`

### 5.2 project_name 单值规则
若本次发布参数中存在：
- `param_key = project_name`

则其值必须为单值，不允许出现典型多值格式：
- 英文逗号 `,`
- 中文逗号 `，`
- 换行

### 5.3 失败提示
当校验失败时，后端返回：
- `cd 选择 argocd 时，project_name 当前仅支持单值`

## 6. 发布建单校验
文件目标：
- `/Users/lingyunxieqing/Desktop/gos/internal/application/usecase/release_order.go`

### 6.1 校验时机
在创建发布单时：
1. 先解析模板
2. 再校验模板参数合法性
3. 再执行 `project_name` 单值校验
4. 通过后才写入发布单

### 6.2 设计原因
把校验放在“创建发布单”阶段，而不是“模板保存”阶段，有两个好处：
- 不强依赖 Jenkins 原始参数是否声明为单选
- 允许平台把原始可多选参数，在 ArgoCD 场景下按“单值使用”收口

## 7. 前端配合要求
前端需要配合做到：
- `CD=ArgoCD` 时，如果 `CI` 中有 `project_name`
- 页面展示为单值语义
- 提交前先拦截明显多值输入

后端仍保留最终校验，避免被绕过。

## 8. GitOps 变量替换现状
### 8.1 已实现
当前 GitOps 写回仍只支持：
- `kustomization.yaml`
- `images[0].newTag`
- 值来源：`image_version`

### 8.2 已实现的提交信息模版
当前 `提交信息模版`：
- 来源于 `gitops.commit_message_template`
- 支持 `{param_key}` 占位符
- 取值来源于本次发布流程参数与系统字段

### 8.3 GitOps 替换规则
本版新增模板级 `GitOps 替换规则`：
- 仅在 `CD=ArgoCD` 时生效
- 规则来源字段支持：
  - `CI` 模板中已勾选的标准平台 Key
  - 已启用的系统内置字段
- 规则目标字段来自 GitOps 仓库扫描出的 YAML 标量字段
- 每条规则保存：
  - `source_param_key`
  - `source_from`
  - `file_path_template`
  - `document_kind`
  - `document_name`
  - `target_path`
  - `value_template`

### 8.4 运行时执行
ArgoCD 执行时，平台会：
1. 先根据发布单参数和系统字段构建取值上下文
2. 渲染 `file_path_template` 与 `value_template`
3. 将规则翻译为具体 YAML 替换动作
4. 写回 GitOps 仓库并提交
5. 再执行原有的 `image_version -> kustomization.yaml -> newTag` 更新
6. 最后触发 ArgoCD Sync

### 8.5 当前约束
- `project_name` 在 `CD=ArgoCD` 场景下仍按单值处理
- 规则目标只支持 YAML 标量叶子节点
- 若规则引用的标准字段在本次发布中未取到值，则执行阶段失败并返回明确错误

## 9. 下一版预留设计
下一版建议继续增强：
- GitOps 规则的差异预览
- 多文件/多资源批量替换执行日志
- 针对 `image_version` 与通用规则的单次提交合并优化

## 10. 联调验收
### 10.1 正常场景
- `CD=ArgoCD`
- `project_name` 为单值
- 发布单创建成功

### 10.2 异常场景
- `CD=ArgoCD`
- `project_name` 为多值（例如 `a,b`）
- 创建发布单失败，并返回明确错误

### 10.3 非 ArgoCD 场景
- `CD=Jenkins` 或无 `CD`
- `project_name` 不受本版单值规则影响

## 11. 实现状态
本版当前实现状态：
- 已在发布单创建阶段增加 `project_name` 单值后端校验
- 已与前端的提示、交互和错误口径对齐
- 已新增模板级 `GitOps 替换规则` 持久化模型与接口
- 已支持扫描 GitOps YAML 标量字段作为规则目标
- 已支持 ArgoCD 执行阶段根据标准平台 Key 渲染并写回 YAML 字段
- 现有 `image_version` 写回仍保留，继续更新 `kustomization.yaml`
- 已新增 GitOps 应用目录智能匹配，支持 `app_key` 与 GitOps 目录名之间的 `_ / - / 环境后缀` 差异兼容
- 已支持模板编辑与字段选择前实时同步最新 YAML 候选字段，降低 GitOps 仓库变更后的误配风险
- 已取消发布记录展示权限限制，发布记录列表与详情查询不再按应用范围和创建者过滤
- 已保留发布动作权限控制，放开的仅是“查看发布记录”能力
- 已将 Jenkins 慢日志场景从“直接失败”调整为“超时重试”，减少日志流因慢响应中断和丢失

## 12. 风险与说明
- 该规则会影响历史上依赖 `project_name` 多选的 ArgoCD 模板/发布方式
- 若旧发布单参数中保存的是多值 `project_name`，基于该参数直接复用创建的新单可能会被校验拦住
- 这是有意为之，用来防止平台在尚未实现多资源批量替换前误改 GitOps 配置

## 13. 今日补充
今日后端进度补充如下：
- 发布记录查询链路已改为“所有已登录用户可查看”，不再要求 `release.view` 或应用范围授权
- GitOps YAML 扫描链路已支持更宽松的应用目录匹配，解决实际仓库目录与平台 `app_key` 不完全一致时扫描不到字段的问题
- GitOps 字段候选会在模板编辑阶段实时刷新，执行阶段也会按最新路径规则解析真实目标文件
- Jenkins 日志流对慢接口已改为超时重试，避免因 `logText/progressiveText` 慢响应导致日志流报错和丢失
