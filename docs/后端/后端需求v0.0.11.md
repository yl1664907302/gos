## 2. 本版目标
在 `v0.0.10` 已具备 `ArgoCD/GitOps` 基础能力、`GitOps 定位字段`、以及 YAML 替换规则雏形的基础上，正式把 `Helm` 场景从“改渲染后的 Kubernetes YAML”收口为“改 Helm values 文件”。

本版目标是把平台职责定清楚：
- 平台不直接操作 Helm 渲染后的 `Deployment/Service/ConfigMap`
- 平台只负责修改 GitOps 仓库中的 Helm `values` 文件
- ArgoCD 继续负责 `sync`
- Helm 继续负责渲染最终资源

## 3. 范围
包含：
- `CD=ArgoCD` 场景下新增 `GitOps 类型 = helm`
- GitOps 规则从 `YAML 字段` 扩展为 `values 路径`
- 发布执行阶段改为写回 `values` 文件并触发 ArgoCD Sync
- 对 `image_version`、`env`、普通标准字段的取值来源做清晰约束
- ArgoCD Helm 应用定位逻辑调整
- CD 步骤改为 GitOps 风格并补齐失败说明

不包含：
- 平台直接执行 `helm upgrade`
- 平台渲染 Helm chart 后再修改渲染结果 YAML
- Helm Chart 创建/删除能力
- Chart 仓库凭据托管
- values 深层复杂结构的可视化批量编辑器

## 4. 背景
当前 `CD=ArgoCD` 的已有实现更偏 `Kustomize`：
1. 根据应用与 `env` 定位 GitOps overlay
2. 解析 `image_version`
3. 修改 `kustomization.yaml -> images[0].newTag`
4. `git commit + git push`
5. 触发 ArgoCD Sync

但是当 GitOps 资产本身是 Helm 时，继续围绕渲染后的 Kubernetes YAML 做映射，会出现几个问题：
- 平台修改的不是源数据，容易和 Helm 模板职责打架
- YAML 结构是渲染结果，不适合作为长期稳定的配置入口
- 同一个 values 改动可能映射到多个最终资源，直接改最终 YAML 会失真

因此 Helm 场景下的正确模型应当是：
- 平台修改 `values.yaml / values-{env}.yaml / 其他受控 values 文件`
- ArgoCD 读取最新 Git 提交
- Helm 根据 values 渲染并部署

## 5. 核心概念
### 5.1 GitOps 类型
模板的 `CD=ArgoCD` 时，新增 `gitops_type`：
- `kustomize`
- `helm`

说明：
- `kustomize` 继续沿用现有 `kustomization.yaml` / YAML 字段规则能力
- `helm` 为本版新增重点

### 5.2 Helm values 规则
Helm 模式下，规则目标不再是 `YAML 字段`，而是：
- `values_file_template`
- `values_path`
- `value_template`

即：
- 改哪个 values 文件
- 改 values 文件中的哪个路径
- 改成什么值

本版已收口为：
- Helm 模式目标文件统一固定为：
  - `platform.values-{env}.yaml`
- 平台当前不再面向业务 values 文件做通用写回

## 6. 数据模型
### 6.1 发布模板
文件目标：

模板的 CD 配置需要新增：

- `gitops_values_rules`

### 6.2 Helm values 规则结构
当前规则表仍复用统一 GitOps 规则结构，但 Helm 模式实际使用字段收敛为：
- `source_param_key`
- `source_from`
- `values_file_template`
- `values_path`
- `value_template`
- `required`
- `sort_no`

字段说明：
- `source_param_key`：来源标准字段，例如 `image_version`
- `source_from`：来源类型，可为 `builtin / ci / manual / fixed`
- `values_file_template`：目标 values 文件模板，例如 `values-{env}.yaml` 或 `{project_name}.values-{env}.yaml`
- `values_path`：values 键路径，例如 `image.tag`
- `value_template`：最终写入值模板，例如 `{image_version}`
- `required`：执行时是否必须取到值

本版实际约束：
- `values_file_template` 在 Helm 模式下只能是：
  - `apps/<应用目录>/helm/platform.values-{env}.yaml`
- 前端不再允许用户自定义该文件路径

### 6.3 不再把 Helm 规则建模为 YAML 字段规则
当 `gitops_type=helm` 时，不再要求：
- `document_kind`
- `document_name`
- `target_path`（Kubernetes 资源路径）

Helm 模式下的目标路径统一为：
- `values_path`

## 7. 规则来源与取值
### 7.1 取值上下文
发布执行时，平台需要构建一张上下文表，用于渲染：
- `values_file_template`
- `value_template`
- 提交信息模版

上下文字段来源包括：
- 基础字段：
  - `env`
  - `application_id`
  - `app_key`
  - `app_name`
- 内置字段：
  - `image_version`
- 发布参数快照中的标准 Key：
  - 如 `project_name`
  - `branch`
  - `namespace`
  - 其他普通标准字段

### 7.2 image_version 规则
当前规则继续沿用：
- `image_version` 为平台内置字段
- Jenkins CI 下默认取本次构建号 `BUILD_NUMBER`

说明：
- Helm 场景也不要求 CI 一定显式声明 `image_version` 参数
- 只要求执行阶段能够稳定解析出 `image_version`
- 当前已实现：
  - 若发布参数中没有 `image_version`
  - 后端会尝试从 Jenkins `buildURL` 中解析 `BUILD_NUMBER`
  - 详情页 `CD 取值进度` 会展示该值来源为 `jenkins_build_number`

### 7.3 env 规则
- `env` 继续来自发布单基础字段 `env_code`
- 不是从 CI 参数兜底反推出来的主来源
- 若模板参数里也存在 `env` / `env_code` 映射，仅作为发布参数快照中的附加值参与渲染

## 8. 模板配置规则
### 8.1 CD=ArgoCD 且 gitops_type=helm
模板保存时，需要校验：
- 模板已启用 `CD`
- `CD` 执行器为 `ArgoCD`
- `gitops_type = helm`
- 模板存在至少一条 Helm values 规则
- 基础字段中必须包含 `env`

### 8.2 规则来源字段
每条规则的来源字段可来自：
- `CI` 已勾选的标准平台 Key
- 系统内置字段
- 后续扩展的手工字段/固定字段

### 8.3 Values 候选选择
模板配置阶段允许用户：
- 基于实时扫描结果选择 `values_path`

当前收口规则：
- 前端不再渲染整块 YAML/values 结构
- 后端提供轻量的 `values_path` 候选列表
- 候选仅来自：
  - `platform.values-{env}.yaml`

## 9. 执行阶段
文件目标：
- `/Users/lingyunxieqing/Desktop/gos/internal/application/usecase/release_order_argocd.go`
- `/Users/lingyunxieqing/Desktop/gos/internal/infrastructure/gitops/service.go`

### 9.1 执行流程
当发布单进入 `CD=ArgoCD` 且 `gitops_type=helm`：
1. 根据 `app_key` 确定应用 GitOps 根目录
2. 构建本次发布的取值上下文
3. 逐条渲染 `values_file_template`
4. 逐条渲染 `value_template`
5. 打开目标 values 文件
6. 按 `values_path` 写入值
7. 写回 Git 仓库
8. `git commit + git push`
9. 触发 ArgoCD Sync
10. 等待 Sync / Health 状态

### 9.2 文件写入规则
Helm 模式下仅允许修改：
- `platform.values-{env}.yaml` 中的标量字段

首版不支持：
- 数组批量追加
- map merge
- 动态新增复杂对象
- 多层复杂 patch 语法

### 9.3 错误处理
执行失败时，需区分以下错误：
- `ArgoCD Application 不存在`
- `values 文件不存在`
- `values_path 不存在`
- `required 规则的 source_param_key 未取到值`
- `git commit / push` 失败
- `ArgoCD sync` 失败

错误需记录到：
- 发布步骤
- CD 执行进度
- 发布详情日志/说明

当前已实现：
- 若 ArgoCD Application 在启动前即失败，CD 不再空白挂起
- 步骤会落成失败，并在详情页明确展示失败原因

## 10. GitOps 服务能力
文件目标：
- `/Users/lingyunxieqing/Desktop/gos/internal/infrastructure/gitops/service.go`

本版 GitOps 服务需要新增能力：
- 读取 Helm values 样板文件并扫描所有标量叶子路径
- 根据 `values_file_template` 渲染出真实文件路径
- 根据 `values_path` 执行受控写值
- 保持直接仓库模式，继续在本地 GitOps 仓库目录中直接修改

说明：
- 不需要调用 `helm template`
- 不需要调用 `helm upgrade`
- 平台只负责 values 文件写回

当前已实现：
- `GET /gitops/values-candidates`
- 实时扫描应用 Helm 目录中的 values 候选
- Helm 模式下仅返回 `platform.values-{env}.yaml` 的路径候选

## 13. 联调验收
### 13.1 正常场景
- 模板 `CD=ArgoCD`
- `gitops_type=helm`
- values 规则配置完整
- 发布执行成功写回 values 文件
- Git 提交成功
- ArgoCD Sync 成功

### 13.2 缺值场景
- `source_param_key` 在本次发布中未取到值
- 执行失败，并给出明确错误

### 13.3 文件不存在场景
- 渲染出的 `values_file_template` 对应真实文件不存在
- 执行失败，并记录目标文件路径

## 14. 风险与说明
- Helm 模式下，values 文件结构必须相对稳定，否则模板规则会快速失效
- 当前平台只操作 `platform.values-{env}.yaml`，业务 values 文件仍需由研发/运维维护
- 为降低复杂度，首版建议先围绕标量字段写回，不要一开始支持复杂对象合并
- ArgoCD Application 需要提前存在；若不存在，当前 CD 会在启动阶段失败并提示 `404`
