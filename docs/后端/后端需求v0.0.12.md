# 后端需求文档 v0.0.12（基于 v0.0.11 增量）

## 1. 文档信息
- 文档版本：`v0.0.12`
- 基线日期：`2026-03-20`
- 上一版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端/后端需求v0.0.11.md`
- 本版主题：`多 ArgoCD / 多 K8s 集群支持与环境自动绑定`
- 当前状态：`已落地并持续迭代`

## 2. 本版目标
在 `v0.0.11` 已具备单 ArgoCD、GitOps、Helm values 写回能力的基础上，扩展平台为支持：
- 同时接入多个 ArgoCD 实例
- 每个 ArgoCD 实例代表一个独立 K8s 集群或独立控制面
- ArgoCD 配置不再仅依赖配置文件，而是入库管理
- GitOps 配置不再仅依赖配置文件，而是以实例形式入库管理
- ArgoCD 实例可关联具体 GitOps 实例
- 发布模板在编辑时根据 `env` 自动绑定 ArgoCD 执行器，而不是人工逐个选择

## 3. 范围
包含：
- 多 ArgoCD 实例数据模型
- ArgoCD 配置入库
- GitOps 实例入库
- 前端新增 ArgoCD 实例管理能力
- `env -> ArgoCD 实例` 自动绑定规则
- `ArgoCD 实例 -> GitOps 实例` 关联规则
- 发布执行时按环境自动解析目标 ArgoCD
- ArgoCD 管理模块从“单实例状态页”升级为“实例 + 应用”双层模型

不包含：
- 多租户 ArgoCD 权限隔离
- 跨 ArgoCD 应用迁移
- 自动创建 ArgoCD Project
- 多云多区域流量治理

## 4. 背景
当前平台的 ArgoCD 能力默认只有单实例假设：
- 配置来自 `config.local.json / config.production.json`
- 启动时只初始化一个 ArgoCD client
- 应用同步、详情查询、手动 sync 都默认对同一个 ArgoCD 实例生效

当发布目标扩展到多个 K8s 集群后，会出现以下问题：
- 无法区分 `dev/test/prod` 应该走哪个 ArgoCD
- 前端无法管理多个 ArgoCD 地址与凭据
- 发布模板无法自动根据环境绑定正确集群
- 发布记录缺少“本次实际命中的 ArgoCD 实例”审计信息

因此本版需要把 ArgoCD 从“全局单例组件”升级为“可管理的集群实例资源”，并把 GitOps 从“全局配置”升级为“可关联的仓库实例资源”。

## 5. 核心概念
### 5.1 ArgoCD 实例
新增“ArgoCD 实例”概念，表示一个独立的 ArgoCD 控制面。

每个实例至少包含：
- `instance_code`
- `name`
- `base_url`
- `auth_mode`
- `token`
- `gitops_instance_id`
- `cluster_name`
- `env_scope`
- `status`

### 5.2 环境绑定规则
模板不再显式选择某个 ArgoCD 绑定记录，而是通过环境自动解析。

建议规则：
- `env=dev` -> 命中 `dev` 对应 ArgoCD 实例
- `env=test` -> 命中 `test` 对应 ArgoCD 实例
- `env=prod` -> 命中 `prod` 对应 ArgoCD 实例

首版建议采用：
- 一个环境默认绑定一个 ArgoCD 实例
- 若后续一个环境有多个实例，再扩展权重/优先级规则

### 5.3 ArgoCD 执行器
`CD=ArgoCD` 仍保留为统一执行器类型，但不再依赖单一全局配置。

执行时需先解析：
- 本次发布环境
- 命中的 ArgoCD 实例
- 再用该实例执行 `application get / sync / health check`

## 6. 数据模型
### 6.1 argocd_instance
建议新增表：`argocd_instance`

字段建议：
- `id`
- `instance_code`
- `name`
- `base_url`
- `auth_mode`
- `token_ciphertext`
- `cluster_name`
- `default_namespace`
- `status`
- `health_status`
- `last_check_at`
- `created_at`
- `updated_at`
- `remark`

说明：
- `token` 不应明文存储，建议加密保存
- `health_status` 用于前端列表快速展示实例可用性

### 6.2 argocd_env_binding
建议新增表：`argocd_env_binding`

字段建议：
- `id`
- `env_code`
- `argocd_instance_id`
- `priority`
- `status`
- `created_at`
- `updated_at`

说明：
- 首版可限制每个 `env_code` 仅允许一条启用绑定
- `priority` 预留给后续多实例同环境扩展

### 6.3 gitops_instance
建议新增表：`gitops_instance`

字段建议：
- `id`
- `instance_code`
- `name`
- `local_root`
- `default_branch`
- `author_name`
- `author_email`
- `commit_message_template`
- `status`
- `created_at`
- `updated_at`
- `remark`

说明：
- GitOps 实例提供工作目录、Git 凭据与提交身份
- 首版按 `ArgoCD 实例 -> GitOps 实例` 建立一对一默认关联
- 运行时由 ArgoCD Application 决定具体 `repoURL / sourcePath`，由 GitOps 实例提供仓库访问能力

### 6.4 argocd_application
现有 `argocd_application` 需补充实例维度：
- `argocd_instance_id`

说明：
- 相同 `app_name` 在不同 ArgoCD 实例中应允许并存
- 唯一键建议升级为：`argocd_instance_id + app_name`

### 6.5 release_order / execution 扩展
发布执行相关数据建议补充：
- `release_order_execution.argocd_instance_id`
- `release_order_execution.gitops_instance_id`
- 或在执行快照表中记录 `argocd_instance_code`

用于审计：
- 本次 CD 实际命中了哪个 ArgoCD
- 本次 CD 实际使用了哪个 GitOps 实例

## 7. 配置来源调整
### 7.1 配置文件职责收缩
`configs/*.json` 中原有单实例 ArgoCD 配置收缩为：
- 是否启用 ArgoCD 模块
- 启动检查默认开关
- 定时同步默认开关

不再承载：
- 具体实例地址
- 具体 token
- 具体 cluster 绑定

### 7.2 实例配置入库
前端新增/编辑 ArgoCD 实例时，后端将实例配置写入数据库。

后端职责：
- 校验 `base_url` 唯一性
- 测试连通性
- 加密保存 token
- 支持启停实例

GitOps 实例同样由前端管理并写入数据库。

后端职责：
- 校验实例编码唯一
- 校验工作目录合法
- 支持启停实例
- 支持按实例读取 Git 状态

## 8. 管理接口
### 8.1 ArgoCD 实例管理
建议新增接口：
- `GET /argocd/instances`
- `POST /argocd/instances`
- `PUT /argocd/instances/:id`
- `POST /argocd/instances/:id/check`
- `POST /argocd/instances/:id/enable`
- `POST /argocd/instances/:id/disable`

### 8.2 GitOps 实例管理
建议新增接口：
- `GET /gitops/instances`
- `POST /gitops/instances`
- `PUT /gitops/instances/:id`
- `POST /gitops/instances/:id/enable`
- `POST /gitops/instances/:id/disable`

### 8.3 环境绑定管理
建议新增接口：
- `GET /argocd/env-bindings`
- `PUT /argocd/env-bindings`

说明：
- 首版可做成“按环境维护唯一实例映射”的简单模型

### 8.4 应用同步接口
现有应用同步接口需增加实例维度：
- 同步时明确实例 ID
- 详情查看时明确实例 ID

## 9. 发布模板规则
### 9.1 模板编辑
当模板 `CD` 走 ArgoCD 时：
- 不再让用户手动选择 ArgoCD 执行器实例
- 模板只保留 `CD=ArgoCD` 与 `gitops_type`
- 实际执行实例由 `env` 自动解析

### 9.2 校验规则
模板保存时需要校验：
- 基础字段中存在 `env`
- 当前环境绑定表中，至少为模板可能使用的环境准备了默认 ArgoCD 实例

首版可简化为：
- 若系统尚未配置任何 `argocd_env_binding`，则模板保存提示管理员先完成环境绑定

## 10. 执行阶段
### 10.1 实例解析流程
`CD=ArgoCD` 执行时：
1. 读取发布单基础环境 `env_code`
2. 查找 `argocd_env_binding`
3. 解析出唯一启用的 ArgoCD 实例
4. 读取该 ArgoCD 实例关联的 GitOps 实例
5. 基于 ArgoCD 实例初始化 client
6. 基于 GitOps 实例准备 Git 工作目录、凭据与提交身份
7. 执行 GitOps 写回、ArgoCD sync、健康检查

### 10.2 错误处理
需要明确区分：
- 当前环境未绑定 ArgoCD 实例
- 当前环境绑定了多个启用实例且无法判定唯一目标
- ArgoCD 实例已停用
- ArgoCD 实例未关联 GitOps 实例
- GitOps 实例已停用
- ArgoCD 实例鉴权失败
- 应用不存在于该实例

这些错误需落到：
- 发布步骤
- 执行单元消息
- 发布详情页错误说明

## 11. ArgoCD 管理模块升级
### 11.1 当前模块问题
当前页面更像“单实例应用列表”，不适合多实例场景。

### 11.2 升级方向
升级后建议拆成三层：
1. `ArgoCD 实例`
2. `环境绑定`
3. `ArgoCD 应用`

实例层展示：
- 名称
- 地址
- 集群名
- 关联 GitOps 实例
- 环境范围
- 健康状态
- 最后检查时间

应用层展示：
- 所属实例
- 应用名
- project
- sync 状态
- health 状态
- 原始链接

## 13. 实施建议
### 一期
- `argocd_instance` 入库

## 14. 结论
`v0.0.12` 的核心不是“再加一个 ArgoCD 页面”，而是把平台的 ArgoCD 模型从单实例升级为多集群可治理模型。

最终目标是：
- ArgoCD 实例可配置、可管理、可审计
- GitOps 实例可配置、可管理、可关联
- 模板配置尽量少人工选择
- 发布执行时由 `env` 自动命中正确集群，并通过 ArgoCD 实例自动解析对应 GitOps 仓库能力
- 发布记录中能回溯本次到底发布到了哪个 ArgoCD / K8s 集群

## 15. 当前落地进度（2026-03-23）
当前后端已经从“多实例设计”继续推进到“执行稳定性与运行审计”层面。

### 15.1 已落地
- 多实例模型
  - `ArgoCD 实例`、`GitOps 实例` 已入库管理。
  - `env -> ArgoCD 实例 -> GitOps 实例` 的运行时解析链路已落地。
- 默认实例兼容
  - 启动时会根据配置文件自动补齐默认 ArgoCD / GitOps 实例，兼容旧环境。
- 环境绑定
  - 发布执行时不再依赖模板手工选实例，而是根据 `env` 自动命中。
- GitOps 执行稳态增强
  - 已补 repo + branch 级锁，避免同仓库并发发布互相踩踏。
  - 已补 `git push` 冲突后的 `fetch/rebase/retry`。
  - 已补 `.git/index.lock` 等过期锁文件自动恢复。
- ArgoCD 执行链路
  - `CD` 已细化为：
    - `更新 Helm Values`
    - `Git 提交`
    - `Git 推送`
    - `触发 ArgoCD`
    - `健康检查`
  - 管线阶段会随步骤实时同步，不再只在结束时一次性回写。
- 发布编排修正
  - 已补“同一时刻只允许一个 execution 运行”的保护，避免 `CI` 未结束就提前启动 `CD`。
  - 已补执行单元和全局完成步骤的自动收尾逻辑，减少旧脏状态。
- CD 自填字段
  - `GitOpsRuleSourceCDInput` 已支持。
  - 当规则来源为 `cd_input` 时，运行时直接使用模板固定值，不再强制要求来自 `CI` 参数映射。
- 启动健壮性
  - schema 初始化超时窗口已从 `5s` 放宽到 `30s`，降低远程 MySQL 启动阶段超时失败概率。

### 15.2 当前已知边界
- 旧订单如果已经在旧逻辑下失败，不会因为代码修复自动回放，通常仍需要重新发单验证。
- Helm 模式当前仍按平台收口：
  - 只写 `platform.values-{env}.yaml`
  - 不直接改业务 values 文件
- Kustomize 路线当前未继续推进，仍以“暂不支持”为主。

### 15.3 下一步建议
- 为运行中服务增加明确的构建标识或版本日志，避免再次误判“源码已改但进程仍是旧版本”。
- 继续补足 GitOps / ArgoCD 失败场景的结构化日志，提升线上排障效率。
- 评估把 Application 与 GitOps 实例的关联粒度从“实例级”进一步细化到“应用级”。
