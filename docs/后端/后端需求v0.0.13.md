- 基线日期：`2026-03-24`

- 本版主题：`发布恢复模型重构：标准回滚与 CD 管线参数重放`
- 当前状态：`联调中`

## 2. 本版目标
重构当前错误的回滚模型，不再使用“按应用查最近成功单并复制参数”的方式，而是建立两种明确的恢复路径：
- `标准回滚`
- `重放回滚`

## 3. 当前问题
当前回滚入口：
- `/Users/lingyunxieqing/Desktop/gos/internal/interfaces/http/release_order_handler.go:299`
- `/Users/lingyunxieqing/Desktop/gos/internal/application/usecase/release_order.go:313`

现状问题：
1. 按 `application_id` 查询最近成功单，而不是由用户明确选择来源成功单。
2. 没有按 `env / 执行器 / 部署目标` 严格限定来源。
3. 本质上只是复制参数重新建单，不是真正恢复历史部署状态。
4. 无法区分：
   - `ArgoCD + Helm` 的真正回滚
   - `CD=管线` 的历史参数重放
5. 无法审计“本次恢复到底恢复了什么”。

## 4. 目标模型
### 4.1 标准回滚
适用范围：
- `CD = ArgoCD`
- `gitops_type = helm`

定义：
- 平台基于一张明确指定的成功发布单，读取该单对应的历史部署快照。
- 创建一张新的 `rollback` 类型发布单。
- 该发布单仅包含 `CD execution`，不包含 `CI execution`。
- 执行时重新写入历史成功单对应的 Helm values 结果，并执行 GitOps -> ArgoCD Sync -> 健康检查。

### 4.2 重放回滚
适用范围：
- `CD = 管线`
- `仅 CI`

定义：
- 平台基于一张明确指定的成功发布单，优先读取与恢复作用域一致的历史参数快照：
  - 若存在 `CD` 参数快照，则优先重放 `CD`
  - 若来源单仅有 `CI`，或存在 `CD` 执行单元但缺失 `CD` 参数快照，则回退重放 `CI`
- 创建一张新的 `replay` 类型发布单。
- 该发布单仅包含目标作用域的 execution，不强制要求同时创建 `CI + CD`。
- 执行时以历史成功单对应作用域参数重新触发一次执行链路。

说明：
- 这不是标准回滚。
- 该动作的本质是：`历史成功参数重放`。

## 5. 恢复类型定义
建议新增：
- `release_order.operation_type`

枚举：
- `deploy`
- `rollback`
- `replay`

说明：
- 普通发布：`deploy`
- 标准回滚：`rollback`
- 重放回滚：`replay`

同时新增：
- `release_order.source_order_id`
- `release_order.source_order_no`

用于记录本次恢复来源于哪张成功发布单。

## 6. 来源单选择规则
恢复类动作必须满足：
1. 来源单必须由用户明确指定。
2. 来源单必须为 `success`。
3. 来源单必须与当前目标属于同一应用。
4. 对标准回滚，来源单必须为 `CD = ArgoCD`。
5. 对 `replay`，来源单必须满足以下之一：
   - `CD = 管线`
   - `仅 CI`

废弃当前模式：
- `CreateRollbackByApplication(applicationID)` 自动挑最近成功单。

建议改成：
- `CreateStandardRollbackByOrder(sourceOrderID, operator)`
- `CreatePipelineReplayByOrder(sourceOrderID, operator)`

## 7. 标准回滚的数据依据
### 7.1 为什么不能只复制参数
对于 `ArgoCD + Helm` 场景，真正需要恢复的是：
- 当次写入 GitOps 的最终 values 结果
- 当次命中的 ArgoCD / GitOps / repo / branch / path
- 当次最终部署成功的镜像版本与关键部署字段

因此不能只依赖 `release_order_param`。

### 7.2 部署快照
建议新增表：`release_order_deploy_snapshot`

字段建议：
- `id`
- `release_order_id`
- `provider`
- `gitops_type`
- `argocd_instance_id`
- `gitops_instance_id`
- `argocd_app_name`
- `repo_url`
- `branch`
- `source_path`
- `env_code`
- `snapshot_payload_json`
- `created_at`

其中 `snapshot_payload_json` 对于 Helm 建议至少保存：
- `values_file_template`
- 实际命中的 values 文件
- 本次写入的 `values_path -> value` 映射
- 关键镜像版本

示例：
```json
{
  "file": "apps/java-nantong-test/helm/platform.values-dev.yaml",
  "values": {
    "image.tag": "471",
    "test.enabled": false
  }
}
```

## 8. 参数重放的数据依据
对于 `replay`，不新增部署快照，直接复用历史成功单的参数快照：
- 优先使用 `pipeline_scope = cd`
- 若不存在 `CD` 参数快照，则回退使用 `pipeline_scope = ci`

必要条件：
- 必须至少能找到一组可重放参数快照
- 若 `CD execution` 存在但缺失 `CD` 参数快照，应自动尝试使用 `CI` 参数快照
- 若 `CI / CD` 都缺失参数快照，则不允许发起重放回滚

## 9. 新建恢复单的执行模型
### 9.1 标准回滚单
新建订单时：
- `operation_type = rollback`
- `source_order_id / source_order_no` 记录来源
- 仅创建 `CD execution`
- `CI execution` 不创建
- 步骤使用 ArgoCD CD 步骤：
  - `gitops_update`
  - `git_commit`
  - `git_push`
  - `argocd_sync`
  - `health_check`

### 9.2 重放回滚单
新建订单时：
- `operation_type = replay`
- `source_order_id / source_order_no` 记录来源
- 若命中 `CD`：仅创建 `CD execution`
- 若命中 `CI`：仅创建 `CI execution`
- 参数从来源成功单对应作用域的参数快照复制
- 同时复制属于标准字库的参数，供提交模板、取值进度与恢复链路复用

## 10. 执行流程
### 10.1 标准回滚
1. 校验来源单
2. 读取部署快照
3. 新建 `rollback` 单
4. 仅启动 `CD execution`
5. 重新写入快照值到 Helm values
6. Git commit / push
7. ArgoCD sync
8. 健康检查
9. 收尾订单状态

### 10.2 重放回滚
1. 校验来源单
2. 优先读取 `CD` 参数快照，缺失时回退读取 `CI` 参数快照
3. 新建 `replay` 单
4. 仅启动目标作用域 execution
5. 组装历史参数
6. 触发对应作用域执行链路
7. 等待执行结束
8. 收尾订单状态

## 11. 接口调整
### 11.1 废弃接口
建议废弃：
- `POST /applications/:id/release-orders/rollback`

原因：
- 该接口按应用回滚，不满足“基于指定成功单恢复”的要求。

### 11.2 新接口
建议新增：
- `POST /release-orders/:id/rollback`
- `POST /release-orders/:id/replay`

语义：
- `:id` 为来源成功单 ID
- `rollback` 仅用于 `CD = ArgoCD`
- `replay` 用于 `CD = 管线` 与 `仅 CI`

返回：
- 新建恢复单的 `release_order`

## 12. 校验规则
### 12.1 标准回滚校验
- 来源单必须成功
- 来源单必须有 `CD execution = argocd`
- 来源单必须有部署快照
- 快照中的 ArgoCD / GitOps 实例当前仍然可用
- 来源单与目标环境必须一致

### 12.2 参数重放校验
- 来源单必须成功
- 来源单必须存在可重放执行单元
- 若来源单有 `CD execution = pipeline`，优先重放 `CD`
- 若来源单仅有 `CI`，或 `CD` 缺少参数快照，则允许回退重放 `CI`
- 目标作用域对应的管线当前仍然存在且启用

## 13. 审计要求
恢复类订单必须可审计：
- 恢复类型：`rollback / replay`
- 来源单号
- 来源镜像版本
- 来源执行器
- 实际命中的重放作用域：`ci / cd`
- 恢复时刻的执行快照
- 执行结果

日志建议增加：
- `rollback_create_start / success / failed`
- `pipeline_replay_create_start / success / failed`
- `rollback_source_selected`
- `replay_source_selected`

## 14. 兼容与迁移
### 14.1 旧“回滚”入口
短期可保留旧按钮，但后端需改为：
- 不再按应用最近成功单自动选择
- 而是显式提示用户从成功单发起恢复

### 14.2 历史成功单
对于历史成功单：
- 若缺少 `release_order_deploy_snapshot`，则不支持标准回滚
- 可在 UI 中提示：`该历史单缺少部署快照，暂不支持标准回滚`

对 `replay`：
- 只要保留了 `CD` 或 `CI` 任一组可重放参数快照即可支持

## 15. 实施建议
### 一期
- 引入 `operation_type`
- 引入 `source_order_id / source_order_no`
- 新增基于成功单的 `rollback / replay` 接口
- 停用按应用自动找最近成功单的回滚模式
- `replay` 允许 `CI-only` 成功单创建恢复单

### 二期
- 落地 `release_order_deploy_snapshot`
- 标准回滚切换为快照驱动
- `replay` 改为优先复制 `CD` 参数快照，缺失时回退复制 `CI` 参数快照
- 同时复制来源单中的标准字库参数，避免 `project_name / branch / image_version` 等 CI 产出字段在恢复单中丢失

### 三期
- 恢复类发布单详情页专项展示
- 历史快照缺失场景补救提示
- 后台审计视图补充“来源单 -> 恢复单”链路

## 16. 结论
`v0.0.13` 的关键原则是：
- `ArgoCD` 只做真正的 `标准回滚`
- `CD 管线 / 仅 CI` 统一走 `重放回滚`

后端必须停止继续使用“按应用找最近成功单并复制参数”的粗粒度逻辑，改为：
- 基于明确来源成功单
- 基于明确恢复类型
- 基于明确快照/参数依据
- 基于实际可重放作用域选择 `CI` 或 `CD`
