# 后端需求文档 v0.0.15（基于 v0.0.13 增量）

## 1. 文档信息
- 文档版本：`v0.0.15`
- 基线日期：`2026-03-25`
- 上一版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端/后端需求v0.0.13.md`
- 本版主题：`阶段一：可靠发布基础能力建设`
- 当前状态：`规划中`

## 2. 阶段目标
阶段一重点不再扩展新的发布模型，而是围绕“可靠发布”补齐基础能力：
- 环境当前版本台账
- 发布前预检
- 并发发布控制与环境锁
- 通知投递闭环

## 3. 本版范围
### 3.1 纳入范围
- 环境当前版本聚合视图
- 发布前预检接口
- 并发发布配置与锁模型
- 执行期环境锁控制
- 通知记录落库与查询

### 3.2 暂不纳入
- 审批流
- 时间窗/冻结期
- 自动回滚
- 蓝绿/金丝雀/分批发布
- 发布指标大盘

## 4. 当前问题
当前平台在“可靠发布”维度存在以下缺口：
1. 平台缺少某应用某环境“当前版本”事实台账。
2. 发布动作在执行阶段才暴露很多问题，缺少统一预检。
3. 并发控制主要依赖执行过程中的临时冲突，缺乏统一配置和锁模型。
4. 通知能力缺失，用户只能靠前端页面轮询结果。

## 5. 目标能力一：环境当前版本台账
## 5.1 目标
平台需要沉淀每个：
- 应用
- 环境
- 执行模型（ArgoCD / 管线 / CI-only）

对应的当前事实状态，至少包括：
- 当前版本
- 当前版本类型（镜像版本 / git_ref / git_sha）
- 当前成功单号
- 当前成功时间
- 当前健康状态
- 当前锁状态

## 5.2 数据来源建议
### ArgoCD
优先来源：
- `release_order_deploy_snapshot`
- 当前 ArgoCD Application 状态

### CD 管线
优先来源：
- 最近成功的 `CD execution`
- 对应 `release_order_param` 与执行结果快照

### CI-only
优先来源：
- 最近成功的 `CI execution`
- 对应参数快照

用途：

## 6. 目标能力二：发布前预检
## 6.1 目标
发布前预检需要在正式执行前回答：
- 模板是否完整
- 参数是否齐全
- 执行器是否可用
- GitOps/ArgoCD 是否可用
- 环境是否被锁定
- 当前是否存在并发发布冲突

## 6.2 预检接口建议
新增：
- `POST /release-orders/precheck`
- `POST /release-orders/:id/precheck`

说明：
- 对新建发布单场景，可基于输入参数预检
- 对已创建待执行发布单场景，可基于 `order_id` 预检

## 6.3 预检返回结构建议
- `summary_status`
- `items[]`
- `items[].code`
- `items[].title`
- `items[].status`
- `items[].message`
- `items[].blocking`

建议至少包含以下检查项：
- `template_complete`
- `required_params_ready`
- `executor_available`
- `gitops_available`
- `argocd_application_exists`
- `env_lock_available`
- `concurrency_available`

## 6.4 预检执行时机
- 手动点发布前
- 重放回滚前
- 标准回滚前
- 后端 `execute` 入口也必须再次执行轻量预检，防止前端绕过

## 7. 目标能力三：并发发布控制与环境锁
## 7.1 目标
平台要把“并发发布控制”从临时冲突处理，升级为显式配置能力。

## 7.2 配置模型
建议新增系统级配置：`release_concurrency_settings`

字段建议：
- `enabled`
- `lock_scope`
- `conflict_strategy`
- `lock_timeout_sec`
- `allow_admin_override`
- `updated_by`
- `updated_at`

枚举建议：
### `lock_scope`
- `application`
- `application_env`
- `gitops_repo_branch`

### `conflict_strategy`
- `reject`
- `queue`

## 7.3 锁表建议
建议新增：`release_execution_lock`

字段建议：
- `id`
- `lock_scope`
- `lock_key`
- `application_id`
- `env_code`
- `release_order_id`
- `release_order_no`
- `status`
- `owner_type`
- `created_at`
- `expired_at`
- `released_at`

说明：
- `lock_key` 根据配置策略生成，例如：
  - 应用级：`app:{application_id}`
  - 应用+环境级：`app:{application_id}:env:{env_code}`
  - GitOps 仓库分支级：`repo:{repo_url}:branch:{branch}`

## 7.4 运行规则
### `reject`
- 若命中冲突锁，直接阻断执行
- 发布单状态保持待执行或失败，并记录原因

### `queue`
- 若命中冲突锁，不立即失败
- 发布单进入 `waiting_for_lock`
- 等待前一单释放锁后再继续执行

## 7.5 锁释放时机
以下情况需要释放锁：
- 发布成功
- 发布失败
- 发布取消
- 锁超时
- 管理员强制释放

## 7.6 管理能力
建议新增接口：
- `GET /system/release-concurrency`
- `PUT /system/release-concurrency`
- `GET /release-locks`
- `POST /release-locks/:id/release`

## 8. 目标能力四：通知闭环
## 8.1 目标
平台需提供发布结果通知，至少覆盖：
- 发布开始
- 发布成功
- 发布失败
- 标准回滚成功/失败
- 重放回滚成功/失败

## 9. 执行链路调整
## 9.1 execute 前置逻辑
后端在真正进入执行前，需要新增以下顺序：
1. 轻量预检
2. 并发锁申请
3. 申请成功才进入执行
4. 执行结束后释放锁
5. 写运行事实表
6. 触发通知

## 9.2 回滚/重放
- 标准回滚也必须走预检与锁申请
- 重放回滚也必须走预检与锁申请
- 不允许恢复类动作绕过并发控制

## 10. 状态扩展建议
为适配并发发布与预检，建议增加或规范化以下状态：
- `waiting_for_lock`
- `precheck_failed`
- `lock_conflicted`
- `lock_released`

若不愿新增主表状态，也至少要在：
- `release_order_step`
- `release_order_pipeline_stage`
- `execution.message`
中能准确表达这些状态。

## 11. 日志与审计要求
需要增加日志节点：
- `precheck_start / success / failed`
- `lock_acquire_start / success / failed`
- `lock_release_start / success / failed`
- `runtime_snapshot_update`
- `notification_send_start / success / failed`

审计上必须可追溯：
- 谁修改了并发发布配置
- 哪张单申请了什么锁
- 为什么获取锁失败
- 哪次通知发给了哪个渠道

## 12. 配置优先级建议
并发发布配置建议支持两层：
1. 全局默认配置
2. 模板/应用级覆盖配置（后续能力，阶段一先不做界面）

阶段一先落：
- 全局配置
- 运行时严格执行

## 13. 实施顺序建议
### 一期
- `application_env_runtime`
- 环境当前版本查询接口
- 发布前预检接口

### 二期
- `release_concurrency_settings`
- `release_execution_lock`
- execute / rollback / replay 全链路加锁
- 系统管理并发配置接口

### 三期
- 通知渠道配置
- 通知记录表
- 发布结果通知发送

## 14. 结论
`v0.0.15` 后端阶段一的目标，是建立“可靠发布”的底座，而不是继续叠加新发布方式。

后端必须开始承担以下职责：
- 维护环境当前事实
- 在执行前做预检
- 用锁保护并发发布
- 在执行后把结果通知出去

只有这些底座补齐后，后续审批、时间窗、自动回滚、策略发布才有稳定落点。
