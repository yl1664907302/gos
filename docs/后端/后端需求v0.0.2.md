# 后端需求 v0.0.2（基于 v0.0.1 增量）

## 1. 文档信息
- 文档版本：`v0.0.2`
- 基线日期：`2026-03-12`
- 上一版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端需求v0.0.1.md`
- 本版主题：`管线管理模块 + 应用管线绑定（CI/CD）+ Jenkins 接入`

## 2. 版本目标
在保留 v0.0.1 应用管理能力的基础上，新增：
- 管线管理模块（管线数据来源于 Jenkins）
- 应用与管线绑定能力（区分 CI/CD）
- Jenkins 后端接入能力（连通检查、任务拉取、任务校验）

关键原则：
- 管线**不允许手动创建**，必须通过 Jenkins 同步获取。

## 3. 范围与边界
包含：
- 从 Jenkins 拉取并同步管线主数据
- 管线列表/详情查询
- 管线可用性校验（verify）
- 应用与管线关系管理（绑定/解绑/查询）
- 列表分页、基础字段校验、唯一性约束

不包含：
- 实际构建执行编排与发布流程
- 构建日志与历史归档
- Webhook 回调消费与签名校验
- 凭据托管平台（本版仅保留 `credential_ref` 引用）

## 4. 技术与架构约束
- 继承 v0.0.1 技术栈：`Golang + Gin + database/sql + MySQL/SQLite`
- 架构分层：`domain / usecase / infrastructure / interfaces / bootstrap`
- 新增基础设施适配：`infrastructure/jenkins`（Jenkins Client）
- 原则：简单服务层、可读性优先、避免过度抽象

## 5. 领域模型需求

### 5.1 实体：Pipeline（管线）
- `id: string`（系统内部 ID）
- `provider: string`（v0.0.2 固定 `jenkins`）
- `job_full_name: string`（Jenkins Job 全路径标识，如 `folderA/jobB`）
- `job_name: string`
- `job_url: string`
- `description: string`
- `credential_ref: string`（凭据引用，不存明文 token）
- `default_branch: string`
- `status: string`（`active | inactive`）
- `last_verified_at: time`（最近一次校验时间，可空）
- `last_synced_at: time`（最近一次同步时间）
- `created_at: time`
- `updated_at: time`

说明：
- `Pipeline` 主数据由 Jenkins 同步生成；不提供手动创建入口。

### 5.2 实体：PipelineBinding（应用管线绑定）
- `id: string`
- `name: string`（绑定名称，前端展示用）
- `application_id: string`
- `application_name: string`（应用名称快照，前端展示用）
- `binding_type: string`（`ci | cd`）
- `provider: string`（`jenkins | argocd`）
- `pipeline_id: string`（关联 `Pipeline.id`，当 provider=jenkins 时使用）
- `external_ref: string`（当 provider=argocd 时使用，如 ArgoCD app 名称）
- `trigger_mode: string`（`manual | webhook`）
- `status: string`（`active | inactive`）
- `created_at: time`
- `updated_at: time`

### 5.3 业务规则
- `Pipeline.provider` 在 v0.0.2 仅允许 `jenkins`
- `Pipeline.job_full_name` 在同 provider 下唯一
- `Pipeline` 不允许通过 API 手动新增
- `PipelineBinding.application_id` 必须存在
- `PipelineBinding.binding_type` 仅允许 `ci/cd`
- `CI` 绑定 provider 固定为 `jenkins`
- `CD` 绑定 provider 可为 `jenkins` 或 `argocd`
- `provider=jenkins` 时必须传 `pipeline_id`
- `provider=argocd` 时必须传 `external_ref`
- 同一应用下同一 `binding_type` 只能存在一条绑定（唯一）
- `trigger_mode` 仅允许 `manual/webhook`
- `status` 仅允许 `active/inactive`
- 绑定置为 `active` 前，关联管线建议已通过 verify

## 6. API 需求

### 6.1 管线管理模块（只读 + 同步 + 校验）

#### 6.1.1 手动同步 Jenkins 管线
- `POST /jenkins/pipelines/sync`
- 行为：
- 从 Jenkins 拉取任务列表并增量写入/更新 `pipelines`
- 同步策略：按 `(provider, job_full_name)` upsert
- 成功：`200 { data: { total, created, updated, skipped } }`

#### 6.1.2 管线列表（分页）
- `GET /pipelines`
- Query：
- `name?`（匹配 `job_name`）
- `provider?`（默认 `jenkins`）
- `status?`
- `page?`（默认 1）
- `page_size?`（默认 20，最大 100）
- 成功：`200 { data: Pipeline[], page, page_size, total }`

#### 6.1.3 管线详情
- `GET /pipelines/{pipeline_id}`
- 成功：`200 { data: Pipeline }`

#### 6.1.4 校验 Jenkins Job 可用性
- `POST /pipelines/{pipeline_id}/verify`
- 行为：
- 调用 Jenkins API 校验 Job 是否存在、可访问
- 更新 `last_verified_at`
- 成功：`200 { data: { verified: true, job_name, job_url } }`

### 6.2 应用与管线绑定

#### 6.2.1 新增绑定
- `POST /applications/{id}/pipeline-bindings`
- 请求体：
- `binding_type`
- `name`（可选，不传则按管线名/`external_ref` 自动生成）
- `provider`（`ci` 可省略，后端强制 `jenkins`）
- `pipeline_id`
- `external_ref`
- `trigger_mode`
- `status`（可选，默认 `active`）
- 成功：`201 { data: PipelineBinding }`

#### 6.2.2 查询绑定列表（分页）
- `GET /applications/{id}/pipeline-bindings`
- Query：
- `binding_type?`
- `provider?`
- `status?`
- `page?`（默认 1）
- `page_size?`（默认 20，最大 100）
- 成功：`200 { data: PipelineBinding[], page, page_size, total }`

#### 6.2.3 查询绑定详情
- `GET /pipeline-bindings/{binding_id}`
- 成功：`200 { data: PipelineBinding }`

#### 6.2.4 更新绑定
- `PUT /pipeline-bindings/{binding_id}`
- 可更新：
- `name`
- `provider`
- `trigger_mode`
- `status`
- `pipeline_id`
- `external_ref`
- 成功：`200 { data: PipelineBinding }`

#### 6.2.5 解除绑定
- `DELETE /pipeline-bindings/{binding_id}`
- 成功：`204 No Content`

### 6.3 错误码约定
- `400`：参数错误/枚举非法
- `404`：应用/管线/绑定不存在
- `409`：唯一约束冲突或前置校验失败
- `500`：服务内部错误

## 7. 数据库需求

### 7.1 新增表：pipelines
建议字段：
- `id`
- `provider`
- `job_full_name`
- `job_name`
- `job_url`
- `description`
- `credential_ref`
- `default_branch`
- `status`
- `last_verified_at`（可空）
- `last_synced_at`
- `created_at`
- `updated_at`

约束：
- `UNIQUE(provider, job_full_name)`
- `INDEX(status, updated_at DESC)`

### 7.2 新增表：pipeline_bindings
建议字段：
- `id`
- `name`
- `application_id`
- `application_name`
- `pipeline_id`
- `binding_type`
- `provider`
- `external_ref`
- `trigger_mode`
- `status`
- `created_at`
- `updated_at`

约束：
- `UNIQUE(application_id, pipeline_id)`
- `UNIQUE(application_id, binding_type)`
- `INDEX(application_id, created_at DESC)`
- 可选外键：
- `application_id -> applications.id`
- `pipeline_id -> pipelines.id`

### 7.3 迁移要求
- 启动 `InitSchema` 时补齐 `pipelines` 与 `pipeline_bindings`
- MySQL/SQLite 均需提供兼容建表语句
- 不影响 v0.0.1 既有 `applications` 表与接口

## 8. Jenkins 接入要求

### 8.1 配置项
建议新增配置段：
- `jenkins.enabled`
- `jenkins.base_url`
- `jenkins.username`
- `jenkins.api_token`
- `jenkins.timeout_sec`
- `jenkins.startup_check_enabled`
- `jenkins.startup_max_retries`
- `jenkins.startup_retry_interval_sec`
- `jenkins.auto_sync_enabled`
- `jenkins.auto_sync_interval_sec`

环境变量覆盖建议：
- `JENKINS_ENABLED`
- `JENKINS_BASE_URL`
- `JENKINS_USERNAME`
- `JENKINS_API_TOKEN`
- `JENKINS_AUTO_SYNC_ENABLED`
- `JENKINS_AUTO_SYNC_INTERVAL_SEC`

### 8.2 启动检查
- 当 `jenkins.enabled=true` 且 `startup_check_enabled=true`：
- 启动阶段执行 Jenkins 连通检查（如 `GET /api/json`）
- 按重试参数执行重试
- 超限后启动失败并输出明确错误

### 8.3 Jenkins Client 能力
- 基础鉴权（Basic Auth + API Token）
- 获取 Job 列表（用于 sync）
- 获取 Job 详情（用于 verify）
- 统一超时与错误映射

## 9. 安全与合规要求
- 禁止返回或存储明文 token/password
- API 返回 `credential_ref`，不返回密文/明文凭据
- Swagger 不暴露敏感字段示例值
- 日志中对凭据字段脱敏

## 10. 与 v0.0.1 兼容性要求
- 保持现有 `/applications` CRUD 与分页契约不变
- 不修改既有响应命名风格（snake_case）
- 前端未接入新模块时，原应用管理流程不受影响

## 11. 验收标准（v0.0.2）
- 管线通过 Jenkins 同步入库，不存在手动创建入口
- 管线列表/详情可查询，分页与筛选可用
- Jenkins 连接校验与 Job verify 接口可用
- 应用与管线绑定全流程可用（增删改查）
- 唯一冲突返回 `409`
- MySQL/SQLite 行为一致
- Swagger 新增接口文档完整可访问
- v0.0.1 应用管理接口回归无异常
