# 后端需求 v0.0.3（基于 v0.0.2 增量）

## 1. 文档信息
- 文档版本：`v0.0.3`
- 基线日期：`2026-03-13`
- 上一版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端需求v0.0.2.md`
- 本版主题：`平台参数管理模块 + 执行器参数模块 + Jenkins 参数同步`

## 2. 版本目标
在保留 v0.0.2 已有能力的基础上，新增：
- 平台参数管理模块：统一平台发布字段标准字库，支持 CRUD
- 执行器参数模块：记录 Jenkins 每条管线的真实参数定义
- 平台标准参数与执行器真实参数之间的映射能力
- Jenkins 定时同步能力扩展：同步执行器参数定义

关键原则：
- 平台发布字段必须优先复用平台标准字段库
- 管线真实参数主数据来源于 Jenkins 定时同步，不允许前端随意伪造
- 每个执行器真实参数生成一条独立记录

## 3. 范围与边界
包含：
- 保留 v0.0.2 的管线管理、应用管线绑定、Jenkins 接入能力
- 平台标准参数字典的新增、查询、编辑、删除
- Jenkins 执行器参数定义同步、查询、平台标准字段映射维护
- 执行器真实参数与平台标准参数 `param_key` 的绑定
- 参数字段分页、唯一性校验、基础枚举校验

不包含：
- 实际发布执行页面与发布单据渲染
- 参数值实例保存与发布历史追踪
- ArgoCD 参数实时拉取
- 自定义执行器 `custom` 的实际接入实现

## 4. 技术与架构约束
- 继承 v0.0.2 技术栈：`Golang + Gin + database/sql + MySQL/SQLite`
- 架构分层：`domain / usecase / infrastructure / interfaces / bootstrap`
- 继续复用现有 Jenkins Client 与定时同步模块
- 原则：简单服务层、可读性优先、避免过度抽象
- 时间字段业务语义为 `datetime`，具体存储可继续沿用现有 `BIGINT/INTEGER UTC 时间戳` 实现策略

## 5. 领域模型需求

### 5.1 沿用 v0.0.2 既有能力
- `Pipeline`、`PipelineBinding`、Jenkins 管线同步、应用与管线绑定能力保持不变
- v0.0.2 已有接口契约默认继续有效

### 5.2 实体：PlatformParamDict（平台参数标准字典）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `varchar(64)` | 主键 |
| `param_key` | `varchar(100)` | 平台标准字段 key，唯一 |
| `name` | `varchar(100)` | 字段名称 |
| `description` | `varchar(500)` | 字段说明 |
| `param_type` | `varchar(50)` | 字段类型 |
| `required` | `tinyint(1)` | 是否默认必填 |
| `builtin` | `tinyint(1)` | 是否内置字段 |
| `status` | `tinyint(1)` | 状态，启用/停用 |
| `created_at` | `datetime` | 创建时间 |
| `updated_at` | `datetime` | 更新时间 |

说明：
- `param_key` 必须全小写
- 建议命名规则：`[a-z][a-z0-9_]*`
- `param_type` 建议与执行器参数类型保持兼容，至少支持：`string / choice / bool / number`
- `status` 业务语义：`1=启用`，`0=停用`

### 5.3 实体：ExecutorParamDef（执行器参数定义）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `varchar(64)` | 主键 |
| `pipeline_id` | `varchar(64)` | 管线 ID |
| `executor_type` | `varchar(50)` | 执行器类型（`jenkins / argocd / custom`） |
| `executor_param_name` | `varchar(100)` | 执行器真实参数名 |
| `param_key` | `varchar(100)` | 平台标准参数 key |
| `param_type` | `varchar(50)` | 参数类型（`string / choice / bool / number`） |
| `required` | `tinyint(1)` | 是否必填 |
| `default_value` | `varchar(500)` | 默认值 |
| `description` | `varchar(500)` | 参数描述 |
| `visible` | `tinyint(1)` | 前端是否展示 |
| `editable` | `tinyint(1)` | 发布时是否允许修改 |
| `source_from` | `varchar(50)` | 参数来源（`sync_jenkins / manual`） |
| `raw_meta` | `json` | Jenkins 原始参数结构 |
| `sort_no` | `int` | 排序 |
| `created_at` | `datetime` | 创建时间 |
| `updated_at` | `datetime` | 更新时间 |

说明：
- `pipeline_id` 关联 `pipelines.id`
- `executor_type` 在 v0.0.3 实际落地以 `jenkins` 为主，`argocd/custom` 保留扩展
- `param_key` 用于映射平台标准字段库，可为空；若填写则必须来自 `PlatformParamDict.param_key`
- `raw_meta` 保存 Jenkins 原始参数结构，MySQL 可用 `JSON`，SQLite 可用 `TEXT(JSON string)`

### 5.4 业务规则
- `PlatformParamDict.param_key` 全局唯一，且必须为小写
- 内置字段 `builtin=1` 默认不允许删除
- 停用状态的标准字段不建议再用于新增映射
- `ExecutorParamDef` 每个真实参数生成一条记录
- `ExecutorParamDef` 唯一键建议为：`(pipeline_id, executor_type, executor_param_name)`
- Jenkins 同步产生的参数记录 `source_from` 固定为 `sync_jenkins`
- `param_key` 若填写，必须能在平台标准字段库中找到对应启用项
- `ExecutorParamDef` 人工仅允许维护 `param_key`
- `visible`、`editable`、`sort_no`、`description`、`default_value` 在 v0.0.3 不提供人工编辑接口
- `executor_param_name` 保持执行器原始命名，不强制转小写

### 6.2 平台参数管理模块

#### 6.2.0 CRUD 路由汇总
- `POST /platform-param-dicts`：新增标准字段
- `GET /platform-param-dicts`：标准字段分页列表
- `GET /platform-param-dicts/{id}`：标准字段详情
- `PUT /platform-param-dicts/{id}`：更新标准字段
- `DELETE /platform-param-dicts/{id}`：删除标准字段

#### 6.2.1 新增标准字段
- `POST /platform-param-dicts`
- 请求体：
- `param_key`
- `name`
- `description`
- `param_type`
- `required`
- `builtin`
- `status`
- 成功：`201 { data: PlatformParamDict }`

#### 6.2.2 标准字段列表（分页）
- `GET /platform-param-dicts`
- Query：
- `param_key?`
- `name?`
- `status?`
- `builtin?`
- `page?`（默认 1）
- `page_size?`（默认 20，最大 100）
- 成功：`200 { data: PlatformParamDict[], page, page_size, total }`

#### 6.2.3 标准字段详情
- `GET /platform-param-dicts/{id}`
- 成功：`200 { data: PlatformParamDict }`

#### 6.2.4 更新标准字段
- `PUT /platform-param-dicts/{id}`
- 可更新：
- `name`
- `description`
- `param_type`
- `required`
- `status`
- `builtin`（是否允许修改可按实现策略控制）
- `param_key`（若允许修改，仍必须全小写且唯一）
- 成功：`200 { data: PlatformParamDict }`

#### 6.2.5 删除标准字段
- `DELETE /platform-param-dicts/{id}`
- 规则：
- `builtin=1` 建议禁止删除
- 若已被 `ExecutorParamDef.param_key` 引用，建议阻止删除并返回冲突
- 成功：`204 No Content`

### 6.3 执行器参数模块

#### 6.3.0 路由汇总
- `GET /applications/{id}/executor-param-defs`：通过应用查询其已绑定的 Jenkins 执行器参数定义
- `GET /pipelines/{id}/param-defs`：查询某条管线的参数定义列表
- `GET /executor-param-defs/{id}`：查询参数定义详情
- `PUT /executor-param-defs/{id}`：更新参数定义的平台标准字段映射
- `POST /jenkins/executor-param-defs/sync`：手动触发 Jenkins 参数同步

说明：
- `ExecutorParamDef` 主数据来源于 Jenkins 同步，因此 v0.0.3 不提供手动 `POST /executor-param-defs` 创建路由
- `ExecutorParamDef` 默认也不提供手动 `DELETE /executor-param-defs/{id}` 删除路由，避免与同步主数据冲突
- `ExecutorParamDef` 不提供对 `executor_param_name`、`param_type`、`required`、`default_value`、`description`、`visible`、`editable`、`sort_no` 的人工编辑路由

#### 6.3.1 查询某条管线的参数定义列表
- `GET /pipelines/{id}/param-defs`
- Query：
- `executor_type?`
- `visible?`
- `editable?`
- `param_key?`
- `page?`（默认 1）
- `page_size?`（默认 20，最大 100）
- 成功：`200 { data: ExecutorParamDef[], page, page_size, total }`

#### 6.3.1.1 通过应用查询已绑定 Jenkins 管线的参数定义
- `GET /applications/{id}/executor-param-defs`
- Query：
- `binding_type?`（默认 `ci`，支持 `ci / cd`）
- `visible?`
- `editable?`
- `param_key?`
- `page?`（默认 1）
- `page_size?`（默认 20，最大 100）
- 规则：
- 先根据 `application_id + binding_type` 查找绑定关系
- 仅支持 `provider=jenkins` 的绑定关系
- 若绑定不存在，返回 `404`
- 若绑定存在但不是 Jenkins 类型，返回 `400`
- 成功：`200 { data: ExecutorParamDef[], page, page_size, total }`

#### 6.3.2 查询参数定义详情
- `GET /executor-param-defs/{id}`
- 成功：`200 { data: ExecutorParamDef }`

#### 6.3.3 更新参数定义的平台标准字段映射
- `PUT /executor-param-defs/{id}`
- 请求体：
- `param_key`
- 成功：`200 { data: ExecutorParamDef }`

说明：
- `executor_param_name`、`executor_type`、`pipeline_id` 不允许通过人工接口修改
- `param_key` 若传空值，表示取消与平台标准字段的映射
- `param_key` 若传非空值，必须命中 `PlatformParamDict.param_key`

#### 6.3.4 手动触发 Jenkins 参数同步
- `POST /jenkins/executor-param-defs/sync`
- 行为：
- 从 Jenkins 拉取每条管线的参数定义
- 以 `(pipeline_id, executor_type, executor_param_name)` 为键执行 upsert
- 成功：`200 { data: { total, created, updated, skipped } }`

### 6.4 错误码约定
- `400`：参数错误、枚举非法、`param_key` 非小写
- `404`：平台参数字典 / 管线 / 参数定义不存在
- `409`：唯一约束冲突、内置字段删除冲突、被引用冲突
- `500`：服务内部错误

## 8. Jenkins 接入与同步要求

### 8.1 同步范围扩展
- 在现有 Jenkins 管线同步能力基础上，扩展拉取 Job 参数定义
- 同步来源为 Jenkins 参数元数据，如字符串参数、选项参数、布尔参数、数字参数等
- 每个参数生成一条 `ExecutorParamDef` 记录

### 8.2 同步策略
- 以 `(pipeline_id, executor_type, executor_param_name)` 作为唯一键执行 upsert
- Jenkins 原始参数结构写入 `raw_meta`
- 同步生成的数据默认：
- `executor_type=jenkins`
- `source_from=sync_jenkins`
- `visible=true`
- `editable=true`
- `sort_no` 可按 Jenkins 参数顺序生成

### 8.3 定时任务要求
- 在 v0.0.2 已有 Jenkins 定时同步任务基础上扩展参数同步
- 支持服务启动后立即执行一次
- 支持按既有间隔参数持续执行
- 同步失败不得影响已启动服务对外提供只读能力，但需输出明确日志

## 9. 安全与合规要求
- 禁止返回或存储 Jenkins 明文凭据
- `raw_meta` 仅保存参数结构，不应写入敏感凭据值
- 日志中对敏感配置脱敏
- Swagger 不暴露敏感字段示例值

## 10. 与 v0.0.2 兼容性要求
- 保持 v0.0.2 既有 `/applications`、`/pipelines`、`/pipeline-bindings` 契约不变
- 不修改既有响应命名风格（snake_case）
- 新模块接入后，不影响已有前端应用管理、Jenkins 管理、管线绑定页面
