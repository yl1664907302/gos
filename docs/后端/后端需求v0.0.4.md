## 2. 版本目标
在 `v0.0.3`（应用管理 + 管线绑定 + Jenkins 管线与参数）基础上，新增发布执行主链路数据模型：
- `release_order`：发布单主表
- `release_order_param`：发布参数快照
- `release_order_step`：发布执行步骤

本版目标是形成可运行的发布单领域模型与数据落库基础，支撑后续发布页面、触发执行、执行追踪。

## 3. 范围与边界
包含：
- 发布单主单创建、查询、状态流转的数据模型
- 发布参数快照留痕（发布时最终参数）
- 发布执行步骤追踪（步骤状态、消息、时间）
- 与应用、管线绑定、执行器参数定义的关联约束

不包含：
- 发布页面前端实现细节
- Jenkins/ArgoCD 实际执行引擎完整接入细节
- 发布审批流、回滚流、灰度策略
- 多租户与复杂权限模型

## 4. 领域模型设计

### 4.1 发布单主表 `release_order`

```sql
CREATE TABLE `release_order` (
  `id` varchar(64) NOT NULL COMMENT '主键ID',
  `order_no` varchar(64) NOT NULL COMMENT '发布单号',

  `application_id` varchar(64) NOT NULL COMMENT '应用ID',
  `application_name` varchar(100) DEFAULT NULL COMMENT '应用名称(冗余)',

  `binding_id` varchar(64) NOT NULL COMMENT '管线绑定ID',
  `pipeline_id` varchar(64) DEFAULT NULL COMMENT '管线ID(冗余)',

  `env_code` varchar(50) NOT NULL COMMENT '环境',

  `git_ref` varchar(200) DEFAULT NULL COMMENT 'Git版本',
  `image_tag` varchar(200) DEFAULT NULL COMMENT '镜像版本',

  `trigger_type` varchar(50) DEFAULT 'manual' COMMENT '触发类型(manual/webhook/schedule)',

  `status` varchar(50) NOT NULL COMMENT '发布状态(pending/running/success/failed/cancelled)',

  `remark` varchar(500) DEFAULT NULL COMMENT '发布备注',

  `triggered_by` varchar(64) DEFAULT NULL COMMENT '触发人',

  `started_at` datetime DEFAULT NULL COMMENT '开始时间',
  `finished_at` datetime DEFAULT NULL COMMENT '结束时间',

  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_no` (`order_no`),
  KEY `idx_application` (`application_id`),
  KEY `idx_binding` (`binding_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='发布单';
```

状态枚举：
- `pending`
- `running`
- `success`
- `failed`
- `cancelled`

触发类型枚举：
- `manual`
- `webhook`
- `schedule`

### 4.2 发布参数快照表 `release_order_param`

说明：用于记录“本次发布最终使用的参数”，与 `executor_param_def` 不同，此表保存的是发布瞬间参数值快照。

```sql
CREATE TABLE `release_order_param` (
  `id` varchar(64) NOT NULL COMMENT '主键ID',

  `release_order_id` varchar(64) NOT NULL COMMENT '发布单ID',

  `param_key` varchar(100) NOT NULL COMMENT '平台参数Key',

  `executor_param_name` varchar(100) DEFAULT NULL COMMENT '执行器参数名',

  `param_value` varchar(1000) DEFAULT NULL COMMENT '参数值',

  `value_source` varchar(50) DEFAULT NULL COMMENT '参数来源(application/environment/release_input/fixed)',

  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',

  PRIMARY KEY (`id`),
  KEY `idx_release_order` (`release_order_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='发布参数快照';
```

示例数据：

| param_key | executor_param_name | param_value |
| --- | --- | --- |
| gitRef | BRANCH | release/1.0.0 |
| imageTag | IMAGE_TAG | 20260313-01 |
| namespace | NAMESPACE | prod |

### 4.3 发布执行步骤表 `release_order_step`

说明：用于追踪发布过程中的执行阶段、阶段状态与执行消息。

```sql
CREATE TABLE `release_order_step` (
  `id` varchar(64) NOT NULL COMMENT '主键ID',

  `release_order_id` varchar(64) NOT NULL COMMENT '发布单ID',

  `step_code` varchar(100) NOT NULL COMMENT '步骤编码',

  `step_name` varchar(200) DEFAULT NULL COMMENT '步骤名称',

  `status` varchar(50) DEFAULT NULL COMMENT '状态(pending/running/success/failed)',

  `message` varchar(1000) DEFAULT NULL COMMENT '执行信息',

  `sort_no` int DEFAULT 0 COMMENT '步骤顺序',

  `started_at` datetime DEFAULT NULL COMMENT '开始时间',

  `finished_at` datetime DEFAULT NULL COMMENT '结束时间',

  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',

  PRIMARY KEY (`id`),
  KEY `idx_release_order` (`release_order_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='发布执行步骤';
```

示例步骤：

| step_code | step_name |
| --- | --- |
| param_resolve | 参数解析 |
| trigger_pipeline | 触发管线 |
| pipeline_running | 管线运行 |
| pipeline_success | 发布成功 |

## 5. 三张表关系

关系含义：
- `release_order`：一次发布任务的主单
- `release_order_param`：该次发布使用的最终参数快照
- `release_order_step`：该次发布执行过程与阶段状态

关系约束建议：
- `release_order_param.release_order_id -> release_order.id`
- `release_order_step.release_order_id -> release_order.id`
- `release_order.binding_id -> app_pipeline_binding.id`（或当前实现中的 `pipeline_bindings.id`）

## 6. 一次发布的数据流

1. 用户发起发布，系统创建 `release_order`，初始状态 `pending`。  
2. 系统根据应用 + 绑定 + 环境 + 发布输入，解析最终参数。  
3. 将最终参数逐条写入 `release_order_param` 作为不可变快照。  
4. 初始化 `release_order_step`（如参数解析、触发管线、运行中、完成）。  
5. 发布开始时更新主单为 `running`，写入 `started_at`。  
6. 执行过程中按步骤更新 `release_order_step.status/message/started_at/finished_at`。  
7. 执行成功则主单更新为 `success`；失败更新为 `failed`；人工取消更新为 `cancelled`。  
8. 主单结束时写入 `finished_at`，形成完整可追踪发布记录。  

## 7. API 需求（发布页面模块）

### 7.1 发布单
- `POST /release-orders`：创建发布单（并落参数快照、初始化步骤）
- `GET /release-orders`：分页查询发布单
- `GET /release-orders/{id}`：发布单详情
- `POST /release-orders/{id}/cancel`：取消发布（状态流转到 `cancelled`）

### 7.2 发布参数快照
- `GET /release-orders/{id}/params`：查询发布参数快照列表

### 7.3 发布步骤
- `GET /release-orders/{id}/steps`：查询发布执行步骤列表
- `POST /release-orders/{id}/steps/{step_code}/start`：步骤开始（内部或执行器回调）
- `POST /release-orders/{id}/steps/{step_code}/finish`：步骤结束（内部或执行器回调）

### 7.4 查询过滤建议
- 按应用：`application_id`
- 按环境：`env_code`
- 按状态：`status`
- 按触发方式：`trigger_type`
- 按时间区间：`created_at/start_time/end_time`

## 8. 状态机与约束

### 8.1 发布单状态流转
- `pending -> running -> success`
- `pending -> running -> failed`
- `pending -> cancelled`
- `running -> cancelled`

约束：
- 终态（`success/failed/cancelled`）不可再流转到非终态
- `finished_at` 仅在终态写入

### 8.2 步骤状态流转
- `pending -> running -> success`
- `pending -> running -> failed`

约束：
- 步骤按 `sort_no` 递增执行（允许按实现配置并行步骤，但需明确）
- 失败步骤应写入可读 `message`
