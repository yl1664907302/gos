# 后端需求 v0.0.6（基于 v0.0.5 增量）

## 1. 文档信息
- 文档版本：`v0.0.6`
- 基线日期：`2026-03-14`
- 上一版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端/后端需求v0.0.5.md`
- 本版主题：`用户管理 + 登录鉴权 + 角色权限 + 应用负责人绑定`

## 2. 版本目标
在现有应用/管线/发布能力基础上，新增统一用户与授权模型，实现：
- 新增登录鉴权能力（登录/退出/当前用户上下文）
- 创建应用时从用户列表选择应用负责人
- 角色区分：管理员、普通用户
- 管理员拥有全量权限
- 普通用户默认无权限，需管理员授权后才能使用对应模块与操作
- 权限细化到发布单创建页“发布参数”可见与可编辑级别

## 3. 范围与边界
包含：
- 登录鉴权（本地账号密码）
- 用户数据模型与 CRUD（后台管理）
- 角色模型（admin / normal）
- 权限字典、用户授权、撤销授权
- 应用与负责人绑定
- 发布参数级权限校验（view/edit）
- Redis 可选增强方案（不作为 v0.0.6 必选交付）

不包含：
- 外部 IAM / LDAP / SSO 接入
- 复杂组织架构（部门树、多级角色继承）
- 审批流引擎

## 4. 领域模型设计

### 4.1 用户主表 `sys_user`
```sql
CREATE TABLE `sys_user` (
  `id` varchar(64) NOT NULL,
  `username` varchar(100) NOT NULL COMMENT '登录名，唯一',
  `display_name` varchar(100) NOT NULL COMMENT '姓名/昵称',
  `email` varchar(200) DEFAULT NULL,
  `phone` varchar(50) DEFAULT NULL,
  `role` varchar(20) NOT NULL COMMENT 'admin/normal',
  `status` varchar(20) NOT NULL DEFAULT 'active' COMMENT 'active/inactive',
  `password_hash` varchar(255) DEFAULT NULL COMMENT '如后续接入本地登录可用',
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_sys_user_username` (`username`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统用户';
```

### 4.2 权限字典表 `sys_permission`
```sql
CREATE TABLE `sys_permission` (
  `id` varchar(64) NOT NULL,
  `code` varchar(100) NOT NULL COMMENT '权限编码，唯一',
  `name` varchar(100) NOT NULL,
  `module` varchar(50) NOT NULL COMMENT 'application/pipeline/release/component/system',
  `action` varchar(50) NOT NULL COMMENT 'view/create/update/delete/execute/cancel/grant',
  `description` varchar(500) DEFAULT NULL,
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_sys_permission_code` (`code`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='权限字典';
```

### 4.3 用户授权表 `sys_user_permission`
```sql
CREATE TABLE `sys_user_permission` (
  `id` varchar(64) NOT NULL,
  `user_id` varchar(64) NOT NULL,
  `permission_code` varchar(100) NOT NULL,
  `scope_type` varchar(30) NOT NULL DEFAULT 'global' COMMENT 'global/application/param_key',
  `scope_value` varchar(200) DEFAULT NULL COMMENT 'application_id 或 param_key',
  `enabled` tinyint(1) NOT NULL DEFAULT 1,
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_sup_user` (`user_id`),
  KEY `idx_sup_code` (`permission_code`),
  KEY `idx_sup_scope` (`scope_type`,`scope_value`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户授权';
```

### 4.4 发布参数权限表 `sys_user_param_permission`
> 用于“发布参数页面字段级”权限控制。
```sql
CREATE TABLE `sys_user_param_permission` (
  `id` varchar(64) NOT NULL,
  `user_id` varchar(64) NOT NULL,
  `param_key` varchar(100) NOT NULL COMMENT '平台标准字段 key',
  `application_id` varchar(64) DEFAULT NULL COMMENT '为空表示全应用生效',
  `can_view` tinyint(1) NOT NULL DEFAULT 0,
  `can_edit` tinyint(1) NOT NULL DEFAULT 0,
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_supp_user` (`user_id`),
  KEY `idx_supp_param` (`param_key`),
  KEY `idx_supp_app` (`application_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户发布参数权限';
```

### 4.5 应用表变更 `application`
- 新增字段：`owner_user_id varchar(64)`（负责人用户ID）
- 保留字段：`owner`（负责人名称，冗余用于展示）

### 4.6 登录会话表 `sys_user_session`
> 若采用 token + 会话持久化，建议落库便于失效控制与审计。
```sql
CREATE TABLE `sys_user_session` (
  `id` varchar(64) NOT NULL,
  `user_id` varchar(64) NOT NULL,
  `access_token` varchar(512) NOT NULL,
  `expired_at` datetime NOT NULL,
  `client_ip` varchar(64) DEFAULT NULL,
  `user_agent` varchar(300) DEFAULT NULL,
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_sus_user` (`user_id`),
  KEY `idx_sus_expired` (`expired_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='登录会话';
```

## 5. 核心规则

### 5.1 角色规则
- `admin`：默认拥有全部权限，不依赖 `sys_user_permission` 明细。
- `normal`：默认无业务权限，仅可访问“我的资料”等基础能力，需管理员授权后解锁业务功能。

### 5.2 应用负责人绑定规则
- 创建应用必须传 `owner_user_id`。
- `owner` 字段由后端根据用户信息自动回填为 `display_name`。
- 管理员可在应用编辑时变更负责人。

### 5.3 发布参数页面权限规则（重点）
- 管理员：展示全部参数，全部可编辑（按原有业务只读规则除外）。
- 普通用户：
- 无 `can_view` 的 `param_key`：参数不返回前端，不展示。
- `can_view=1, can_edit=0`：参数只读展示，提交时不可覆盖。
- `can_edit=1`：参数可编辑并可提交。
- 提交创建发布单时，后端再次校验参数权限，禁止越权提交。

### 5.4 登录与鉴权规则
- 登录账号必须为 `status=active` 用户。
- 密码使用安全哈希（建议 `bcrypt`）存储在 `password_hash`。
- 鉴权建议采用 Bearer Token（JWT 或服务端会话 token）。
- 所有受控 API 都需经过认证中间件并注入当前用户上下文。
- `admin` 角色短路放行；`normal` 继续走权限校验。

## 6. API 需求

### 6.1 登录鉴权
- `POST /auth/login`：账号密码登录
- `POST /auth/logout`：退出当前登录
- `GET /me`：返回当前用户 + 角色 + 权限快照

### 6.2 用户管理（管理员）
- `GET /users`：分页查询用户
- `GET /users/{id}`：用户详情
- `POST /users`：创建用户
- `PUT /users/{id}`：更新用户
- `DELETE /users/{id}`：删除/停用用户（建议逻辑停用）

### 6.3 权限管理（管理员）
- `GET /permissions`：权限字典列表
- `GET /users/{id}/permissions`：查询用户已授权权限
- `POST /users/{id}/permissions`：批量授权
- `DELETE /users/{id}/permissions`：批量撤销授权

### 6.4 发布参数权限（管理员）
- `GET /users/{id}/param-permissions`
- `POST /users/{id}/param-permissions`
- `PUT /users/{id}/param-permissions/{permission_id}`
- `DELETE /users/{id}/param-permissions/{permission_id}`

### 6.5 应用管理接口变更
- `POST /applications`：新增 `owner_user_id` 字段（必填）
- `PUT /applications/{id}`：支持更新 `owner_user_id`
- `GET /users/options`：用于下拉选择负责人（id/display_name）

### 6.6 发布参数查询接口增强
- `GET /applications/{id}/pipeline-param-defs`
- 返回每个参数增加：`can_view`、`can_edit`
- 对无 `can_view` 参数后端直接过滤，不下发

## 7. 权限控制落点（后端）
- HTTP 中间件：识别当前用户与角色
- Service 层二次校验：防绕过
- Repository 层不直接做权限逻辑
- 关键校验点：
- 应用 CRUD
- 管线绑定 CRUD
- 标准字库 CRUD
- 管线参数映射编辑
- 发布单创建、执行、取消
- 发布参数提交（逐项 param_key 校验）

## 8. 初始数据与兼容
- 系统初始化需内置管理员账号（如 `admin`）
- 系统初始化需支持管理员初始密码配置（环境变量或启动配置）
- 初始化权限字典（覆盖现有模块）
- 旧应用数据兼容：
- 若无 `owner_user_id`，可一次性迁移为默认管理员或按 `owner` 名称匹配用户

## 9. 验收标准（v0.0.6）
- 创建应用可从用户下拉选择负责人并成功落库
- 管理员可完成用户与权限授权管理
- 普通用户在未授权时无法访问受控接口
- 授权后可按授权范围访问功能
- 发布参数页面参数展示与编辑能力符合 `can_view/can_edit`

## 11. Redis 可选增强项（非必选）
- 目标：在不改变当前 MySQL 主模型前提下，提升登录鉴权与权限读取性能。
- 会话缓存：
- 建议 Key：`auth:session:{token}`
- 内容：`user_id`、`role`、`expired_at`
- TTL：与登录 token 过期时间一致
- 权限快照缓存：
- 建议 Key：`auth:perm:{user_id}:{version}`
- 内容：权限点集合、参数级权限集合
- 触发失效：管理员授权/撤权后递增 `version` 或主动删除旧 key
- 登录安全增强：
- 建议 Key：`auth:login:ip:{ip}`、`auth:login:user:{username}`
- 用途：失败次数统计、短时限流、风控封禁窗口
- 一致性策略：
- 退出登录时删除会话缓存
- 用户停用/改角色/改权限时，主动清理该用户所有会话缓存与权限缓存
- 降级策略：
- Redis 不可用时回退 MySQL 查询，不影响核心业务可用性
