# 后端需求 v0.0.7（基于 v0.0.6 增量）

## 1. 文档信息
- 文档版本：`v0.0.7`
- 基线日期：`2026-03-16`
- 上一版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端/后端需求v0.0.6.md`
- 本版主题：`发布模板支持 CI/CD 双管线 + 发布单多执行单元 + 双日志/双进度`

## 2. 版本目标
在现有“应用 -> 管线绑定 -> 发布模板 -> 发布单”模型基础上，升级为多执行单元发布模型，实现：
- 一个发布模板可同时配置 `CI` 与 `CD`
- `CI` 固定使用 Jenkins
- `CD` 支持 Jenkins 或 ArgoCD
- 发布单创建时可同时填写 `CI 参数` 与 `CD 参数`
- 发布执行时支持：
- 只执行 `CI`
- 只执行 `CD`
- 顺序执行 `CI -> CD`
- 发布详情可分别展示：
- `CI` 日志
- `CD` 日志
- `CI` 管线进度
- `CD` 管线进度
- 在无 `CD` 绑定或模板未启用 `CD` 的情况下，相关模块不展示

## 3. 范围与边界
包含：
- 发布模板升级为多绑定模型
- 发布单升级为多执行单元模型
- 参数快照、执行步骤、日志、管线进度按 `scope` 分治
- Jenkins `CI/CD` 双执行单元链路
- ArgoCD `CD` 执行单元模型预留

不包含：
- 多环境并发发布
- 复杂审批流
- Jenkins / ArgoCD 平台级账号管理重构
- GitLab / Github Actions 等新执行器

## 4. 核心设计原则

### 4.1 一次发布单，对应多个执行单元
- 发布单主表只表示“一次业务发布行为”
- 真正执行由多个执行单元承担：
- `ci`
- `cd`

### 4.2 一切执行相关数据都要带 `pipeline_scope`
- 参数快照
- 执行步骤
- 日志流
- 管线进度
- 运行结果

### 4.3 模板决定发布结构
- 新建发布单时，用户不再手动拼装 `CI/CD`
- 由发布模板决定：
- 是否启用 `CI`
- 是否启用 `CD`
- 每个 scope 绑定哪条管线
- 每个 scope 暴露哪些参数

## 5. 领域模型升级

### 5.1 发布模板主表 `release_template`
保留“模板主信息”职责，建议字段：
- `id`
- `name`
- `application_id`
- `application_name`
- `status`
- `remark`
- `created_at`
- `updated_at`

说明：
- 模板主表不再直接持有单个 `binding_id`
- 绑定关系下沉到模板绑定明细表

### 5.2 发布模板绑定表 `release_template_binding`
用于描述模板下每个执行单元的绑定关系。

```sql
CREATE TABLE `release_template_binding` (
  `id` varchar(64) NOT NULL,
  `template_id` varchar(64) NOT NULL,
  `pipeline_scope` varchar(20) NOT NULL COMMENT 'ci/cd',
  `binding_id` varchar(64) NOT NULL,
  `binding_name` varchar(128) DEFAULT NULL,
  `provider` varchar(32) NOT NULL COMMENT 'jenkins/argocd',
  `pipeline_id` varchar(64) DEFAULT NULL,
  `enabled` tinyint(1) NOT NULL DEFAULT 1,
  `sort_no` int NOT NULL DEFAULT 1,
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_template_scope` (`template_id`, `pipeline_scope`),
  KEY `idx_template_binding_template` (`template_id`),
  KEY `idx_template_binding_binding` (`binding_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='发布模板绑定明细';
```

规则：
- 一个模板下最多一条 `ci`
- 一个模板下最多一条 `cd`
- 若模板未启用某个 scope，则该 scope 不创建记录

### 5.3 发布模板参数表 `release_template_param`
升级为按 scope 管理参数。

建议字段：
- `id`
- `template_id`
- `template_binding_id`
- `pipeline_scope`
- `binding_id`
- `pipeline_param_def_id`
- `param_key`
- `display_name`
- `required`
- `default_value`
- `editable`
- `sort_no`
- `created_at`
- `updated_at`

规则：
- 模板参数必须来自对应 scope 绑定的真实管线参数
- 模板参数必须已映射平台标准 Key
- `display_name` 优先使用平台标准字段名称

### 5.4 发布单主表 `release_order`
主表继续表示“一次发布”，建议增加模板信息冗余：
- `template_id`
- `template_name`
- `creator_user_id`
- `triggered_by`
- `status`
- `remark`
- `created_at`
- `updated_at`

兼容建议：
- 现有 `binding_id`、`pipeline_id` 字段标记为“兼容保留”
- 新逻辑以执行单元表为准

### 5.5 发布单执行单元表 `release_order_execution`
用于承载 `CI/CD` 的独立执行上下文。

```sql
CREATE TABLE `release_order_execution` (
  `id` varchar(64) NOT NULL,
  `release_order_id` varchar(64) NOT NULL,
  `pipeline_scope` varchar(20) NOT NULL COMMENT 'ci/cd',
  `binding_id` varchar(64) NOT NULL,
  `binding_name` varchar(128) DEFAULT NULL,
  `provider` varchar(32) NOT NULL COMMENT 'jenkins/argocd',
  `pipeline_id` varchar(64) DEFAULT NULL,
  `status` varchar(32) NOT NULL COMMENT 'pending/running/success/failed/cancelled/skipped',
  `queue_url` varchar(500) DEFAULT NULL,
  `build_url` varchar(500) DEFAULT NULL,
  `external_run_id` varchar(128) DEFAULT NULL COMMENT 'ArgoCD 等外部运行标识',
  `started_at` datetime DEFAULT NULL,
  `finished_at` datetime DEFAULT NULL,
  `created_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_order_scope` (`release_order_id`, `pipeline_scope`),
  KEY `idx_order_execution_order` (`release_order_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='发布单执行单元';
```

### 5.6 发布参数快照表 `release_order_param`
建议新增字段：
- `pipeline_scope`
- `binding_id`

说明：
- `CI` 参数与 `CD` 参数必须可区分
- 发布详情页需按 scope 独立展示

### 5.7 发布执行步骤表 `release_order_step`
建议新增字段：
- `step_scope`：`global/ci/cd`
- `execution_id`：可空，对应 `release_order_execution.id`

步骤建议：
- `global:param_resolve`
- `ci:trigger_pipeline`
- `ci:pipeline_running`
- `ci:pipeline_success`
- `cd:trigger_pipeline`
- `cd:pipeline_running`
- `cd:pipeline_success`
- `global:release_finish`

规则：
- 若模板无 `cd`，则不创建 `cd` 步骤
- 若模板只有 `cd`，则跳过 `ci` 步骤

### 5.8 发布管线阶段表 `release_order_pipeline_stage`
现有阶段模型继续保留，但必须补 `pipeline_scope`。

建议字段新增：
- `execution_id`
- `pipeline_scope`

说明：
- `CI Jenkins` 与 `CD Jenkins` 阶段数据分开存储
- `CD ArgoCD` 不写 Jenkins 阶段，可改为部署状态视图

## 6. 核心规则

### 6.1 模板规则
- 一个应用可有多个模板
- 每个模板可独立启用：
- `CI`
- `CD`
- 模板配置 `CI` 时：
- 只能绑定应用下 `binding_type=ci` 且 `provider=jenkins` 的绑定
- 模板配置 `CD` 时：
- 可绑定应用下 `binding_type=cd`
- `provider` 可为 `jenkins` 或 `argocd`

### 6.2 参数规则
- 模板参数必须先完成平台标准 Key 映射
- `CI 参数` 只能从 `CI` 绑定管线参数中选
- `CD 参数` 只能从 `CD` 绑定管线参数中选
- 同一 `param_key` 可同时存在于 `ci` 与 `cd`

### 6.3 执行规则
- 默认执行顺序：`CI -> CD`
- 若 `CI` 失败：
- 默认不继续执行 `CD`
- 若模板仅配置 `CD`：
- 可直接执行 `CD`
- 若模板仅配置 `CI`：
- 只执行 `CI`

### 6.4 展示规则
- 应用未绑定 `CD` 或模板未启用 `CD`：
- 发布详情不展示 `CD 参数快照`
- 不展示 `CD 日志`
- 不展示 `CD 管线进度`
- 不展示 `CD 步骤`

## 7. API 设计

### 7.1 发布模板
- `GET /release-templates`
- `GET /release-templates/{id}`
- `POST /release-templates`
- `PUT /release-templates/{id}`
- `DELETE /release-templates/{id}`

新增返回结构：
- 模板主信息
- `bindings`：`ci/cd` 绑定配置
- `params`：按 `pipeline_scope` 分组参数

### 7.2 模板参数来源查询
- `GET /applications/{id}/release-template-sources`

返回：
- 应用下可用的 `CI/CD` 绑定
- 每个绑定可选的已映射平台标准 Key 参数

### 7.3 新建发布单
- `POST /release-orders`

请求建议升级为：
- `application_id`
- `template_id`
- `remark`
- `params`

其中 `params` 结构建议：
- `pipeline_scope`
- `param_key`
- `executor_param_name`
- `param_value`
- `value_source`

### 7.4 发布详情
- `GET /release-orders/{id}`
- `GET /release-orders/{id}/params`
- `GET /release-orders/{id}/steps`
- `GET /release-orders/{id}/executions`

### 7.5 日志
- `GET /release-orders/{id}/logs/stream?scope=ci`
- `GET /release-orders/{id}/logs/stream?scope=cd`

规则：
- `scope=ci` 返回 CI 执行单元日志
- `scope=cd` 返回 CD 执行单元日志

### 7.6 管线进度
- `GET /release-orders/{id}/pipeline-stages?scope=ci`
- `GET /release-orders/{id}/pipeline-stages?scope=cd`
- `GET /release-orders/{id}/pipeline-stages/{stage_id}/log`

规则：
- 仅 Jenkins scope 返回真实 stage
- ArgoCD scope 可返回“同步中/已同步/失败”等部署状态

## 8. 权限要求
- 保持现有应用级发布权限模型不变
- 用户能看到某应用，即可看到该应用模板与发布单
- 用户能创建该应用发布单，即可创建包含 `CI/CD` 的模板实例发布
- 参数快照权限、日志权限、进度权限继续沿用发布单可见范围

## 9. 执行流程

### 9.1 模板同时包含 CI + CD
1. 读取模板主信息
2. 解析模板绑定：
- `ci`
- `cd`
3. 生成发布单主记录
4. 生成 `release_order_execution`
- 一条 `ci`
- 一条 `cd`
5. 生成按 scope 的参数快照
6. 生成按 scope 的执行步骤
7. 执行：
- 先 `ci`
- 成功后再 `cd`
8. 分别轮询：
- `CI` 日志与阶段
- `CD` 日志与阶段/状态

### 9.2 模板仅包含 CI
- 仅生成 `ci` 执行单元
- 仅展示 `CI` 模块

### 9.3 模板仅包含 CD
- 仅生成 `cd` 执行单元
- 仅展示 `CD` 模块

## 10. 兼容与迁移建议
- 现有单绑定模板：
- 自动迁移为只含一个 scope 的模板
- 若原绑定类型为 `ci`，迁为 `ci`
- 若原绑定类型为 `cd`，迁为 `cd`
- 现有发布单：
- 保持历史兼容，不强制回填为双执行单元
- 新逻辑对新建发布单生效

## 11. 验收标准
- 一个模板可同时配置 `CI` 与 `CD`
- 创建发布单时可同时提交 `CI/CD` 参数
- 发布后可分别查看 `CI/CD` 参数快照
- 发布后可分别查看 `CI/CD` 日志
- 发布后可分别查看 `CI/CD` 管线进度
- 无 `CD` 时页面与接口不返回 `CD` 模块数据
- Jenkins `CD` 与 ArgoCD `CD` 在模型层均可兼容

## 12. 实施顺序建议
- M1：模板模型升级（`release_template_binding` + scope 参数）
- M2：发布单模型升级（`release_order_execution` + scope 参数快照）
- M3：执行步骤、日志、管线进度 scope 化
- M4：前端创建页与详情页双视图联调
