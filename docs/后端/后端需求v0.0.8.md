## 2. 本版目标
在现有 `CI/CD` 双执行单元发布模型基础上，先完成 ArgoCD / GitOps 的三层能力：
- 组件管理下新增 `ArgoCD管理`，可查看、同步、跳转原始控制台
- 组件管理下新增 `GitOps管理`，可查看工作目录、仓库、分支、提交与工作区状态
- 发布链路支持 `CD=ArgoCD`
- 当 `CD=ArgoCD` 时，平台不直接“传参数给 ArgoCD”，而是：
  - 读取 ArgoCD Application 元数据
  - 操作本地 GitOps 工作目录
  - 修改声明式仓库中的 `kustomization.yaml`
  - 提交并推送 Git
  - 触发 ArgoCD Sync
  - 回收 ArgoCD 状态并写入发布执行状态

## 3. 核心原则
### 3.1 平台标准 Key
- 新增内置标准 Key：`image_version`
- 含义：平台统一的镜像版本输入
- 规则：
  - 由平台内置，用户不可编辑、不可删除
  - 所有启用中的内置标准 Key，都视为 `CI -> ArgoCD CD` 的必备输入
  - 当 `CD=ArgoCD` 且模板同时启用 `CI` 时，要求 `CI` 绑定管线先完成这些内置字段映射，并在模板参数中勾选

### 3.2 ArgoCD CD 模式
- `ArgoCD` 不走 Jenkins 风格的“执行器参数回传”
- 平台通过 GitOps 写回驱动 CD：
  1. 读取 ArgoCD Application 对应的 `repo_url`、`source_path`、`target_revision`
  2. 在本地 Git 工作目录中更新声明式文件
  3. 提交 commit 并 push
  4. 调用 ArgoCD Sync
  5. 轮询 Sync / Health 状态

### 3.3 绑定边界
- `应用管理 -> 管线绑定` 只负责维护 Jenkins 绑定
- `CD=ArgoCD` 不再通过“管线绑定”创建独立记录
- 是否走 ArgoCD，改由“发布模板”决定：
  - `CD` 选择 Jenkins 绑定：当前执行器为 `jenkins`
  - `CD` 不选择绑定：默认走 `argocd`

## 4. 范围
包含：
- 单实例 ArgoCD 配置、连通性检查、元数据同步
- `argocd_application` 元数据表
- 组件管理下的 ArgoCD 列表、详情、手动同步、原始链接
- 组件管理下的 GitOps 状态查询页
- GitOps 本地工作目录配置
- 受控的 GitOps 写回服务
- `CD=ArgoCD` 的发布执行初版

不包含：
- 多实例 ArgoCD 管理

- 任意 YAML 在线编辑
- Helm values 通用写回
- ArgoCD 事件流 / Pod 日志完整接入
- 自动从 CI 回推镜像产物

## 5. 配置设计
### 5.1 ArgoCD 配置
```json
{
  "argocd": {
    "enabled": true,
    "base_url": "https://192.168.49.125:30443",
    "insecure_skip_verify": true,
    "auth_mode": "token",
    "token": "${ARGOCD_TOKEN}",
    "username": "admin",
    "password": "${ARGOCD_PASSWORD}",
    "startup_check_enabled": true,
    "sync_enabled": true,
    "sync_interval_sec": 300,
    "request_timeout_sec": 30
  }
}
```

### 5.2 GitOps 配置
```json
{
  "gitops": {
    "enabled": true,
    "local_root": "/data/gitops",
    "default_branch": "master",
    "username": "gitops_bot",
    "password": "${GITOPS_PASSWORD}",
    "token": "",
    "author_name": "gos-bot",
    "author_email": "gos@example.com",
    "command_timeout_sec": 30
  }
}
```

### 5.3 配置规则
- `argocd.enabled=false`：不初始化 ArgoCD Client，不注册 ArgoCD 管理页同步任务
- `gitops.enabled=false`：允许使用 ArgoCD 管理页，但禁止 `CD=ArgoCD` 的执行能力
- `gitops.local_root`：平台本地 Git 工作目录
- 若 `local_root` 本身就是目标 GitOps 仓库，平台应直接在该仓库中操作，不再额外 clone 子目录
- `gitops.default_branch`：当 ArgoCD Application 未返回 revision 时使用的默认分支

## 7. GitOps 写回设计
### 7.1 当前版本支持的唯一写回策略
- 文件：`<source_path>/kustomization.yaml`
- 规则：更新 `images[].newTag`
- 限制：
  - 当前只支持单镜像条目
  - 不支持任意 YAML 路径写入
  - 不支持 Helm values 写回

### 7.2 执行步骤
1. 根据应用信息解析目标 ArgoCD Application
   - 平台运行时按应用 `app_key` 自动推导 `apps/<app_key>`
   - 首选按 GitOps 应用目录 + 标准 Key `env` 拼出真实 `source_path`
   - 拼接规则：`apps/<应用目录>/overlays/<env>`
   - 兼容历史数据按 Application 名称直接匹配
2. 读取 Application 的：
   - `repo_url`
   - `source_path`
   - `target_revision`
3. 从发布参数中解析 `image_version`
4. 准备本地 Git 工作目录
5. 修改 `kustomization.yaml`
6. 执行 `git add / commit / push`
7. 调用 `ArgoCD Sync`
8. 轮询 `SyncStatus / HealthStatus / OperationPhase`

### 7.3 本地工作目录规则
- 优先复用 `gitops.local_root` 本身作为工作仓库
- 若 `local_root` 不是目标仓库，再退化为“根目录 + 稳定命名工作副本”模式
- 每次写回前先 `fetch`、`checkout`、`pull --rebase`
- Git 作者统一使用平台配置

## 8. 发布模板校验
### 8.1 模板保存规则
当模板包含：
- `CI` 绑定
- `CD` 执行单元
- 且 `CD` 未选择 Jenkins 绑定（即走 `argocd`）

则校验：
- `CI` 绑定管线中，所有启用中的内置标准字段都必须已经映射到执行器参数
- `CI` 模板参数中，所有启用中的内置标准字段都必须已经被勾选
- 模板参数中仍必须包含 `env`

校验失败时返回：
- `cd 选择 argocd 时，ci 绑定管线必须先映射这些内置字段：...`
- `cd 选择 argocd 时，请在 ci 模板参数中勾选这些内置字段：...`

### 8.2 运行前兜底
创建/执行发布单时，若最终解析不到 `image_version`，必须直接失败，不进入 GitOps 写回。

## 9. 发布执行规则
### 9.1 `CD=ArgoCD` 执行逻辑
- `trigger_pipeline` 步骤：
  - 表示开始 GitOps 写回和触发 Sync
- `pipeline_running` 步骤：
  - 表示 ArgoCD Sync 已触发，等待状态回传
- `pipeline_success` 步骤：
  - 由 Tracker 根据 ArgoCD 状态推进

### 9.2 执行成功条件
满足以下条件时，CD 执行单元记为成功：
- `sync_status=Synced` 或空
- `health_status=Healthy`
- `operation_phase != Running`

### 9.3 执行失败条件
满足以下任一条件时，CD 执行单元记为失败：
- `operation_phase=Failed`
- `health_status=Degraded`

### 9.4 追踪容错
- 若 ArgoCD Application 暂时不存在：
  - 2 分钟内：返回“稍后自动重试”
  - 超过 2 分钟：标记执行失败

## 15. 风险与注意事项
- 当前 GitOps 写回只支持单镜像 `kustomization.yaml`
- Git 推送依赖工作目录权限、网络与 Git 凭据
- ArgoCD Application 的 `source_path` 与 `target_revision` 必须可信
- ArgoCD 已从“CD 绑定”迁移到“模板直接启用”；系统仍兼容历史“直接填 Application 名称”的旧数据
- `CD=ArgoCD` 模板必须包含标准 Key `env`，否则无法拼出实际 overlay 目录
- 当前版本未实现 ArgoCD 事件流，因此“日志模块”仍是占位态
