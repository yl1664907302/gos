# 后端需求文档 v0.1（未来规划）

## 1. 文档信息
- 文档版本：`v0.1`
- 基线日期：`2026-03-19`
- 参考版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端/后端需求v0.0.11.md`
- 本版主题：`生产孤岛 Agent + Shell 脚本执行链路`
- 当前状态：`未来规划，尚未开始实现`

## 2. 背景
当前平台已经具备：
- `CI` 侧 Jenkins 触发与跟踪能力
- `CD=ArgoCD` 的 GitOps / Helm values 写回能力

但在生产环境存在以下现实约束：
- 发布基建无法直接与生产内网互通
- 生产环境与发布平台之间存在防火墙隔离
- 生产侧仅开放少量公网入口，且不适合长期暴露高权限运维接口

因此 `v0.1` 需要引入“生产孤岛 Agent”模式：
- 平台不主动打入生产内网
- 生产侧 Agent 主动访问平台
- Agent 轮询、领取、执行、回传生产发布任务
- Agent 能主动拉取平台托管的 Shell 脚本并在本地执行

## 3. 目标
构建一套适用于“生产孤岛/网络隔离”场景的发布链路：
1. 平台继续作为统一发布入口
2. 生产侧通过 Agent 主动拉取待执行任务
3. Agent 在生产内网本地拉取并执行平台托管的 Shell 脚本
4. Agent 通过 Webhook / API 向 `gos` 回传状态、日志、结果
5. 平台前端统一展示生产发布进度
6. 平台托管脚本支持动态更新、版本管理、变量填充

## 4. 范围
包含：
- Agent 接入认证
- Agent 轮询任务
- Agent 领取任务
- Shell 脚本托管与下发
- Shell 脚本版本管理
- Shell 脚本动态更新策略
- Shell 脚本变量填充
- Agent 回传状态
- Agent 回传日志
- 发布单与执行单元对 Agent 模式的适配
- 生产环境“孤岛/小岛”维度的资源管理

不包含：
- Agent 自更新
- 平台主动 SSH 进入生产机
- 未经审核的任意脚本远程下发
- 多租户隔离细节设计

## 5. 核心概念
### 5.1 小岛（Island）
“小岛”表示一个受网络隔离保护、需要通过 Agent 接入的平台外部部署域。

建议字段：
- `island_code`
- `island_name`
- `env_code`
- `network_type`
- `status`
- `remark`

### 5.2 Agent
Agent 是部署在生产孤岛内部的轻量执行程序，职责包括：
- 轮询待执行任务
- 领取并锁定任务
- 拉取平台托管的 Shell 脚本或脚本包
- 本地执行部署动作
- 回传状态与日志

说明：
- 心跳仍建议保留，但在首版优先级上低于“脚本拉取 + 执行 + 回传”
- Agent 的核心价值不只是在线状态，而是成为生产孤岛内的标准执行器

### 5.3 Agent CD Provider
新增一种 CD 执行器类型：
- `provider = island-agent`

说明：
- `CI` 仍可继续使用 Jenkins
- `CD` 在生产隔离场景下由 `island-agent` 执行

## 7. 接口设计
### 7.1 Agent 认证
建议所有 Agent 接口统一走：
- `Authorization: Bearer <agent-token>`

后端需支持：
- Token 校验
- Agent 与 Island 绑定校验
- 过期/停用 Agent 拒绝访问

### 7.2 心跳接口
- `POST /agent/heartbeat`

请求体建议：
- `agent_code`
- `agent_name`
- `hostname`
- `version`
- `island_code`
- `status`

返回：
- 平台确认时间
- Agent 当前状态

说明：
- 心跳接口保留，但在首版中属于辅助能力
- 即使没有复杂心跳体系，也必须先有任务拉取、脚本下载、结果回传能力

### 7.3 轮询任务接口
- `POST /agent/tasks/poll`

用途：
- Agent 主动查询是否有可执行的生产发布任务

返回建议：
- 无任务时返回空
- 有任务时返回最小执行负载：
  - `release_order_id`
  - `release_order_no`
  - `execution_id`
  - `application_id`
  - `application_name`
  - `env_code`
  - `payload`

其中 `payload` 建议至少包含：
- `script_code`
- `script_version`
- `script_download_url`
- `script_checksum`
- `arguments`
- `timeout_sec`
- `working_dir`
- `render_variables`

### 7.4 领取任务接口
- `POST /agent/tasks/claim`

用途：
- Agent 显式领取任务并建立租约

### 7.5 脚本下载接口
- `GET /agent/scripts/:script_code`

用途：
- Agent 根据任务载荷拉取指定版本的脚本内容或压缩包

返回建议：
- `script_code`
- `version`
- `shell_type`
- `content`
- `checksum`
- `updated_at`
- `variables`

说明：
- Agent 拉取脚本时，应同时拿到脚本正文和变量元信息
- 平台需明确返回本次任务绑定的脚本版本

### 7.5.1 脚本版本策略
建议支持：
- 显式按版本下载：`script_code + version`
- 默认按最新启用版本下载：`script_code + latest_active`

首版建议：
- CI 完成后、CD 启动前冻结脚本版本
- 同一张发布单后续 CD 执行始终使用冻结版本
- 避免脚本动态更新后影响已创建任务的一致性

### 7.6 状态回传接口
- `POST /agent/tasks/report`

请求体建议：
- `release_order_id`
- `execution_id`
- `status`
- `step_code`
- `message`
- `started_at`
- `finished_at`

### 7.7 日志回传接口
- `POST /agent/tasks/logs`

请求体建议：
- `release_order_id`
- `execution_id`
- `pipeline_scope`
- `offset`
- `content`
- `timestamp`

说明：
- 日志应支持分段回传
- 平台需要支持断点续传与顺序拼接

## 8. 发布执行模型
### 8.1 创建发布单
当模板 `CD provider = island-agent`：
- 发布单创建成功后，`CD` 执行单元状态初始化为 `pending`
- 等待目标小岛 Agent 主动领取

### 8.2 Agent 领取并执行
流程建议：
1. Agent 轮询任务
2. Agent 领取任务
3. 平台将 `CD execution` 更新为 `running`
4. Agent 根据任务信息拉取 Shell 脚本
5. Agent 在生产内网本地执行脚本
5. Agent 周期性回传步骤、日志、状态
6. 成功/失败后关闭租约

### 8.3 执行动作边界
平台下发给 Agent 的应是“结构化任务 + 已托管脚本引用”，而不是任意即时 shell。

首版可支持：
- `argocd-sync`
- `helm-upgrade`
- `script-ref`

其中：
- `script-ref` 指向平台脚本中心中的脚本编码与版本
- Agent 通过脚本下载接口主动拉取
- 不允许平台在创建发布单时直接塞一段未审核 shell 文本

### 8.4 脚本执行建议
首版建议 Shell 执行统一采用：
- `/bin/bash`
- 临时工作目录
- 标准输出/错误输出分流采集
- 超时控制
- 非零退出码判失败

执行前建议记录：
- 脚本版本
- checksum
- 渲染后的参数
- 执行机器标识

### 8.5 变量填充
平台需要在“CI 完成后、CD 任务下发前”完成脚本变量渲染。

建议支持的占位格式：
- `{{ env }}`
- `{{ image_version }}`
- `{{ project_name }}`
- `{{ release_order_no }}`

变量来源建议包括：
- 发布单基础字段：如 `env`
- 平台标准 Key：如 `image_version`、`project_name`
- 固定值
- 小岛/Agent 上下文：如 `island_code`、`agent_name`

首版建议：
- 仅支持字符串替换
- 不支持条件判断、循环、任意函数执行
- 未取到必填变量时，发布直接失败

平台需同时保留：
- 原始脚本正文
- 渲染后脚本正文快照
- 渲染变量列表

### 8.6 动态更新策略
脚本支持动态更新，但必须遵循：
1. 平台脚本中心允许新增版本
2. 模板可指定“固定版本”或“默认使用最新启用版本”
3. 一旦 CI 成功并准备进入 CD，脚本版本立即冻结
4. 执行时只能拉取该冻结版本

## 9. 发布步骤与状态
### 9.1 CD 步骤建议
`provider = island-agent` 时，CD 步骤建议为：
- `cd:task_waiting`
- `cd:task_claimed`
- `cd:script_downloading`
- `cd:script_running`
- `cd:deploy_success`
- `cd:deploy_failed`

### 9.2 失败场景
需要明确区分：
- 没有在线 Agent
- 任务领取超时
- 脚本下载失败
- 脚本校验失败
- 变量渲染失败
- 脚本版本不存在
- Agent 执行失败
- 日志回传异常
- 结果回传超时

## 10. 安全要求
- Agent token 必须单独管理，不得复用用户 token
- 所有 Agent 接口必须记录审计日志
- Agent 仅允许访问所属 Island 的任务
- 建议支持 IP 白名单
- 禁止平台直接下发任意未审核 shell 内容
- 脚本下载建议附带 checksum/signature 校验

## 11. 运维与可观测
平台需具备：
- Island 列表
- Agent 在线状态
- 最近心跳时间
- 当前占用任务
- 最近执行结果
- 平台脚本列表
- 脚本版本历史
- 某次发布实际执行的脚本版本
- 某次发布的渲染后脚本快照
- 某次发布的变量替换结果

需支持：
- 手动停用 Agent
- 手动失效 Token
- 查看 Agent 回传日志

## 12. 与现有链路关系
### 12.1 CI
不变：
- 仍可使用 Jenkins

### 12.2 CD
新增：
- `provider = island-agent`

### 12.3 ArgoCD
未来可支持两种模式：
- 平台直连 ArgoCD（当前已有）
- Agent 在生产孤岛内调用本地 ArgoCD（未来扩展）

## 13. 实施建议
建议分三期推进：

### 一期
- Island / Agent 数据模型
- 轮询任务接口
- 任务领取接口
- 脚本托管接口
- 脚本下载接口
- 基础状态回传
- Shell 脚本执行与日志回传
- 基础变量填充
- CI 完成后冻结脚本版本

### 二期
- 心跳与 Agent 在线治理
- 前端 Agent 发布进度展示
- 超时与租约机制
- 脚本版本切换与回滚
- 渲染后脚本快照审计

### 三期
- Agent + 本地 ArgoCD 集成
- 多小岛并行调度
- 安全策略收口

## 14. 结论
`v0.1` 将把 `gos` 从“发布平台”进一步扩展为“可覆盖生产孤岛场景的统一发布控制面”。

其关键不在于平台主动打入生产，而在于：
- 平台暴露标准任务与回传接口
- 生产 Agent 主动拉取脚本与任务并执行
- 最终仍由平台统一管理发布记录、状态、日志与审计
