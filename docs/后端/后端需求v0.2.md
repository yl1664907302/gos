# 后端需求文档 v0.2（未来规划）

## 1. 文档信息
- 文档版本：`v0.2`
- 基线日期：`2026-03-30`

- 当前状态：`未来规划，尚未开始实现`

## 2. 背景
当前平台已经具备：
- Jenkins CI 执行与跟踪
- ArgoCD / GitOps CD 发布链路
- 发布单、回滚、并发执行、预检与排队

但特殊环境，尤其是生产环境，存在现实约束：
- 平台无法直连特殊网络区域
- 发布动作依赖人工在特殊网络中下载制品、拷贝文件、执行重启
- 当前流程缺少统一审计、任务状态、Agent 心跳与日志回传

因此 `v0.2` 需要把“生产执行任务”抽象成一种：
- 由 Agent 主动拉取
- 在受控工作目录内执行
- 基于命令模板与变量渲染
- 与发布单强关联
- 适用于特殊环境
的扩展执行模式

## 3. 定位
`agent_task` 不应替换平台所有环境的默认 CD。

它的定位应当是：
- 特殊环境的特殊执行模式
- 发布单的一种扩展执行器
- 平台不能直连目标网络时的标准方案

因此平台模型建议是：
- 普通环境继续使用 `gitops / pipeline`
- 特殊环境可配置 `execution_mode = agent_task`

## 4. 目标
后端需要支持：
1. Agent 注册、认证、心跳与状态管理
2. 特殊环境的 `agent_task` 执行模式
3. 生产执行任务创建、轮询、领取、执行、回传
4. 受控 Shell 任务模型，而不是任意远程命令
5. 任务与发布单、执行单元、日志、回滚的打通
6. Agent 管理接口与任务审计能力

## 5. 核心原则
### 5.1 不是任意 Shell
平台不能支持前端临时输入任意 shell 字符串。

应限制为：
- 平台托管的脚本模板
- 平台托管的命令模板
- 平台允许的变量来源
- 受控工作目录内执行

### 5.2 Agent 主动拉取
平台不主动进入特殊网络区域。
执行流程必须是：
- 平台建任务
- Agent 轮询接口
- Agent 领取任务
- Agent 本地执行
- Agent 回传状态与日志

### 5.3 特殊环境专用
`agent_task` 应当按环境或模板启用。
普通环境默认不启用。

### 5.4 任务必须可审计
每次任务执行必须能审计：
- 谁发起
- 哪个发布单
- 哪个 Agent 领取
- 哪个脚本模板
- 渲染后的命令/参数
- 开始/结束时间
- 退出码
- stdout/stderr

## 6. 新增执行模式
建议新增统一字段：
- `execution_mode`

建议枚举：
- `gitops`
- `pipeline`
- `agent_task`
- `manual`

使用建议：
- 模板、环境绑定或发布策略中可配置 `execution_mode`
- 命中特殊环境时，可从普通 `cd provider` 切换到 `agent_task`

## 8. Agent 认证与接入
### 8.1 认证方式
建议使用：
- `Authorization: Bearer <agent-token>`

要求：
- token 与 Agent 实例绑定
- 支持停用/重置 token
- 支持环境与标签范围校验

### 8.2 心跳
建议接口：
- `POST /agent/heartbeat`

请求建议：
- `agent_code`
- `hostname`
- `host_ip`
- `version`
- `status`
- `current_task_id`
- `current_concurrency`
- `work_dir`
- `artifact_dir`

用途：
- 更新在线状态
- 刷新当前任务与心跳时间
- 供 Agent 管理页展示

## 9. 任务接口
### 9.1 轮询任务
- `POST /agent/tasks/poll`

用途：
- Agent 主动查询是否有待执行任务

返回建议：
- 无任务：空
- 有任务：最小任务载荷
  - `task_id`
  - `release_order_id`
  - `release_order_no`
  - `execution_id`
  - `task_type`
  - `work_dir`
  - `command_template`
  - `resolved_command`
  - `variables`
  - `artifact_url`
  - `artifact_checksum`
  - `timeout_sec`

### 9.2 领取任务
- `POST /agent/tasks/:id/claim`

用途：
- Agent 显式领取任务，防止多 Agent 重复消费

### 9.3 任务开始
- `POST /agent/tasks/:id/start`

### 9.4 日志回传
- `POST /agent/tasks/:id/logs`

说明：
- 支持 stdout/stderr
- 建议分批上报
- 后端可截断超长日志并标记截断状态

### 9.5 任务结束
- `POST /agent/tasks/:id/finish`

请求建议：
- `status`
- `exit_code`
- `failure_reason`
- `summary`

## 10. 执行链路
### 10.1 建单
- 用户创建发布单
- 平台照常做预审、预检、审批、并发控制

### 10.2 命中特殊环境
如果当前环境命中 `execution_mode = agent_task`：
- 不走 ArgoCD / 普通 Jenkins CD
- 生成 `release_agent_task`
- 发布单/执行单元进入“等待 Agent 领取”状态

### 10.3 Agent 执行
- Agent 轮询任务
- 领取任务
- 在工作目录内执行命令模板
- 回传日志、状态、结果

### 10.4 平台收口
- 成功：发布单进入成功终态
- 失败：发布单进入失败终态
- 可取消：平台能取消待领取/执行中的任务

## 11. 命令模板与变量
### 11.1 命令模板
命令模板建议支持：
- 脚本路径模板
- 命令参数模板
- 工作目录模板

例如：
- `scripts/deploy.sh {app_key} {env} {artifact_url}`
- `scripts/restart.sh {project_name}`

### 11.2 变量建议
建议支持：
- `{order_no}`
- `{app_key}`
- `{project_name}`
- `{env}`
- `{artifact_url}`
- `{artifact_checksum}`
- `{git_sha64}`
- `{image_version}`
- `{operation_type}`
- `{source_order_no}`

### 11.3 安全限制
建议明确限制：
- 命令模板只能来自平台配置
- 渲染变量只能来自平台允许来源
- 运行路径必须限制在 Agent 工作目录内
- 禁止路径逃逸
- 禁止临时拼接未经校验的用户输入

## 14. 并发与锁
即使是 Agent 任务，也需要复用平台已有并发控制：
- 同 app 同 env 不并发
- 同批并发单可按队列执行
- Agent 自身也有 `max_concurrency`

因此后端需要同时考虑：
1. 发布单级互斥
2. Agent 侧接单能力
3. Agent 任务执行队列

## 15. 回滚
Agent 任务需要支持：
- 标准回滚
- 重放回滚

表现形式：
- 平台生成新的 `release_order`
- 再生成对应的 `release_agent_task`
- 渲染回滚命令模板与变量

## 16. 审计与通知
### 16.1 审计
必须记录：
- 发起人
- 审批人
- Agent
- 模板版本
- 渲染后命令
- 变量
- 日志摘要
- 退出码

### 16.2 通知
建议通知：
- 任务已创建
- Agent 已领取
- 执行成功
- 执行失败
- Agent 离线导致任务未领取

## 17. 前后端协作边界
前端只展示：
- 模板
- Agent
- 任务
- 变量
- 结果

后端负责：
- 环境命中执行模式
- 任务创建与调度
- Agent 认证
- 命令模板渲染
- 审计落库

## 18. 验收标准
后端侧验收建议：
1. 特殊环境可命中 `agent_task` 模式
2. Agent 可注册、心跳、轮询、领取、执行、回传
3. 任务执行全链路可追踪
4. 任务仅允许受控模板执行，不支持任意命令输入
5. 平台可查看 Agent 在线状态、主机信息、最近结果
6. 发布单、回滚、并发控制与 Agent 任务可打通
