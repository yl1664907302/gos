# 后端需求文档 v0.3（需求设计）

## 1. 文档信息
- 文档版本：`v0.3`
- 基线日期：`2026-03-30`
- 参考版本：`/Users/lingyunxieqing/Desktop/gos/docs/后端/后端需求v0.2.md`
- 本版主题：`发布后 Hook 模块`
- 当前状态：`需求设计，待开发`

## 2. 背景
当前平台已经支持：
- Jenkins CI / CD
- ArgoCD / GitOps 发布链路
- Agent 任务、脚本管理、常驻任务
- 发布单、回滚、并发执行、排队与预检

但主发布流程结束后，平台缺少一个标准扩展点来执行“发布后动作”。
典型场景包括：
- 发布成功后触发烟雾测试
- 发布成功后执行缓存刷新、配置拉取、二次同步
- 发布失败后执行补偿脚本
- 在特殊环境中执行额外 Agent 脚本

因此 `v0.3` 需要把“发布后 Hook”抽象成一个标准模块，配置在发布模板中，并在主发布流程结束后执行。

## 3. 目标
后端需要支持：
1. 发布模板级 Hook 配置
2. 主发布流程结束后的 Hook 调度与执行
3. Jenkins / Agent 两类 Hook 执行器
4. Hook 触发条件、失败策略、串行顺序
5. Hook 与发布单、执行单元、日志、Agent 任务的关联
6. Hook 结果对发布单最终状态的影响

## 4. 核心原则
### 4.1 Hook 是主发布流程的扩展阶段
Hook 不是独立工单，也不是普通任务。
它必须：
- 绑定到发布模板
- 派生到具体发布单
- 在主发布流程结束后执行

### 4.2 Hook 不替代主发布
主发布流程仍然是：
- CI
- CD
- 健康检查
- Agent 特殊环境发布

Hook 是其后的扩展阶段。

### 4.3 Hook 首版只做发布后阶段
首版只支持：
- `post_release`

不做：
- pre_release hook
- mid_release hook
- 多阶段 hook graph

### 4.4 Hook 必须可审计
需要完整记录：
- 来源模板
- 来源发布单
- Hook 名称
- 类型
- 触发条件
- 失败策略
- 变量解析值
- 开始/结束时间
- 日志与失败原因

## 5. Hook 模型
### 5.1 Hook 阶段
建议枚举：
- `post_release`

### 5.2 Hook 触发条件
建议枚举：
- `on_success`
- `on_failed`
- `always`

### 5.3 Hook 类型
首版建议支持：
- `jenkins_pipeline`
- `agent_task`

### 5.4 Hook 失败策略
建议枚举：
- `block_release`
- `warn_only`

### 5.5 Hook 顺序
- 模板内支持多个 Hook
- 按 `sort_no` 串行执行
- 只有当前一个 Hook 完成后才进入下一个

## 7. 发布模板配置
### 7.1 模板保存
保存模板时：
- 同步保存 Hook 列表
- 同步保存变量映射
- 校验排序与名称唯一性

### 7.2 配置校验
后端应校验：
- Hook 类型是否合法
- 触发条件是否合法
- 失败策略是否合法
- Jenkins Hook 是否绑定有效 Pipeline
- Agent Hook 是否绑定有效脚本
- 变量来源是否合法
- `sort_no` 是否有效

## 8. 执行链路
### 8.1 主发布流程结束
当主发布流程结束后：
1. 读取模板中的 `post_release` Hook
2. 根据主流程结果判断哪些 Hook 应该执行
3. 创建 `release_order_hook_execution`
4. 按顺序串行执行

### 8.2 Hook 跳过
如果触发条件不满足：
- 该 Hook 创建执行记录并写为 `skipped`
- 摘要中保留跳过说明

### 8.3 Hook 执行成功
- 当前 Hook 状态改为 `success`
- 继续执行下一个 Hook

### 8.4 Hook 执行失败
若失败策略为 `block_release`：
- 当前 Hook 状态改为 `failed`
- 停止后续 Hook
- 整张发布单最终进入失败终态

若失败策略为 `warn_only`：
- 当前 Hook 状态改为 `warn_ignored`
- 继续执行后续 Hook
- 发布单最终保持主流程结果

## 9. Jenkins Hook
### 9.1 配置
`hook_type = jenkins_pipeline`

`provider_config_json` 建议包含：
- `jenkins_instance_id`
- `pipeline_id`
- `pipeline_name`
- `params_mapping`

### 9.2 执行
- 触发指定 Jenkins Pipeline
- 记录构建号、链接、启动结果
- 可轮询阶段摘要或至少记录最终结果

## 10. Agent Hook
### 10.1 配置
`hook_type = agent_task`

`provider_config_json` 建议包含：
- `script_id`
- `script_name`
- `agent_selector_json`
- `work_dir`
- `timeout_sec`

### 10.2 执行
- 复用现有 Agent 任务体系
- 为 Hook 生成一条临时 Agent 任务
- 任务来源标识为：`release_hook`
- 任务执行结果回写到 `release_order_hook_execution`

### 10.3 日志
- 复用 Agent stdout/stderr 回传
- 挂接到 Hook 日志视图

## 11. 变量解析
### 11.1 变量来源顺序
建议优先级：
1. Hook 固定值
2. 发布单当前参数快照
3. 标准平台 Key 解析值
4. 内置字段
5. 来源成功单字段（回滚场景）

### 11.2 常见变量
建议默认支持：
- `app_key`
- `project_name`
- `env`
- `image_version`
- `branch`
- `git_ref`
- `order_no`
- `operation_type`
- `source_order_no`

## 12. 状态机建议
主发布流程与 Hook 的关系建议为：
- 主发布完成前：Hook 相关记录为 `pending`
- 主发布完成后：Hook 才进入 `running/skipped`
- Hook 全部完成后：收口整张发布单最终状态

建议新增步骤编码：
- `hook:prepare`
- `hook:<hook_code>`
- `hook:summary`

## 13. 接口建议
### 13.1 模板接口
- `GET /release-templates/:id/hooks`
- `PUT /release-templates/:id/hooks`

### 13.2 发布单详情接口
- `GET /release-orders/:id/hooks`
- `GET /release-orders/:id/hooks/:hookExecutionID/logs`

要求：
- `/release-orders/:id` 或详情聚合接口需要返回 Hook 执行摘要，供详情页顶部直接展示进度
- 至少包含：`has_hook`、`hook_summary_status`、`hook_running_count`、`hook_failed_count`、`hook_total_count`

### 13.3 Hook 重试（后续）
首版可不开放；若后续开放建议为：
- `POST /release-orders/:id/hooks/:hookExecutionID/retry`

## 14. 权限建议
建议新增：
- `release.template.hook.view`
- `release.template.hook.edit`
- `release.hook.view`
- `release.hook.log.view`
- `release.hook.retry`

## 15. 首版限制
首版建议限制：
1. 只支持 `post_release`
2. 只支持串行执行
3. 只支持 `jenkins_pipeline / agent_task`
4. 不支持浏览器临时输入任意脚本
5. 不支持运行中修改 Hook 配置
6. Hook 失败后的自动重试可后续再做

## 16. 验收要点
1. 模板中可配置多个发布后 Hook
2. 主发布流程结束后按顺序触发 Hook
3. Hook 能正确根据成功/失败条件决定是否触发
4. `block_release / warn_only` 能正确影响最终发布结果
5. Agent Hook 能复用现有脚本管理和 Agent 任务体系
6. 发布单详情能完整查看 Hook 状态、变量、日志与失败原因
7. 发布单详情主视图能直接展示 Hook 执行进度摘要，不依赖额外跳转
