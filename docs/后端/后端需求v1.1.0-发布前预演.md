## 2. 背景
当前后端已经具备：
- 发布单创建
- 执行前预检 `PrecheckExecute`
- 参数取值进度计算
- Hook 配置与执行
- 并发锁与排队判断
- 审批流判断
- GitOps 规则处理

但这些能力大多建立在“发布单已经创建”的前提下。
当前缺少一个“建单前的统一发布计划计算接口”，导致：
- 创建前无法完整回答这次会怎么执行
- 极速发布缺少足够透明的事前可见性
- 前后端都难以围绕同一份发布计划协作

因此需要新增 `发布前预演` 能力。

## 3. 目标
后端需要支持：
1. 在不落库、不触发外部执行的前提下，构建一份完整发布计划
2. 输出审批、并发、参数、执行单元、Hook、GitOps 的预演结果
3. 让 `preview` 与 `create` 共用同一套计划构建逻辑
4. 为创建页、极速发布、后续续跑等能力提供统一基础模型

## 4. 核心原则

### 4.1 预演只计算，不执行
预演接口不得：
- 创建发布单
- 创建执行单元持久化记录
- 占用并发锁
- 调用 Jenkins / ArgoCD / Agent 真正执行
- 发送通知
- 写入审计终态

### 4.2 预演结果必须尽量接近真实执行计划
预演不能只是“浅校验”，而应尽量复用真实创建 / 执行前的计算链路，确保：
- 参数来源一致
- 审批判断一致
- Hook 命中结果一致
- GitOps 目标一致
- 并发判断口径一致

### 4.3 创建与预演共用计划构建器
建议抽象：
- `ReleasePlanBuilder`

要求：
- `PreviewReleasePlan` 调用该构建器
- `CreateReleaseOrder` 也调用该构建器
- 避免后续出现“预演显示 A，实际创建成 B”

## 5. 建议接口
建议新增：
- `POST /release-orders/preview`

请求体建议与创建发布单保持一致：
- `application_id`
- `template_id`
- `env_code`
- `git_ref`
- `trigger_type`
- `remark`
- `params`

说明：
- 即使 `trigger_type / remark` 在预演中不是核心，也建议保持结构一致，减少 DTO 分裂

## 7. 计划构建器建议
建议抽象以下步骤：
1. 读取应用、模板、绑定、参数、Hook、GitOps 规则
2. 校验环境权限与模板适配性
3. 构造内存中的发布计划对象
4. 解析最终参数
5. 计算审批结果
6. 计算执行单元摘要
7. 计算 Hook 命中结果
8. 计算 GitOps 目标摘要
9. 计算并发 / 队列预判

建议新增内部模型：
- `ReleasePlan`
- `ReleasePlanExecution`
- `ReleasePlanParam`
- `ReleasePlanHook`
- `ReleasePlanGitOps`

## 9. 错误处理要求
预演接口的错误要分层：

### 9.1 业务阻断
例如：
- 模板不存在
- 环境未授权
- 必填参数缺失
- 模板绑定失效且无可回退快照

建议：
- 正常返回 `200`
- 在 `blocked items` 中体现
- 不直接让接口失败

### 9.2 系统异常
例如：
- 数据库错误
- 仓储查询异常
- 模板加载异常

建议：
- 返回 `5xx`
- 前端可重试

## 10. 与创建发布单的关系
创建发布单应逐步调整为：
1. 调用统一计划构建逻辑
2. 若存在阻断项，则拒绝创建
3. 若无阻断项，则将计划对象持久化为真实发布单、执行单元、步骤、参数、Hook 快照

也就是说：
- `preview` 是计划的只读投影
- `create` 是计划的持久化落地

## 12. 验收标准
以下视为后端完成：
1. 提供 `POST /release-orders/preview`
2. 接口不落库、不执行外部动作
3. 能返回审批、预检、参数、执行单元、Hook、GitOps 摘要
4. 创建发布单与预演能复用同一套计划构建逻辑
5. 预演结果与真实创建后的发布计划不存在明显口径漂移
