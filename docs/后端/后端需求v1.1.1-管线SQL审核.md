## 2. 背景
当前平台已经具备：
- 发布单创建 / 执行 / 回滚 / 取消
- 执行前预检
- 模板、审批、并发、Hook、Agent 等主链路能力

但数据库变更语句仍缺少统一入口。
本期并不引入 SQL 执行能力，而是聚焦在：
- SQL 备案
- SQL 留档
- SQL 审计
- SQL 语法与基础安全审核
- 执行前阻断

## 3. 目标
后端需要支持：
1. 发布单可选挂载 SQL 备案信息
2. 默认发布单视为“无 SQL 变更”
3. 极速发布请求一律按“无 SQL 变更”处理
4. 在执行前预检中新增 SQL 审核项
5. SQL 审核不通过时阻断发布执行
6. SQL 审核结果可在发布单详情与审计链路中追踪

## 4. 核心原则

### 4.1 v1 不执行 SQL
本期明确不做：
- 数据库真实执行
- 真实库连接校验
- 在线 DDL
- SQL 回滚执行

### 4.2 SQL 只作为发布单的伴随备案信息
SQL 备案的作用是：
- 记录这次发布涉及哪些 SQL 变更
- 在执行前做审核
- 形成审计链路

而不是变成发布执行单元。

### 4.3 执行前审核才做强阻断
- 创建发布单时允许用户带 SQL 备案正常建单
- 真正点击执行时，才进行 SQL 审核并决定是否阻断

### 4.4 极速发布强制无 SQL
- 极速发布的请求体即使携带 SQL，也应被后端忽略或拒绝
- 平台语义上，极速发布 = 无 SQL 的快速发版路径

## 6. 创建发布单接口调整
### 6.1 普通创建
创建发布单请求建议支持新增：
- `sql_record`
  - `enabled`
  - `title`
  - `change_type`
  - `sql_content`
  - `change_remark`
  - `rollback_note`

规则：
- 未传或 `enabled=false` 时，视为无 SQL
- 允许创建成功，不在创建时做最终阻断

### 6.2 极速发布
规则建议：
- 若请求为极速发布路径，则后端统一按无 SQL 处理
- 若仍收到 SQL 字段，可选两种策略：
  1. 忽略 SQL 字段
  2. 返回校验错误

建议选项：
- `返回校验错误` 更稳

错误文案建议：
- `极速发布不支持 SQL 备案，请使用标准创建发布单`

## 7. 执行前预检扩展
当前已有：
- `/Users/lingyunxieqing/Desktop/gos/internal/application/usecase/release_order_precheck.go`

建议新增预检项：
- `sql_review`

执行顺序建议：
1. SQL 审核
2. 发布单状态检查
3. 执行单元检查
4. 模板绑定 / 快照回退检查
5. 并发与队列检查

原因：
- SQL 审核是最先要回答“这张单是否允许发”的问题

## 8. SQL 审核规则 v1

### 8.1 审核结果分级
- `pass`
- `warn`
- `blocked`

### 8.2 规则建议
第一版仅做静态规则，建议包含：
1. SQL 为空：`blocked`
2. SQL 语法错误：`blocked`
3. `UPDATE` 无 `WHERE`：`blocked`
4. `DELETE` 无 `WHERE`：`blocked`
5. `DROP TABLE`：`warn` 或生产环境 `blocked`
6. `TRUNCATE`：`warn` 或生产环境 `blocked`
7. `ALTER TABLE`：`warn`
8. 多条高风险语句并存：`warn`

### 8.3 无 SQL 情况
若发布单未配置 SQL 备案，则返回：
- `status = pass`
- `message = 本次发布未配置 SQL 备案`

## 9. 审核实现建议
Go 后端建议优先使用 SQL parser，而不是自行手写规则解析。
建议优先考虑：
- `pingcap/parser`
- 或其他兼容 MySQL 方言的 parser

实现步骤建议：
1. 解析 SQL 文本
2. 拆分语句类型
3. 对每条语句应用规则
4. 聚合总体风险等级
5. 生成预检项结果与摘要

## 10. 返回结构建议
在现有 `ReleaseOrderPrecheckOutput` 中，新增一项：
- `sql_review`

具体表现为：
- 在 `items[]` 中增加：
  - `key = sql_review`
  - `name = SQL 审核`
  - `status = pass / warn / blocked`
  - `message = 审核结果摘要`

如需更细粒度，建议额外扩展：
- `sql_review_details[]`
  - `rule_key`
  - `rule_name`
  - `status`
  - `message`
  - `statement_excerpt`

## 11. 审计要求
需要沉淀：
- 谁提交了 SQL 备案
- SQL 原文是什么
- 执行前审核结果是什么
- 哪次执行被 SQL 阻断过
- 审核结果摘要与风险等级

即使 v1 不执行 SQL，也要保证它能被后续审计追溯。

## 12. 与详情页的关系
详情页应能读取：
- SQL 备案内容
- 最新审核结果
- 风险等级
- 阻断原因

若用户点击执行时被阻断，应能在详情页直接看到：
- 是哪一条规则未通过
- 对应 SQL 片段或类型

## 13. 与发布前预演的关系
后续可与发布前预演联动：
- 若创建页已存在 SQL 备案
- 则预演中也展示 SQL 审核预估结果

但本期第一优先级仍是：
- 执行前预检阻断

## 14. 验收标准
以下视为后端完成：
1. 发布单支持可选 SQL 备案字段并落库
2. 默认发布单为无 SQL 变更
3. 极速发布不允许携带 SQL 备案
4. 执行前预检中新增 SQL 审核项
5. SQL 语法或安全规则不通过时阻断执行
6. 发布单详情可查询 SQL 备案与审核结果
7. 整体能力不涉及真实 SQL 执行
