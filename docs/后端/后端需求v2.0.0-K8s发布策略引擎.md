# 后端需求文档 v2.0.0（K8s 发布策略引擎）

## 1. 文档信息
- 文档版本：`v2.0.0`
- 基线日期：`2026-04-17`
- 上一阶段参考：`/Users/lingyunxieqing/Desktop/gos/docs/后端/后端需求v1.1.1-管线SQL审核.md`
- 本版主题：`K8s 发布策略引擎`
- 当前状态：`需求设计，待开发`

## 2. 背景
当前平台已经具备：
- 应用 / 项目 / 环境模型
- 发布模板与执行单元模型
- Jenkins / ArgoCD / GitOps 主链路
- 发布单、确认生效、标准回滚 / 标准重放
- Hook、Agent、通知、并发、审批

但针对 K8s 发布行为，平台仍存在明显缺口：
- 发布策略仍主要分散在 YAML / Helm values 中
- 平台缺少统一的 K8s 发布策略抽象
- 平台缺少统一的 K8s 目标实例 / 运行时实例抽象
- 无法明确表达“怎么发布”和“通过什么链路交付”
- 回滚只能恢复版本快照，不能完整恢复策略快照
- 后续接入 Argo Rollouts 时，若数据模型未预留，会导致再次重构

因此 v2 需要先把 K8s 发布策略平台化，并为中期接入 Argo Rollouts 做模型预留。

## 3. 目标
后端需要支持：
1. 建立统一的发布策略领域模型
2. 明确区分 `交付引擎` 与 `发布策略引擎`
3. 引入 `K8s 目标实例 / 运行时实例` 模型
4. v2.0 先支持原生 K8s 策略：
   - `RollingUpdate`
   - `Recreate`
   - 探针
   - 资源
   - PDB
5. 发布单执行时生成并持久化 `策略快照`
6. 标准回滚时除版本快照外，同时支持恢复策略快照
7. 为中期 Argo Rollouts 的 `Canary / BlueGreen` 预留数据结构与执行入口

## 4. 核心原则

### 4.1 交付方式与发布策略必须拆成两个维度
后端模型必须区分：
- `delivery_engine`：怎么把变更送到环境
- `strategy_engine`：到了 K8s 之后怎么滚动发布

不允许把它们混成一个 `engine` 字段。

建议定义：
- `delivery_engine`
  - `argocd`
  - `jenkins`
  - `agent`
- `strategy_engine`
  - `none`
  - `k8s_native`
  - `argo_rollouts`

说明：
- `ArgoCD` 是交付引擎，不是发布策略引擎
- `Argo Rollouts` 是高级发布策略引擎，不是 GitOps 同步引擎

### 4.2 v2.0 先做原生 K8s 策略平台化
本期先完成：
- 原生 `Deployment` 的 `RollingUpdate / Recreate`
- 探针
- 资源
- PDB

不在本期实现：
- `Canary`
- `BlueGreen`
- `Analysis`
- 流量路由

### 4.3 不做自由 YAML 配置器
平台不应直接暴露整段任意 YAML 让用户自行编辑。

本期策略配置应以结构化字段落库，再由渲染器写入：
- Helm values
- Kustomize patch
- 其他 GitOps 载体

### 4.4 策略也必须纳入快照体系
发布成功后，除现有执行快照 / 部署快照外，必须新增：
- `strategy_snapshot`

原因：
- 标准回滚恢复的不只是镜像 / tag
- 还应恢复当时生效的发布策略

### 4.5 策略能力按 `app + env` 绑定
策略不建议只挂在模板上。

应采用：
- `application_id + env_code -> strategy_binding`

模板负责：
- 执行链路
- 参数映射
- Hook / 审批 / 并发规则

策略绑定负责：
- 当前环境下的 K8s 发布行为

### 4.6 平台里最好开始引入一层 K8s 目标实例 / 运行时实例模型
v2.0 虽然不要求平台直接保存 kubeconfig 并直连 Kubernetes，但平台必须开始引入一层：
- `K8s 目标实例 / 运行时实例`

这层模型的职责是统一回答：
- 当前环境落在哪个集群
- 当前环境对应哪个 namespace / workload / service
- 当前环境是否支持原生 K8s 策略
- 当前环境是否支持 Argo Rollouts
- 当前环境与哪个 ArgoCD 实例 / 应用关联

若缺失这层模型，后续接入：
- 策略预检
- Rollouts
- 标准回滚策略恢复
- 集群能力判断

都只能继续依赖模板参数和人工约定，无法平台化。

## 5. 范围

### 5.1 本期包含
- 原生 K8s 发布策略模型
- K8s 目标实例 / 运行时实例模型
- `app + env` 级策略绑定
- 发布单策略快照
- 标准回滚恢复策略快照
- 后端渲染与预检能力
- 与 ArgoCD / GitOps 的联动

### 5.2 本期不包含
- Rollout CRD 真正落地
- 金丝雀 / 蓝绿流量控制
- AnalysisTemplate
- 自动晋级 / 自动回滚
- Service Mesh / Ingress 流量切分

## 6. 数据模型建议

### 6.1 发布策略模板表
建议新增：
- `release_strategy_template`

字段建议：
- `id`
- `name`
- `description`
- `strategy_engine`
- `strategy_type`
- `config_json`
- `status`
- `creator_id`
- `creator_name`
- `created_at`
- `updated_at`

说明：
- 模板负责定义“这类策略长什么样”
- `config_json` 采用结构化 JSON，不直接存整段 YAML

### 6.2 应用环境策略绑定表
建议新增：
- `application_env_strategy_binding`

字段建议：
- `id`
- `application_id`
- `env_code`
- `delivery_engine`
- `strategy_engine`
- `strategy_type`
- `strategy_template_id`
- `strategy_config_override_json`
- `allow_release_override`
- `override_whitelist_json`
- `status`
- `created_at`
- `updated_at`

说明：
- 同一应用不同环境可绑定不同策略
- `override_whitelist_json` 用于限制发布时允许覆盖的少量字段

### 6.3 K8s 目标实例表
建议新增：
- `k8s_instance`

字段建议：
- `id`
- `name`
- `code`
- `cluster_name`
- `environment_code`
- `api_server`
- `default_namespace`
- `access_mode`
- `argocd_instance_id`
- `agent_group_id`
- `supports_native_strategy`
- `supports_rollouts`
- `traffic_provider`
- `ingress_provider`
- `service_mesh_provider`
- `status`
- `remark`
- `created_at`
- `updated_at`

说明：
- v2.0 可先只做“目标实例元数据”，不要求持久化 kubeconfig
- `access_mode` 建议预留：
  - `argocd`
  - `agent`
  - `direct`

### 6.4 应用环境运行时绑定表
建议新增：
- `application_env_runtime_binding`

字段建议：
- `id`
- `application_id`
- `env_code`
- `k8s_instance_id`
- `namespace`
- `workload_name`
- `service_name`
- `ingress_name`
- `argo_app_name`
- `status`
- `created_at`
- `updated_at`

说明：
- 用于明确 `app + env` 的真实运行时归属
- 后续策略渲染、回滚恢复、能力预检都优先依赖这张表，而不是临时从模板参数倒推

### 6.5 发布单策略快照字段
建议在发布单或关联快照表中新增：
- `strategy_snapshot_json`

快照建议至少包含：
- `delivery_engine`
- `strategy_engine`
- `strategy_type`
- `template_id`
- `template_name`
- `resolved_config`
- `render_target_kind`
- `render_target_api_version`
- `render_metadata`

### 6.6 app_release_state 增补
建议在现有 `app_release_state` 中补充：
- `strategy_engine`
- `strategy_type`
- `strategy_snapshot_json`
- `k8s_instance_id`
- `namespace`
- `workload_name`

原因：
- 当前生效版本不应只记录版本号
- 还要记录“当前生效策略”
- 还要记录“当前策略实际落在什么运行时目标上”

## 7. 策略配置结构建议

### 7.1 通用字段
所有策略统一建议支持：
- `replicas`
- `resource_requests`
- `resource_limits`
- `startup_probe`
- `readiness_probe`
- `liveness_probe`
- `pdb`

### 7.2 k8s_native / rolling_update
建议结构：
- `rolling_update.max_surge`
- `rolling_update.max_unavailable`
- `min_ready_seconds`
- `progress_deadline_seconds`
- `revision_history_limit`

### 7.3 k8s_native / recreate
建议结构：
- `termination_grace_period_seconds`
- `min_ready_seconds`
- `progress_deadline_seconds`

### 7.4 预留 argo_rollouts / canary
本期不执行，但结构先预留：
- `steps`
- `set_weight`
- `pause_duration`
- `analysis_templates`
- `auto_promotion`

### 7.5 预留 argo_rollouts / blue_green
本期不执行，但结构先预留：
- `active_service`
- `preview_service`
- `auto_promotion`
- `scale_down_delay_seconds`
- `preview_verification_required`

## 8. 组合关系与执行边界

### 8.1 合法组合建议
v2.0 主要支持：
1. `delivery_engine = argocd` + `strategy_engine = k8s_native`
2. `delivery_engine = jenkins` + `strategy_engine = none`
3. `delivery_engine = agent` + `strategy_engine = none`

说明：
- 只有进入 K8s / GitOps 渲染链路的场景，才真正使用 K8s 发布策略
- Jenkins / Agent 直连脚本场景，暂不强行套 K8s 策略

### 8.2 v2.1 预留组合
后续扩展为：
- `delivery_engine = argocd` + `strategy_engine = argo_rollouts`

## 9. 发布执行链调整

### 9.1 创建发布单阶段
创建时需要解析：
1. 应用
2. 环境
3. 模板
4. `application_env_strategy_binding`

生成：
- `resolved_strategy_config`

并写入发布单快照。

### 9.2 执行前预检新增策略检查
在现有预检基础上新增：
- `strategy_precheck`

建议校验：
1. 当前环境是否存在策略绑定
2. `delivery_engine` 与模板执行单元是否兼容
3. `strategy_engine` 与目标执行路径是否兼容
4. 探针配置是否合法
5. 资源配置是否合法
6. PDB 配置是否合法
7. 发布时覆盖字段是否超出白名单

### 9.3 GitOps 渲染阶段
当前已有 GitOps values 更新逻辑，本期需要增加：
- 从策略快照渲染 Deployment 策略块
- 渲染探针
- 渲染资源
- 渲染 PDB
- 结合 `application_env_runtime_binding` 解析目标 namespace / workload / argo_app

建议抽出：
- `StrategyRenderer`

职责：
1. 输入 `strategy_snapshot`
2. 输出标准化渲染结果
3. 交给 GitOps values rule 写回器落地

## 10. 标准回滚与策略快照

### 10.1 回滚基线
标准回滚仍以：
- `app_release_state.current_live`
- `app_release_state.previous_live_state_id`

作为来源。

### 10.2 回滚恢复内容
标准回滚恢复内容必须包含：
1. 目标版本参数快照
2. 目标部署快照
3. 目标策略快照

不能只恢复镜像 / tag。

### 10.3 回滚执行链
当 `delivery_engine = argocd` 时：
1. 加载目标 `deploy_snapshot`
2. 加载目标 `strategy_snapshot`
3. 渲染回滚后的 GitOps values
4. Git commit / push
5. ArgoCD Sync
6. 健康检查
7. 进入 `pending_confirm`

## 12. 与现有模板模型的关系
本期不要求把策略绑定到模板主表上，但模板详情接口可聚合返回：
- 当前模板适配的策略引擎
- 当前应用环境解析出的策略摘要

原则：
- 模板负责“怎么执行”
- 策略绑定负责“怎么滚动发布”
- 运行时绑定负责“实际落在哪个 K8s 目标实例上”

## 13. K8s 实例模型的边界

### 13.1 v2.0 先做元数据，不强上直连凭据
本期建议：
- 建立 `k8s_instance`
- 建立 `application_env_runtime_binding`
- 与 ArgoCD 实例、应用、namespace 建立关联

本期不要求：
- 平台保存 kubeconfig
- 平台直接调用 Kubernetes API
- 平台承担多集群凭据轮换

### 13.2 为什么要先建模
即使平台当前仍主要通过：
- GitOps
- ArgoCD

完成交付，后端仍然需要知道：
- 目标集群是谁
- 目标 namespace 是谁
- 目标 workload 是谁
- 当前环境是否具备策略能力 / Rollouts 能力

否则：
- 策略预检无法平台化
- 回滚恢复无法明确目标
- 后续 Rollouts 接入只能再次重构

## 14. 迁移策略

### 13.1 存量应用
未配置策略绑定的存量应用，默认解析为：
- `delivery_engine = 原执行链路推断值`
- `strategy_engine = none`

对已有 ArgoCD + K8s 应用，可给出默认迁移规则：
- `strategy_engine = k8s_native`
- `strategy_type = rolling_update`

### 13.2 渐进启用
建议按以下顺序：
1. dev
2. test
3. prod

避免一次性全量迁移。

## 15. 与中期 Argo Rollouts 的衔接
本期虽然不落地 Rollouts，但必须预留：
- `strategy_engine = argo_rollouts`
- `strategy_type = canary / blue_green`
- `strategy_config_json` 中的 rollout 字段
- `strategy_snapshot_json` 的通用格式
- `k8s_instance.supports_rollouts`
- `traffic_provider`

这样后续接入时，主要变更点会集中在：
1. 渲染目标从 `Deployment` 切换为 `Rollout`
2. 新增 rollout-specific 预检
3. 新增流量与分析阶段

而不是再次重做表结构。

## 16. 验收标准
以下视为后端完成：
1. 建立 `delivery_engine` 与 `strategy_engine` 双维度模型
2. 建立 `k8s_instance` 与 `application_env_runtime_binding` 模型
3. 支持 `app + env` 绑定原生 K8s 发布策略
4. 发布单创建时生成并保存策略快照
5. 执行前预检新增策略校验
6. GitOps 渲染可输出原生 K8s 策略、探针、资源、PDB
7. 标准回滚可恢复策略快照
8. 数据模型已为 Argo Rollouts 预留扩展位
