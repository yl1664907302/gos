# 后端需求文档 v2.1.0（小程序发布）

## 1. 文档信息
- 文档版本：`v2.1.0`
- 基线日期：`2026-04-29`
- 本版主题：`多平台小程序发布`
- 当前状态：`规划中`

## 2. 背景
当前 GOS 已经覆盖了 Jenkins（CI 构建）、ArgoCD / GitOps（CD 部署）、Agent 脚本（生产孤岛执行）三条执行链路，但尚未覆盖小程序发布场景。

小程序发布与普通服务部署有本质区别：
- 产物不是镜像或 YAML，而是小程序代码包
- 目标不是 K8s 集群，而是微信/支付宝等第三方平台
- 流程中包含「审核」环节，是异步、不可控的外部依赖
- 发布凭证（上传密钥）极其敏感，不应暴露在 CI 脚本中

参考 CLAUDE.md 中的产品备注——小程序发布应建模为**独立的 Executor 类型**，而非 Jenkins 临时任务。

## 3. 目标
构建一套适合多平台小程序发布的治理链路：

1. 小程序发布融入现有 Release Order 体系，复用审批、状态流转、权限
2. 支持多平台（微信、支付宝等）通过统一接口抽象扩展
3. CI 只负责构建产物，GOS 负责调用平台 API 上传/审核/发布
4. 凭证加密存储，CI 脚本不接触小程序密钥
5. 发布模板可配置小程序发布策略（平台、发布模式等）
6. 审核结果通过回调 + 轮询获取，GOS 自动更新工单状态

## 4. 范围

包含：
- 小程序平台配置管理（Application 级绑定）
- 小程序版本记录
- CI 构建产物回调
- 代码上传（调用小程序平台 API）
- 提交审核 / 查询审核状态 / 审核回调接收
- 全量发布 / 分阶段发布 / 版本回滚
- 与 Release Order / Release Template 的集成
- 微信小程序 API 客户端实现
- 凭证加密存储（复用现有 crypto 方案）

不包含：
- 支付宝 / 抖音平台客户端（架构预留，后续版本实现）
- 小程序代码编译 / 构建（由 Jenkins CI 完成）
- 小程序开发者工具的 GUI 操作
- 小程序的 CDN 资源管理

## 5. 核心概念

### 5.1 MiniProgramConfig（小程序配置）

小程序配置挂在 Application 上，一个 Application 对应一个小程序平台的发布配置。

字段设计：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 主键 |
| `application_id` | string | 关联 Application |
| `platform` | string | 平台: `wechat` / `alipay` / `douyin` |
| `app_name` | string | 小程序名称（可读标识） |
| `app_id` | string | 小程序 AppID |
| `app_secret` | string | 加密存储 |
| `private_key` | string | 代码上传密钥，加密存储 |
| `project_path` | string | 小程序项目在仓库中的相对路径 |
| `status` | string | `active` / `inactive` |
| `created_at` | int64 | 创建时间 |
| `updated_at` | int64 | 更新时间 |

约束：
- 一个 Application 在同一平台上只能有一条配置
- `app_secret` 和 `private_key` 使用 `internal/support/secure` 加密

### 5.7 审批（Approval）与审核（Review）的关系

这是小程序发布设计中最关键的区分。GOS 现有的审批体系和微信平台的审核是完全不同的两件事。

**内部审批（Approval）—— 复用现有体系**

- 审批方：团队内部的审批人（`ApprovalApproverIDs`）
- 时机：在构建执行之前
- 操作：人工在 GOS 页面提交审批、通过、驳回
- 数据：`ReleaseOrderApprovalRecord` 表
- 可选：模板可配置是否启用审批

**平台审核（Review）—— 新增链路**

- 审核方：微信/支付宝平台
- 时机：代码上传之后
- 操作：GOS 调用平台 API 提交，平台异步返回结果
- 数据：`miniprogram_reviews` 表
- 必选：发布到小程序平台必须先通过平台审核

**完整状态流转**

```
            内部审批（复用）                    平台审核（新增）
         ┌─────────┴──────────┐          ┌──────────┴──────────┐
draft → pending_approval → approved → building → uploaded → review_submitted
                         ↘ rejected                  │
                                              ┌──────┴──────┐
                                              ▼             ▼
                                        review_approved  review_rejected
                                              │             │
                                              ▼             ▼
                                          publishing    可修改重提
                                              │
                                              ▼
                                        deploy_success
```

关键规则：
1. 内部审批是发布工单的生命周期阶段（`OrderStatus`），不可跳过批准直接执行
2. 平台审核是小程序版本的状态（`MiniProgramReview.ReviewStatus`），与工单状态联动但不混用
3. `miniprogram_reviews` 表独立于 `ReleaseOrderApprovalRecord` 表，两者字段语义不同
4. 平台审核驳回时，工单状态变为 `review_rejected`，但**不走审批驳回的 `RejectedAt` / `RejectedBy` 字段**——那属于内部审批的语义
5. 审核驳回后，用户可以修改代码重新构建、重新提交审核，形成新的审核记录，旧记录保留为历史

### 5.8 平台客户端接口抽象

```go
// MiniProgramPlatformClient 小程序平台统一接口
type MiniProgramPlatformClient interface {
    // Upload 上传代码包
    Upload(ctx context.Context, req UploadRequest) (UploadResult, error)
    // SubmitReview 提交审核
    SubmitReview(ctx context.Context, req ReviewRequest) (ReviewResult, error)
    // QueryReviewStatus 查询审核状态
    QueryReviewStatus(ctx context.Context, auditID string) (ReviewStatus, error)
    // Publish 发布（全量或分阶段）
    Publish(ctx context.Context, req PublishRequest) (PublishResult, error)
    // RollbackVersion 回滚到指定版本
    RollbackVersion(ctx context.Context, versionID string) error
    // GetQRCode 获取体验版二维码
    GetQRCode(ctx context.Context, versionID string) (string, error)
}

type UploadRequest struct {
    AppID        string
    PrivateKey   string
    Version      string
    Description  string
    ArtifactPath string
}

type ReviewRequest struct {
    AppID   string
    Version string
    Remark  string
}

type PublishRequest struct {
    AppID      string
    Version    string
    Type       string // full / partial
    Percentage int
}
```

每种平台提供各自的实现：`internal/infrastructure/miniprogram/wechat_client.go`、`alipay_client.go` 等。

## 6. CI 触发与回调流程

### 6.1 前置条件

发布工单必须先通过**内部审批**（`OrderStatus` 到达 `approved`），才能进入 CI 构建阶段。如果模板配置了 `ApprovalEnabled = false`（无需审批），则跳过审批直接进入构建。

### 6.2 整体时序

```
Jenkins (CI)                    GOS                        微信平台
    │                            │                            │
    │◀── TriggerBuild(params) ───│                            │
    │    git_ref, app_id,        │                            │
    │    version, miniprogram    │                            │
    │                            │                            │
    │ checkout + build            │                            │
    │ 产物归档                    │                            │
    │                            │                            │
    │─── webhook 回调 ──────────▶│                            │
    │    status, artifact_url    │                            │
    │                            │                            │
    │                            │── Upload ─────────────────▶│
    │                            │                            │
    │                            │── SubmitReview ───────────▶│
    │                            │                            │
    │                            │   ... 等待审核 ...          │
    │                            │                            │
    │                            │◀── 审核结果回调 ────────────│
    │                            │                            │
    │                            │── Publish ────────────────▶│
```

### 6.2 CI 触发

发布工单启动时，GOS 调用 `JenkinsReleaseExecutor.TriggerBuild()`，传入 CI 构建参数：

```go
buildParams := map[string]string{
    "GIT_REF":           order.GitRef,
    "APP_ID":            config.AppID,
    "VERSION":           version,
    "MINIPROGRAM_PLATFORM": config.Platform,
    "PROJECT_PATH":      config.ProjectPath,
    // 不传 app_secret / private_key
}
```

CI 构建完成后，通过 webhook 回调 GOS：

```
POST /api/v1/miniprogram/build-callback
{
    "release_order_id": "xxx",
    "status": "success",         // success / failed
    "artifact_url": "https://...",
    "build_log_url": "https://..."
}
```

轮询作为兜底机制——如果 webhook 未收到，`TrackReleaseExecution` 定时查询 Jenkins build 状态。

### 6.3 上传阶段

GOS 收到 CI 构建成功通知后，执行上传：

1. 从制品库拉取代码包
2. 解密 `private_key`
3. 调用平台 API 上传
4. 更新 `MiniProgramVersion.Status` 为 `uploaded`
5. 更新 ReleaseOrder 流水线阶段

### 6.4 审核阶段

上传成功后，GOS 调用平台 API 提交审核：

1. 调用 `SubmitReview()`
2. 更新版本状态为 `reviewing`
3. 更新 ReleaseOrder 状态为 `review_submitted`
4. 等待审核结果（回调 + 定时轮询 `QueryReviewStatus`）

审核通过：
- `MiniProgramReview.ReviewStatus` → `approved`
- `MiniProgramVersion.Status` → `approved`
- `ReleaseOrder.OrderStatus` → `review_approved`（注意不是 `approved`，后者是内部审批通过的含义）

审核驳回：
- `MiniProgramReview.ReviewStatus` → `rejected`，`ReviewerNote` 记录驳回原因
- `MiniProgramVersion.Status` → `rejected`
- `ReleaseOrder.OrderStatus` → `review_rejected`
- 用户可修改代码后，在同一工单下重新触发「重新提交审核」操作
- 通知相关人员，可修改后重新提交

### 6.5 发布阶段

审核通过后，用户可执行发布：

1. 全量发布：`Publish(type=full)`
2. 分阶段发布：`Publish(type=partial, percentage=30)`，逐步递增
3. 发布成功后，版本状态 → `published`
4. 支持回滚：`RollbackVersion()` 回退到上一个正式版本

## 7. 数据库变更

### 7.1 新增表

```sql
-- 小程序配置表
CREATE TABLE miniprogram_configs (
    id VARCHAR(64) PRIMARY KEY,
    application_id VARCHAR(64) NOT NULL,
    platform VARCHAR(32) NOT NULL,
    app_name VARCHAR(128) NOT NULL,
    app_id VARCHAR(128) NOT NULL,
    app_secret TEXT NOT NULL,       -- 加密存储
    private_key TEXT NOT NULL,      -- 加密存储
    project_path VARCHAR(256) NOT NULL DEFAULT '',
    status VARCHAR(32) NOT NULL DEFAULT 'active',
    created_at BIGINT NOT NULL,
    updated_at BIGINT NOT NULL,
    UNIQUE KEY uq_app_platform (application_id, platform),
    INDEX idx_application_id (application_id)
);

-- 版本表
CREATE TABLE miniprogram_versions (
    id VARCHAR(64) PRIMARY KEY,
    app_id VARCHAR(64) NOT NULL,
    version VARCHAR(64) NOT NULL,
    description TEXT NOT NULL,
    release_order_id VARCHAR(64) NOT NULL,
    source_ref VARCHAR(256) NOT NULL,
    artifact_path VARCHAR(512) NOT NULL,
    status VARCHAR(32) NOT NULL DEFAULT 'built',
    created_at BIGINT NOT NULL,
    INDEX idx_release_order_id (release_order_id),
    INDEX idx_app_id (app_id)
);

-- 上传记录表
CREATE TABLE miniprogram_uploads (
    id VARCHAR(64) PRIMARY KEY,
    version_id VARCHAR(64) NOT NULL,
    release_order_id VARCHAR(64) NOT NULL,
    upload_result TEXT NOT NULL,
    ext_info TEXT NOT NULL,
    created_at BIGINT NOT NULL,
    INDEX idx_release_order_id (release_order_id),
    INDEX idx_version_id (version_id)
);

-- 审核记录表
CREATE TABLE miniprogram_reviews (
    id VARCHAR(64) PRIMARY KEY,
    version_id VARCHAR(64) NOT NULL,
    release_order_id VARCHAR(64) NOT NULL,
    submit_result TEXT NOT NULL,
    review_status VARCHAR(32) NOT NULL DEFAULT 'waiting',
    reviewer_note TEXT NOT NULL,
    submitted_at BIGINT NOT NULL,
    reviewed_at BIGINT,
    INDEX idx_release_order_id (release_order_id),
    INDEX idx_version_id (version_id)
);

-- 发布记录表
CREATE TABLE miniprogram_publishes (
    id VARCHAR(64) PRIMARY KEY,
    version_id VARCHAR(64) NOT NULL,
    release_order_id VARCHAR(64) NOT NULL,
    publish_type VARCHAR(32) NOT NULL DEFAULT 'full',
    percentage INT NOT NULL DEFAULT 100,
    status VARCHAR(32) NOT NULL DEFAULT 'pending',
    published_at BIGINT,
    INDEX idx_release_order_id (release_order_id),
    INDEX idx_version_id (version_id)
);
```

### 7.2 现有表变更

`applications` 表无需新增字段。小程序平台信息通过 `miniprogram_configs` 表关联，不污染应用主表。

### 8.2 CI 构建回调

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/api/v1/miniprogram/build-callback` | CI 构建完成回调（Jenkins webhook） |

需要签名校验，防止伪造回调。

### 8.3 审核回调

| 方法 | 路径 | 说明 |
|---|---|---|
| `POST` | `/api/v1/miniprogram/review-callback` | 微信审核结果回调 |

### 8.4 发布操作

这些操作融入现有 Release Order 生命周期，不新增独立路由：

| 操作 | 说明 |
|---|---|
| 上传 | 在 ReleaseOrder 启动时自动执行 |
| 提交审核 | ReleaseOrder Step: `miniprogram_submit_review` |
| 发布 | ReleaseOrder Step: `miniprogram_publish` |
| 回滚 | ReleaseOrder Step: `miniprogram_rollback` |

### 8.5 版本查询

| 方法 | 路径 | 说明 |
|---|---|---|
| `GET` | `/api/v1/miniprogram/versions` | 版本列表，支持按 app_id 过滤 |
| `GET` | `/api/v1/miniprogram/versions/:id` | 版本详情（含上传/审核/发布记录） |
