## 设计目标
- 查询区必须轻，不再使用厚重白卡
- 搜索与筛选必须是工作台顶部的控制件，不应抢卡片主体
- 统计区必须表达“当前要处理什么”，不是单纯展示资产总量
- 图表与状态卡必须和应用卡片保持同一深色主题语言
- 卡片是页面主角，查询区只承担定位与过滤职责
- UI 展现必须和真实查询行为一致，避免“看起来像全局查询，实际只查当前页”这类错觉

## 一 搜索框规范

### 1. UI 位置
- 位于标题栏右侧操作区
- 以放大镜按钮作为入口，不默认常驻输入框
- 点击后在内容区顶部中央展开悬浮搜索框
- 搜索浮层只覆盖右侧内容区，不覆盖左侧菜单栏

### 2. UI 视觉
- 搜索框为半实心磨砂玻璃风格
- 背景为高透明白，不使用纯白硬输入框
- 有轻边框、内高光和背景模糊
- 输入框宽度固定，不跟随内容伸缩
- 建议列表位于搜索框下方，和输入框属于同一悬浮层体系

### 3. 输入内容
- 只保留一个输入项：`名称或 Key`
- 不再拆成“名称查询”和“Key 查询”两个独立输入框
- 回车可直接触发正式查询
- 点击建议项可直接带值并触发正式查询

### 4. 实时建议查询行为
- 搜索建议为真实后端实时查询，不是本地前端模糊匹配
- 输入变化后进行防抖，请求间隔为 `180ms`
- 查询接口：`GET /applications`
- 请求参数：
  - `keyword`
  - `project_id`，继承当前项目筛选值
  - `status`，继承当前列表状态筛选值
  - `page=1`
  - `page_size=6`
- 后端模糊匹配规则：
  - `a.app_key LIKE %keyword%`
  - `a.name LIKE %keyword%`
- 结果为前 6 条应用记录
- 做了过期请求保护，旧请求返回不会覆盖新输入结果

### 5. 正式查询行为
- 用户按回车或点击建议项后，才会更新工作台主列表条件
- 正式查询会写入 `listStore.keyword`
- 正式查询后统一执行：
  - `loadApplications()`
  - `loadOverviewMetrics()`
  - `loadRecentReleases()`
  - `loadReleaseStateSummaries()`
- 触发正式查询时，分页重置到第一页

### 6. 使用边界
- 搜索建议是“候选定位”，不是最终列表本身
- 建议列表只解决输入辅助，不承担完整筛选展示
- 当输入为空时，不展示建议列表

## 二 项目下拉查询框规范

### 1. UI 位置
- 位于标题栏操作区
- 在搜索按钮右侧
- 属于直接可操作控件，不再使用二次弹框承载筛选项

### 2. UI 视觉
- 使用和标题栏其他按钮同源的透明玻璃风格
- 但其语义是选择器，不是普通按钮
- 控件长度允许明显长于普通按钮，以容纳项目名
- 占位文本与已选中文本需要垂直居中
- 当前文案占位为：`项目`

### 3. 交互规则
- 点击控件本体直接下拉
- 支持清空
- 不再包裹 `popover`
- 当前值变化时直接触发筛选，不需要额外确认按钮

### 4. 项目选项加载行为
- 页面初始化时加载项目选项
- 请求接口：项目列表接口
- 当前前端实现参数：
  - `page=1`
  - `page_size=200`
  - `status=active`
- 选项格式：`项目名 (project_key)`

### 5. 项目筛选生效行为
- 项目变化后重置分页到第一页
- 然后统一执行：
  - `loadApplications()`
  - `loadOverviewMetrics()`
  - `loadRecentReleases()`
  - `loadReleaseStateSummaries()`
- 说明：
  - 应用列表会按当前 `project_id` 重新查询
  - 搜索建议实时查询也会继承当前 `project_id`

### 6. 当前实现限制
- 当前标题栏项目选择器不启用本地输入搜索
- 交互目标是“直接下拉选项目”，不是“在选择器里再次模糊检索项目”

## 三 ECharts 统计区规范

### 1. 区域角色
- 位于标题栏和应用卡片网格之间
- 是应用工作台的总览层，不是单独的分析页
- 作用是回答两个问题：
  - 近 24 小时发布整体走势如何
  - 当前最值得点击处理的是哪几类状态

### 2. UI 结构
- 左侧为一张主图表卡：`近 24 小时执行概览`
- 右侧为四张状态摘要卡
- 当前状态卡固定为：
  - `待审批`
  - `发布中`
  - `今日失败`
  - `可发布`
- 布局为：
  - 左：主图表
  - 右：2 x 2 状态卡

### 3. 图表视觉规范
- 图表底板使用深色控制台底色，不使用白底 ECharts 默认风格
- 顶部允许保留一条很弱的蓝绿高光线
- 网格线、坐标轴、刻度文字都要弱化，避免监控大屏感
- Tooltip 使用深色玻璃面板，不使用默认浅灰浮层
- 系列颜色固定：
  - 成功：绿色
  - 执行：蓝色
  - 失败：红色
- 图表只允许做平滑折线 + 面积，不做柱图、饼图、仪表盘

### 4. 图表文案规范
- 左上标题：
  - 标签：`发布态势`
  - 主标题：`近 24 小时执行概览`
- 右上辅助信息：
  - `全部应用 x · 近 24 小时共 y 次状态变化`
  - 当无波动时：`全部应用 x · 近 24 小时暂无波动`
- 底部脚注：
  - `统计口径：按发布单最近状态更新时间汇总`

### 5. 状态卡视觉规范
- 状态卡与图表卡保持同一深色体系
- 不做大面积纯色铺满，只用局部色调表达状态
- 卡片信息层次固定为三层：
  - 标签
  - 大数字
  - 一行解释
- 可点击的状态卡右上角显示 `查看`
- 不可点击的状态卡不显示跳转提示

### 6. 状态卡交互规范
- `待审批`：
  - 点击跳转审批工作台 `/release-approvals`
- `发布中`：
  - 点击跳转发布单列表 `/releases`
  - 携带过滤条件 `status=deploying`
- `今日失败`：
  - 点击跳转发布单列表 `/releases`
  - 携带过滤条件：
    - `status=deploy_failed`
    - `created_at_from=当天`
    - `created_at_to=当天`
- `可发布`：
  - 当前为只读提示，不做点击跳转

### 7. 图表实际查询与统计口径

#### 应用全集
- 来源：`loadOverviewMetrics()`
- 先通过 `listAllApplicationsForOverview()` 拉取当前用户可见的全部应用
- 不受当前分页影响

#### 发布单全集
- 在应用全集确定后，逐个应用请求发布单列表
- 当前实现参数：
  - `application_id`
  - `page=1`
  - `page_size=100`
- 汇总后按 `created_at` 倒序存入 `overviewRecentReleaseOrders`

#### 图表时间口径
- 固定统计最近 24 个整点小时桶
- 时间轴为：
  - 当前小时向前回溯 23 小时
  - 共 24 个点
- 每条发布单按 `updated_at -> finished_at -> started_at -> created_at` 的优先顺序取时间
- 再根据时间落入对应小时桶

#### 图表状态归类
- 成功线：
  - `deploy_success`
- 执行线：
  - `deploying`
- 失败线：
  - `deploy_failed`

#### 状态卡统计口径
- `待审批`：
  - `pending_approval`
  - `approving`
  - 按发布单 ID 去重
- `发布中`：
  - `deploying`
  - 按发布单 ID 去重
- `今日失败`：
  - `deploy_failed`
  - `updated_at` 或回退时间命中今天
- `可发布`：
  - 当前用户对应用有 `release.create` 权限
  - 且应用存在可用模板

### 8. 当前实现边界
- 当前图表不是历史趋势表，只是“最近状态更新时间”趋势
- 当前 `发布中` 只统计 `deploying`，未把 `queued`、`building` 合并进来
- 当前 `今日失败` 用自然日，不是最近 24 小时滚动窗口
- 如果后续要升级口径，应先修改本文档再改实现

## 四 应用卡片规范

### 1. 卡片角色
- 卡片是应用工作台主视图，不是传统表格替代品
- 每张卡片代表一个应用的当前工作面板
- 卡片优先承担“状态浏览 + 快速操作 + 环境切换”三类职责

### 2. 折叠态 UI
- 默认展示折叠态
- 折叠态展示内容：
  - 项目名
  - 负责人
  - 应用中文名称
  - 应用 Key
  - 最近发布单号
  - 当前发布状态
  - 操作按钮：发布、详情或查单、配置
- `详情` 与 `查单` 互斥：
  - 有环境视图数据时显示 `详情`
  - 无环境视图数据时显示 `查单`

### 3. 详情态 UI
- 详情态不再展开完整大面板，不做卡片合并
- 点击 `详情` 后卡片内部切换为环境视图
- 详情态当前只保留：
  - 当前环境标题
  - 生效发布单气泡
  - 最近发布单气泡
  - 底部按钮：切换、发布单、返回
- 当仅有一个环境时，不显示 `切换`
- 当无环境数据时，显示空态而不是禁用 `详情`

### 4. 卡片视觉
- 卡片使用深色控制台风格
- 折叠态和详情态保持同一设计语言
- 运行中应用允许整卡进入发布动效态
- 动效触发条件依赖真实 `runningCount > 0`
- 不允许使用伪发布状态常亮

### 5. 卡片按钮规则
- `发布`：主操作按钮
- `详情/查单`：次主操作按钮
- `配置`：管理操作入口
- 普通用户可以看到 `配置` 按钮，但按钮不生效
- 管理员或具备应用管理权限的用户可点击 `配置`

## 五 卡片实际查询与数据拼装规则

### 1. 卡片主列表数据来源
- 应用卡片主数据来自 `loadApplications()`
- 查询接口：`GET /applications`
- 实际使用参数：
  - `keyword`
  - `project_id`
  - `status`
  - `page`
  - `page_size`
- 也就是说：
  - 卡片区域本身是分页数据
  - 当前页只展示本次分页返回的应用

### 2. 卡片附加状态数据来源
当前页应用加载完成后，会基于当前页应用 ID 再补充两类数据：

#### 最近发布数据
- 函数：`loadRecentReleases()`
- 针对当前页每个应用分别请求发布单列表
- 每个应用取最近 12 条发布单
- 用途：
  - 最近发布单号
  - 环境视图中的最近记录
  - 折叠态状态文案

#### 生效状态摘要
- 函数：`loadReleaseStateSummaries()`
- 仅针对当前页应用集合请求
- 用途：
  - 环境视图中的“生效”发布单号
  - 当前/上次版本信息
  - 标准回滚能力提示的前置环境数据

### 3. 卡片列表和统计区的查询范围不同
这是当前页面里最需要明确写清楚的一点。

#### 顶部统计区
- 来源：`loadOverviewMetrics()`
- 会通过 `listAllApplicationsForOverview()` 分页拉取当前用户可见的全部应用
- 不受当前分页影响
- 因此图表和右侧状态卡都是全局汇总口径

#### 应用卡片列表
- 来源：`loadApplications()`
- 严格受当前筛选条件与分页影响
- 因此卡片区只展示当前页数据

### 4. 查询一致性要求
- 搜索建议查询口径要和正式列表查询口径一致
- 项目下拉框变更后，搜索建议、主列表、卡片附加数据都必须同步更新
- 卡片不能展示脱离当前列表结果集的附加环境数据
- 统计区可以是全局口径，但必须明确和主列表分页口径区分

## 六 后端查询规范

### 1. 应用列表接口
- 路由：`GET /applications`
- 关键参数：
  - `keyword`
  - `key`
  - `name`
  - `project_id`
  - `status`
  - `application_id`
  - `page`
  - `page_size`

### 2. 关键模糊查询规则
- `keyword` 同时匹配：
  - `app_key`
  - `name`
- SQL 规则：
  - `(a.app_key LIKE ? OR a.name LIKE ?)`

### 3. 鉴权口径
- `/applications` 返回结果受应用可见权限约束
- 所以前端的搜索建议、主列表和总计，最终都基于当前用户的可见应用集合

---

## 六 落地约束
- 搜索建议必须继续走真实接口，不回退到前端本地模糊匹配
- 项目筛选必须保持“直接下拉”模式，不再回到 `popover` 弹层模式
- 卡片视图的所有附加状态都必须以当前页应用集合为边界
- 如果后续新增状态筛选项，搜索建议接口参数也要同步携带，保证建议和正式结果一致

## 七 后续演进建议
- 搜索建议可增加高亮匹配字段，但不改变单输入框结构
- 项目选择器可在必要时增加内部搜索，但前提是不破坏“直接下拉”的主交互
- 卡片后续如增加更多状态标签，必须优先复用现有环境摘要数据，避免再次叠加无边界请求
