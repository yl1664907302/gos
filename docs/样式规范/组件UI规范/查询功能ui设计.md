# 查询功能UI设计

## 1. 目的
- 固化当前“放大镜模糊查询 + 顶栏多条件查询”的统一方案。
- 明确查询入口的位置、视觉样式、实时预选结果、正式查询触发方式和多条件筛选行为。
- 避免页面回退到“标题下方一整块白卡查询区 + 多输入框 + 查询/重置”的旧形态。

## 2. 适用范围
- 前端技术栈：`Vue 3 + Ant Design Vue`
- 适用页面：列表页、工作台页、参数页、资源管理页等“以表格或卡片为主视图”的页面
- 当前基线页面：
  - 执行器参数页：`/Users/lingyunxieqing/Desktop/gos/frontend/src/views/component/ExecutorParamManagementView.vue`
  - 应用工作台页：`/Users/lingyunxieqing/Desktop/gos/frontend/src/views/application/ApplicationListView.vue`

## 3. 与旧查询区规范的关系
- 本文档优先级高于 `/Users/lingyunxieqing/Desktop/gos/docs/样式规范/组件UI规范/04-查询区.md` 中的“`a-card + inline form` 固定查询区”做法。
- 对于需要“顶部轻控制、主内容更突出”的页面，统一采用本文档方案。
- 如果页面仍保留传统查询卡片，必须先说明业务原因，不能默认复用旧样式。

## 4. 总体布局要求

### 4.1 页面结构
- 页面头部只保留一行：
  - 左侧：页面标题
  - 右侧：查询与筛选操作区
- 不再在标题下方追加独立“查询卡片”。
- 不再保留“应用选择器 + 平台 Key 输入框 + 查询/重置按钮”这种横向大表单。

### 4.2 操作区位置
- 查询与筛选控件固定放在标题栏右侧 `page-header-actions` 区域。
- 从左到右的推荐顺序：
  - 放大镜查询入口
  - 多条件筛选框
  - `查询`按钮
  - 上下文辅助按钮，例如 `查看管线绑定`
- 搜索与筛选属于页面头部控制件，不应挤占主表格上方空间。

## 5. 放大镜模糊查询规范

### 5.1 入口位置
- 使用单独的放大镜按钮作为入口。
- 按钮位于标题栏右侧操作区的最左端。
- 默认不常驻宽输入框，避免页面头部过重。

### 5.2 按钮视觉
- 复用页面头部玻璃态按钮壳层。
- 推荐规格：
  - 宽高：`42px`
  - 圆角：`16px`
  - 背景：半透明浅色玻璃
  - 边框：浅描边
  - 阴影：轻高光 + 轻投影
- 图标单独居中，不显示额外文本。

### 5.3 浮层位置
- 点击放大镜后，在右侧内容区顶部中央展开浮层搜索面板。
- 浮层遮蔽只覆盖主内容区，不覆盖左侧菜单。
- 浮层推荐布局：
  - 顶部为输入框
  - 下方为实时预选结果列表
- 推荐宽度：`min(100%, 480px)`

### 5.4 浮层视觉
- 输入框与建议列表属于同一悬浮层体系。
- 推荐视觉：
  - 半透明白色玻璃底
  - 轻描边
  - 内高光
  - 背景模糊
  - 轻投影
- 禁止使用硬白输入框、厚重黑遮罩、独立深色下拉面板。

### 5.5 输入内容
- 只保留一个统一关键词输入框。
- 占位文案必须直接写清命中范围，例如：
  - `应用 / 应用Key / 平台Key`
- 支持：
  - 回车正式查询
  - `Esc` 关闭浮层
  - 点击预选结果直接带值并触发正式查询

## 6. 实时预选结果规范

### 6.1 查询来源
- 预选结果必须走后端真实模糊查询。
- 禁止只在前端对当前页结果做假过滤。

### 6.2 请求时机
- 用户输入变化后进行防抖请求。
- 当前基线防抖时间：`220ms`
- 为空时不请求，不展示结果列表。

### 6.3 查询参数
- 统一传入：
  - `keyword`
  - 当前多条件筛选值
  - `page=1`
  - `page_size=6`
- 预选结果必须继承当前顶部多条件筛选值，避免“建议结果”和“正式结果”不一致。

### 6.4 结果命中范围
- 执行器参数页当前基线命中范围：
  - `application.name`
  - `application.app_key`
  - `executor_param_def.param_key`
- 若后续页面复用此模式，必须在各自文档里写清命中字段，不能只写“支持模糊搜索”。

### 6.5 结果项结构
- 建议项使用两行结构：
  - 第一行：主标题
  - 第二行：辅助信息
- 执行器参数页当前基线：
  - 主标题：`应用名 (应用Key)`
  - 副标题：`平台Key · 所属执行器`
- 建议项必须是可点击按钮，而不是纯文本列表。

### 6.6 结果状态
- 查询中展示轻量加载态，例如 `正在查询`
- 要有过期请求保护，旧请求返回不能覆盖新输入结果
- 输入为空或关闭浮层时，应立即清空建议列表

## 7. 正式查询行为规范

### 7.1 触发方式
- 正式查询只能由以下动作触发：
  - 放大镜浮层中按回车
  - 点击预选结果
  - 点击顶部 `查询` 按钮

### 7.2 行为要求
- 正式查询触发后：
  - 分页回到第一页
  - 刷新主列表
  - 同步路由查询参数
- 如果关键词为空且页面存在初始上下文，应恢复页面默认上下文，而不是保留脏状态。

### 7.3 空态规则
- 当页面没有默认上下文，且用户也没有输入关键词时，不主动加载无意义全量数据。
- 应展示明确空态提示，例如：
  - `请使用右上角搜索应用 / 应用Key / 平台Key`

## 8. 多条件查询框规范

### 8.1 位置
- 多条件筛选框位于放大镜按钮右侧、`查询`按钮左侧。
- 直接铺在标题栏右侧操作区内，不再包一层卡片或 popover。
- 多条件筛选和主搜索同属一个操作带，不能拆成“顶部按钮 + 下方筛选卡”两层结构。

### 8.2 控件组成
- 当前执行器参数页基线控件：
  - `绑定类型`
  - `状态`
  - `展示`
  - `可编辑`
  - `查询`
- 多条件筛选用于收敛主结果，不再重复承载“应用”“平台 Key”这类已被统一关键词吸收的条件。

### 8.3 控件视觉
- 所有下拉框与 `查询` 按钮必须使用同源玻璃态样式。
- 推荐规格：
  - 高度：`42px`
  - 圆角：`16px`
  - 内边距：`14px`
  - 最小宽度：`138px`
  - 背景：半透明浅色玻璃
  - 边框：浅描边
  - Hover / Focus：只做轻提亮
- 文本、占位符、箭头颜色需要统一，保持同一层级质感。

### 8.4 清空规则
- 顶部筛选框不显示输入框右侧的清空 `x` 图标。
- 如果业务仍需要“回到无筛选状态”，应通过以下方式之一解决：
  - 下拉选项显式提供 `全部`
  - 页面提供独立的恢复上下文入口
- 禁止依赖默认 `allow-clear` 作为唯一回退手段。

### 8.5 生效方式
- 多条件筛选改值后先进入暂存状态。
- 只有点击 `查询` 按钮后，主列表才正式刷新。
- 这种模式适用于“条件较多、频繁自动查询会打断使用节奏”的页面。

## 9. 查询按钮规范

### 9.1 位置
- `查询`按钮位于多条件筛选框之后。
- 按钮和下拉框必须保持同一视觉层级，不能做成单独主色大按钮。

### 9.2 文案
- 固定文案：`查询`
- 不再在同一位置使用 `重置`

### 9.3 行为
- 点击后触发正式查询：
  - 重置分页到第一页
  - 以当前关键词和多条件筛选值刷新主列表
  - 写回当前路由参数

## 10. 响应式要求
- 桌面端：
  - 操作区横向排列
  - 支持换行，但整体仍保持右上角聚合
- 平板与小屏：
  - 标题区允许折行为上下结构
  - 多条件筛选框可转为满宽或分行排列
  - 搜索浮层左侧遮蔽边界改为 `0`
- 禁止为了小屏回退到旧的厚重查询卡片。

## 11. 禁用项
- 禁止恢复标题下方整块白卡查询区。
- 禁止把“应用查询”和“平台 Key 查询”拆成两个输入框。
- 禁止使用前端本地假模糊替代后端真实预选查询。
- 禁止让顶部筛选框一改即查，同时又保留一个没有实际作用的 `查询` 按钮。
- 禁止把搜索浮层遮蔽到左侧菜单上。
- 禁止在同一页面同时出现“放大镜浮层搜索”和“下方完整查询卡”两套搜索入口。
